# 题目口述：百万级表如何做查询优化

**适用场景：** 大表查询优化、慢接口治理、列表页性能优化、数据库稳定性建设。  
**回答主线：** 百万级表先不要急着分库分表，先判断慢在 **扫描行数、排序、回表、深分页、统计聚合、热点数据** 里的哪一类，再按 SQL（Structured Query Language，结构化查询语言）、索引、表结构、缓存和架构分层处理。

## 1. 百万级表查询优化先看什么？（基本难度）

### 1.1 最简练版

**百万级表不一定慢，慢的是访问路径不合理。**  
我会先看 SQL 的查询条件、排序字段、分页方式和返回字段，再用 `EXPLAIN` 看 `type`、`key`、`rows`、`Extra`。  
如果扫描行数大、出现 `Using filesort`、大量回表或深分页，就优先从 SQL 改写和联合索引入手。  
只有单表治理后仍受容量、写入、热点或维护成本限制，才考虑缓存、归档、读写分离或分库分表。

### 1.2 详细解释版

百万级表优化不能只盯着“表有多少行”，而要看一次请求实际读了多少行、排了多少行、回表多少次，以及这个请求有多高频。

| 关注点 | 判断方式 | 常见问题 |
|---|---|---|
| 查询条件 | `where` 是否有高选择性条件 | 只按低选择性状态查，扫很多行 |
| 执行计划 | `type`、`key`、`rows`、`Extra` | 全表扫、额外排序、临时表 |
| 排序分页 | `order by` 和 `limit` | 排序无法复用索引、深分页 |
| 返回字段 | 是否 `select *` | 破坏覆盖索引，增加回表和网络传输 |
| 数据分布 | 租户、状态、时间是否倾斜 | 大租户或热点状态拖慢查询 |
| 链路成本 | QPS（Queries Per Second，每秒查询数）、P95（95th Percentile，第 95 百分位延迟） | 单次不慢但总成本很高 |

常见处理顺序：

1. **先定位慢 SQL 和真实参数**，不要用空泛 SQL 判断。
2. **再看执行计划**，确认访问路径和扫描量。
3. **优化 SQL 和索引**，让过滤、排序、分页尽量走同一条索引路径。
4. **治理大查询场景**，深分页、统计、导出单独处理。
5. **最后评估架构手段**，包括缓存、归档、只读库、分库分表。

### 1.3 例子

假设订单表 `order_tab` 有 `500` 万行，接口慢 SQL 是：

```sql
select *
from order_tab
where status = 1
order by created_at desc
limit 100000, 20;
```

这个 SQL 的问题不是“表有 500 万行”这么简单，而是：

| 问题 | 影响 |
|---|---|
| `status = 1` 选择性低 | 可能扫大量正常状态订单 |
| `select *` | 列表页读出很多不必要字段 |
| `limit 100000, 20` | 需要先扫并丢弃 10 万条 |
| 缺少匹配排序的联合索引 | 可能额外排序 |

可以先把查询改成按租户和时间窗口收敛：

```sql
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = 1
  and created_at >= ?
  and created_at < ?
order by created_at desc, id desc
limit 20;
```

再配合索引：

```sql
create index idx_tenant_status_time_id
on order_tab(tenant_id, status, created_at, id);
```

这样优化目标就从“处理百万级表”变成了 **让一次查询只扫几十到几百行**。

## 2. 单条 SQL 应该怎么优化？（中等难度）

### 2.1 最简练版

**单条 SQL 优化的核心是少扫行、少排序、少回表、少返回。**  
我会先用 `EXPLAIN` 看是不是命中预期索引、扫描行数是否过大、是否出现 `Using filesort` 或 `Using temporary`。  
然后按 `where + order by + limit` 设计联合索引，避免函数包列、隐式类型转换和 `select *`。  
如果分页很深，就改游标翻页或延迟关联。

### 2.2 详细解释版

单条 SQL 可以按四个方向治理：

| 方向 | 目标 | 做法 |
|---|---|---|
| 过滤 | 尽早缩小范围 | 高选择性条件放入联合索引 |
| 排序 | 避免额外排序 | 让 `order by` 复用索引顺序 |
| 回表 | 减少随机读 | 控制返回字段，使用覆盖索引 |
| 分页 | 控制扫描上限 | 游标翻页、延迟关联、限制页深 |

常见坏味道：

- 对索引列做函数：`date(created_at) = '2026-05-06'`
- 隐式类型转换：字符串列用数字条件比较
- 左模糊匹配：`like '%abc'`
- 大 offset：`limit 100000, 20`
- 不必要的 `select *`
- `order by` 字段和 `where` 索引路径不匹配

### 2.3 例子

原 SQL：

```sql
select id, order_no, buyer_id, status, amount, created_at
from order_tab
where tenant_id = 10001
  and date(created_at) = '2026-05-06'
order by created_at desc
limit 20;
```

问题分析：

| 问题 | 说明 |
|---|---|
| `date(created_at)` | 对列做函数计算，影响索引定位 |
| 缺少 `status` 或业务过滤 | 查询范围可能太大 |
| 排序字段没有和过滤条件形成联合索引 | 容易额外排序 |

改写后：

```sql
select id, order_no, buyer_id, status, amount, created_at
from order_tab
where tenant_id = 10001
  and created_at >= '2026-05-06 00:00:00'
  and created_at <  '2026-05-07 00:00:00'
order by created_at desc, id desc
limit 20;
```

索引：

```sql
create index idx_tenant_time_id
on order_tab(tenant_id, created_at, id);
```

验证时重点看：

| 指标 | 优化前 | 优化后 |
|---|---:|---:|
| `rows` | 可能几万到几十万 | 几十到几百 |
| `Extra` | 可能 `Using where; Using filesort` | 尽量避免大排序 |
| 接口 P95 | 秒级抖动 | 稳定到百毫秒内 |

## 3. 索引和表结构怎么治理？（高难度）

### 3.1 最简练版

**索引治理要围绕核心查询设计，不是给每个字段都建索引。**  
联合索引要匹配高频 SQL 的过滤、排序和分页路径，覆盖索引只覆盖高频小字段。  
表结构上要控制行宽、主键长度、大字段和冷热数据，避免读放大。  
新增索引前还要评估写入成本、磁盘空间、DDL（Data Definition Language，数据定义语言）风险和是否存在重复索引。

### 3.2 详细解释版

索引设计常用原则：

| 原则 | 说明 |
|---|---|
| 高频优先 | 先服务核心链路，不为低频后台查询牺牲写入 |
| 等值在前 | 多个等值条件通常放在联合索引前部 |
| 排序跟上 | 连续等值条件后接排序列，更容易避免额外排序 |
| 控制宽度 | 覆盖索引只放小字段、必要字段 |
| 清理重复 | `(a,b)` 很多时候可以覆盖 `(a)` 的查询 |

表结构治理清单：

| 维度 | 建议 | 原因 |
|---|---|---|
| 主键 | 短、稳定、尽量递增 | 二级索引叶子节点会保存主键 |
| 字段类型 | 用合适长度和类型 | 减少页占用，提高缓存命中 |
| 大字段 | 拆到详情表或对象存储 | 避免列表查询读放大 |
| 冷热数据 | 历史归档或分区管理 | 控制活跃数据规模 |
| 时间字段 | 统一精度和时区 | 方便范围查询和归档 |

### 3.3 例子

订单表原结构：

```sql
create table order_tab (
  id bigint primary key,
  tenant_id bigint not null,
  status tinyint not null,
  amount decimal(12,2) not null,
  address varchar(512) not null,
  remark text,
  created_at datetime not null,
  key idx_status(status),
  key idx_created_at(created_at)
);
```

高频查询是租户订单列表：

```sql
select id, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc, id desc
limit 20;
```

更合理的治理方式：

| 动作 | 目的 |
|---|---|
| 新增 `(tenant_id, status, created_at, id)` | 匹配过滤、排序和分页 |
| 列表页不查 `address`、`remark` | 减少回表和网络传输 |
| 详情页按主键单独查大字段 | 把高频轻查询和低频重查询拆开 |
| 评估删除 `idx_status` | 避免低价值重复索引拖慢写入 |

如果 `remark` 很大且详情页也不总是需要，可以拆成 `order_ext`：

```sql
create table order_ext (
  order_id bigint primary key,
  address varchar(512) not null,
  remark text
);
```

这样高频列表读更轻，Buffer Pool（缓冲池）也更容易缓存热点页。

## 4. 深分页、统计和导出怎么处理？（高难度）

### 4.1 最简练版

**深分页、统计和导出不能都靠主库实时大查询硬扛。**  
深分页优先改成游标翻页，必须跳页时用覆盖索引加延迟关联。  
大范围统计优先做汇总表、缓存或离线计算。  
批量导出要异步化、分批扫描、限速，尽量走只读库或离线链路。

### 4.2 详细解释版

三类场景的处理方式：

| 场景 | 典型风险 | 推荐处理 |
|---|---|---|
| 深分页 | offset 越大，扫描越多 | 游标翻页、延迟关联、限制最大页深 |
| 实时统计 | 大范围 `count`、`group by` | 汇总表、缓存、异步计算 |
| 批量导出 | 长事务、大扫描、抢资源 | 异步任务、分批游标、只读库、限速 |

深分页不要只加索引，因为索引只能减少排序和回表，不能让大 offset 免费消失。  
统计类查询要区分强实时和弱实时，弱实时可以接受 T+1 或分钟级延迟，就不要压主库。  
导出类请求要有任务状态、结果文件、失败重试和并发控制。

### 4.3 例子

深分页原 SQL：

```sql
select id, order_no, created_at
from order_tab
where tenant_id = ?
order by created_at desc, id desc
limit 100000, 20;
```

游标翻页改法：

```sql
select id, order_no, created_at
from order_tab
where tenant_id = ?
  and (
    created_at < ?
    or (created_at = ? and id < ?)
  )
order by created_at desc, id desc
limit 20;
```

统计原 SQL：

```sql
select status, count(*)
from order_tab
where tenant_id = ?
  and created_at >= ?
group by status;
```

汇总表改法：

```sql
select status, sum(order_cnt)
from order_stat_daily
where tenant_id = ?
  and stat_date between ? and ?
group by status;
```

导出链路可以这样设计：

| 步骤 | 说明 |
|---|---|
| 提交任务 | 校验时间范围和导出上限 |
| 后台分批扫描 | 每批按主键或时间游标取固定数量 |
| 写结果文件 | 文件落对象存储或内部文件系统 |
| 通知下载 | 返回任务状态和下载地址 |
| 限速隔离 | 控制并发，避免影响核心接口 |

## 5. 什么时候上缓存、归档或分库分表？（高难度）

### 5.1 最简练版

**缓存、归档和分库分表是不同层级的手段，不能混着用。**  
读多写少、热点明显的数据适合缓存；历史数据占比高、查询集中在近期数据，适合归档。  
当单库单表在容量、写入吞吐、DDL 成本或故障隔离上成为瓶颈，才考虑分库分表。  
分库分表会带来跨分片查询、事务、扩容和运维复杂度，不是慢 SQL 的第一解。

### 5.2 详细解释版

| 手段 | 解决什么 | 不适合什么 |
|---|---|---|
| 缓存 | 热点读、重复读、高 QPS | 强一致写多场景 |
| 归档 | 活跃表过大、历史数据拖慢 | 高频跨冷热查询 |
| 读写分离 | 读压力高于写压力 | 强一致读、复制延迟敏感 |
| 分库分表 | 容量、写入、隔离性瓶颈 | 查询维度多且无法确定分片键 |

选择分库分表前要回答：

1. 分片键是什么，是否覆盖核心查询？
2. 跨分片查询、分页、统计怎么做？
3. 全局唯一 ID 怎么生成？
4. 扩容迁移怎么保证一致性？
5. 历史数据、热点分片怎么处理？

### 5.3 例子

订单表从 `500` 万增长到 `8000` 万，查询主要看最近 `90` 天订单，历史订单访问很少。此时不一定先分库分表，可以先做冷热归档：

| 数据 | 存储位置 | 查询方式 |
|---|---|---|
| 最近 90 天 | `order_tab` 活跃表 | 核心列表和详情 |
| 90 天以前 | `order_history` 历史表 | 低频历史查询 |
| 统计报表 | 汇总表或离线宽表 | 按天、按租户聚合 |

如果后续写入量持续上升，比如单库写入 QPS 长期接近上限，且大部分查询都能按 `tenant_id` 定位，就可以考虑按 `tenant_id` 分片：

```text
tenant_id hash
  -> order_db_00.order_tab_00
  -> order_db_01.order_tab_03
  -> order_db_02.order_tab_07
```

但要提前处理代价：跨租户统计、后台检索、全局排序和数据迁移都不能再像单表一样简单。

## 6. 一段完整回答怎么组织？（中等难度）

### 6.1 最简练版

**完整回答可以按“定位 -> SQL 和索引 -> 特殊场景 -> 架构手段 -> 验证”来讲。**  
先说明百万级表不等于必须拆，关键看一次查询的访问成本。  
再讲用 `EXPLAIN` 找扫描、排序、回表和分页问题，并用联合索引、覆盖索引和 SQL 改写解决。  
最后补充缓存、归档、分库分表的触发条件，以及上线后用指标验证。

### 6.2 详细解释版

回答结构可以这样组织：

| 层次 | 内容 |
|---|---|
| 结论 | 百万级表先做访问路径治理，不急着拆 |
| 定位 | 慢日志、真实参数、`EXPLAIN`、数据分布、接口指标 |
| SQL | 避免函数包列、`select *`、深分页和大范围排序 |
| 索引 | 按 `where + order by + limit` 设计联合索引 |
| 场景 | 深分页、统计、导出分别治理 |
| 架构 | 缓存、归档、读写分离、分库分表按瓶颈选择 |
| 验证 | 看 `rows`、P95、P99、CPU（Central Processing Unit，中央处理器）、I/O（Input/Output，输入输出）、慢日志 |

### 6.3 例子

可以这样回答：

> 百万级表我不会上来就说分库分表，先看这条查询到底扫了多少行、有没有额外排序、有没有大量回表。我的处理顺序一般是先从慢日志拿到真实 SQL 和参数，再用 `EXPLAIN` 看 `type`、`key`、`rows` 和 `Extra`。如果发现扫描行数大或者 `Using filesort`，就按 `where + order by + limit` 设计联合索引，同时把列表页字段收敛，尽量走覆盖索引。
>
> 对深分页我会改游标翻页，必须保留页码时用延迟关联降低回表成本；对统计和导出，我不会让主库同步大范围扫描，而是做汇总表、缓存或异步任务。只有当单表在容量、写入吞吐、DDL 成本或故障隔离上确实撑不住，才考虑归档、读写分离或分库分表。最后一定要用灰度指标验证，比如扫描行数、P95、P99、DB CPU、I/O 和慢日志数量是否真的下降。
