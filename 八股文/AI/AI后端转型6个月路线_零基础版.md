## 1. 最简练版

你最适合转的不是纯算法岗，而是 **AI 应用 / Agent 后端工程师**，因为这个方向同时需要大模型能力和资深后端工程能力。你的 9 年后端经验可以迁移到 **高并发、服务治理、数据链路、权限、审计、成本、可观测性、线上稳定性**，这些正是 AI 应用从 demo 走向生产时最缺的能力。6 个月内不要追求“从零训练大模型”，而是要做出 **两个生产级作品：企业知识库 RAG 系统 + 后端线上排障 Agent**。最终目标是让别人看到你不是“会调模型 API 的后端”，而是能把 AI 应用做成真实系统的人。

## 3. 基础认知

### 3.1 AI 应用后端到底做什么

AI 应用后端不是单纯写一个接口调用模型。真实工作更像是：

```text
用户请求
  |
  v
鉴权 -> 限流 -> Prompt 构造 -> 检索知识 -> 调用模型 -> 流式返回
  |          |             |             |            |
  v          v             v             v            v
审计       缓存          向量库        模型网关       日志指标
```

你要负责的是 **模型之外的工程系统**：

| 模块 | 你要解决的问题 |
|---|---|
| 接入层 | 谁能用、用多少、是否超限、是否要排队 |
| Prompt 层 | 如何构造上下文，如何防注入，如何版本管理 |
| 检索层 | 如何从企业文档里找准内容，如何降低幻觉 |
| 模型层 | 选哪个模型，失败怎么办，成本如何控制 |
| 工具层 | 模型能调哪些工具，如何防止越权 |
| 观测层 | 每次回答花多少钱，慢在哪里，错在哪里 |
| 评测层 | 新版本是不是变好，是否出现退化 |

### 3.2 大模型应用的核心链路

![Transformer 架构图](https://upload.wikimedia.org/wikipedia/commons/8/8f/The-Transformer-model-architecture.png)

图意说明：Transformer 是现代大语言模型的基础架构之一。你不需要一开始推导公式，但要理解模型是根据上下文预测后续文本，应用侧重点是 **上下文构造、检索补充、工具调用和结果校验**。图片来源：Wikimedia Commons。

一个大模型应用通常分为 7 步：

1. 用户提出问题。
2. 系统做鉴权、限流、参数校验。
3. 系统判断是否需要检索知识库。
4. 系统把问题、历史对话、检索结果、业务规则拼成 Prompt。
5. 系统通过模型网关调用 DeepSeek / Qwen / Doubao / OpenAI 等模型。
6. 模型返回内容，系统做安全过滤、引用补充、格式化。
7. 系统记录日志、耗时、Token、成本、质量反馈。

关键不是“模型有多聪明”，而是：

> **你给模型的上下文是否准确，工具权限是否安全，失败时系统是否可控。**

### 3.3 小白必须先懂的 20 个词

| 词 | 全称和中文含义 | 先理解到什么程度 |
|---|---|---|
| LLM | Large Language Model，大语言模型 | 能根据上下文生成文本的模型 |
| Token | 模型处理文本的基本单位 | 成本和上下文长度都和 Token 有关 |
| Prompt | 提示词 | 给模型的指令、背景、格式要求 |
| Context Window | 上下文窗口 | 一次请求里模型最多能看到的内容 |
| Embedding | 向量表示 | 把文本变成向量，方便相似度检索 |
| RAG | Retrieval-Augmented Generation，检索增强生成 | 先检索知识，再让模型回答 |
| Chunk | 文档切片 | 把长文档切成小块用于检索 |
| Vector DB | Vector Database，向量数据库 | 存向量并做相似度搜索 |
| Rerank | 重排序 | 对初步检索结果重新排序，提高相关性 |
| Agent | 智能体 | 能规划步骤、调用工具、完成任务的 AI 系统 |
| Tool Calling | 工具调用 | 模型决定调用外部 API 或函数 |
| MCP | Model Context Protocol，模型上下文协议 | 让模型连接工具和数据源的协议 |
| LLMOps | Large Language Model Operations，大模型运维 | 模型应用的部署、监控、评测、治理 |
| Guardrail | 护栏 | 防止越权、泄密、幻觉和危险输出 |
| Prompt Injection | 提示词注入 | 用户诱导模型忽略原规则或泄露数据 |
| Hallucination | 幻觉 | 模型编造不存在的事实 |
| SSE | Server-Sent Events，服务端事件流 | 常用于模型流式输出 |
| Function Calling | 函数调用 | 让模型按结构化参数调用函数 |
| Eval | Evaluation，评测 | 用测试集判断回答质量 |
| Observability | 可观测性 | 日志、指标、链路追踪和告警 |

### 3.4 不建议一开始深挖的内容

| 内容 | 为什么先不深挖 |
|---|---|
| 从零训练大模型 | 成本高、周期长、岗位门槛偏研究 |
| Transformer 数学推导 | 应用岗不要求先推公式，先会工程落地 |
| CUDA / 算子优化 | 偏 AI Infra 或高性能计算，6 个月窗口不优先 |
| 复杂多模态生成 | 图片、视频、语音链路更长，先把文本应用打透 |
| 微调大模型 | 可以了解 LoRA，但先把 RAG、Agent、评测做好 |

## 4. 六个月路线

### 4.1 第 1 个月：大模型基础和模型网关

目标：从 0 开始跑通 **模型调用 + 流式输出 + 后端接口 + 基础网关能力**。

| 周 | 主题 | 每周产出 |
|---|---|---|
| 第 1 周 | LLM 基础、Prompt、Token、API 调用 | 能用 Python 调通一个模型，并理解输入输出 |
| 第 2 周 | FastAPI / SSE / 异步调用 | 写出一个支持流式输出的聊天接口 |
| 第 3 周 | 模型网关雏形 | 支持模型路由、超时、重试、错误码封装 |
| 第 4 周 | 限流、缓存、成本统计 | 每次请求记录 Token、耗时、模型、费用估算 |

第 1 周每天步骤：

| Day | 做什么 | 验收标准 |
|---|---|---|
| Day 1 | 注册并准备一个模型 API Key，可选 DeepSeek / Qwen / Doubao / OpenAI | 本地能请求模型返回一句话 |
| Day 2 | 学 Prompt 基础：角色、任务、约束、输出格式 | 写 5 个 Prompt，对比效果 |
| Day 3 | 学 Token、上下文窗口、温度参数 | 能解释为什么长上下文更贵、更慢 |
| Day 4 | 用 Python 写最小调用脚本 | 输入问题，输出模型回答 |
| Day 5 | 加入错误处理和超时 | 模型超时后返回友好错误 |
| Day 6 | 写成命令行小工具 | `python chat.py "解释 Redis 缓存击穿"` 能运行 |
| Day 7 | 复盘并写笔记 | 写下 10 个概念和 5 个坑 |

第 2 周每天步骤：

| Day | 做什么 | 验收标准 |
|---|---|---|
| Day 1 | 新建 FastAPI 项目 | `/health` 返回 `ok` |
| Day 2 | 写 `/chat` 普通接口 | Postman / curl 可调用 |
| Day 3 | 写 SSE 流式接口 | 前端或 curl 能看到逐段输出 |
| Day 4 | 加请求 ID | 日志里每个请求都能串起来 |
| Day 5 | 加超时和重试 | 模型失败时最多重试 1 到 2 次 |
| Day 6 | 加 Redis 缓存 | 同样问题可命中缓存 |
| Day 7 | 写接口文档 | README 能说明如何启动 |

第 3 周每天步骤：

| Day | 做什么 | 验收标准 |
|---|---|---|
| Day 1 | 抽象 `ModelProvider` 接口 | 可以接不同模型供应商 |
| Day 2 | 实现模型路由 | 按请求参数选择模型 |
| Day 3 | 实现降级策略 | 主模型失败切备用模型 |
| Day 4 | 实现限流 | 单用户每分钟限制请求数 |
| Day 5 | 记录调用日志 | 包括模型、耗时、状态码、错误 |
| Day 6 | 统计 Token 和估算成本 | 每天能看总成本 |
| Day 7 | 画模型网关架构图 | 能讲清楚链路 |

第 4 周每天步骤：

| Day | 做什么 | 验收标准 |
|---|---|---|
| Day 1 | 加 Prompt 模板管理 | Prompt 不硬编码在业务里 |
| Day 2 | 加 Prompt 版本号 | 每次请求记录模板版本 |
| Day 3 | 加用户鉴权 | 没有 Token 不能调用 |
| Day 4 | 加敏感信息脱敏 | 手机号、邮箱、Key 不直接进日志 |
| Day 5 | 加基础指标 | QPS、p95、错误率、Token 数 |

## 5. 两个核心项目

### 5.1 项目一：企业知识库 RAG 系统

项目定位：

> 做一个支持企业内部文档问答的 RAG 系统，解决模型不知道私有知识、容易幻觉、答案不可追溯的问题。

核心功能：

| 模块 | 功能 |
|---|---|
| 文档导入 | 支持 Markdown、PDF、TXT |
| 文档切片 | 按标题、长度、段落切分 |
| 向量化 | 调用 Embedding 模型生成向量 |
| 检索 | 支持向量检索、关键词检索、混合检索 |
| Rerank | 对候选片段重新排序 |
| 回答生成 | 基于引用材料回答 |
| 引用溯源 | 展示答案来源 |
| 评测 | Recall@5、引用准确率、拒答率 |
| 观测 | QPS、p95、错误率、Token、成本 |

核心架构：

```text
文档上传
  |
  v
解析器 -> 清洗 -> Chunk 切片 -> Embedding -> 向量库
                                      |
用户问题                              v
  |                              相似度检索
  v                                  |
鉴权 -> 改写问题 -> 混合检索 -> Rerank -> Prompt 构造 -> 模型回答
  |                                                              |
  v                                                              v
审计日志                                                    引用来源
```

关键指标：

| 指标 | 目标 |
|---|---|
| Recall@5 | 70% 到 85% 起步 |
| p95 延迟 | 普通问题 3 秒以内，复杂问题 8 秒以内 |
| 引用准确率 | 80% 以上 |
| 无答案拒答率 | 能拒绝明显不在知识库的问题 |
| 缓存命中率 | 高频问题 20% 以上 |

### 5.2 项目二：后端线上排障 Agent

项目定位：

> 做一个能辅助后端工程师排查线上问题的 Agent，把日志、指标、数据库、Kafka、故障手册串起来，输出根因假设和止血方案。

核心功能：

| 模块 | 功能 |
|---|---|
| 告警输入 | 接收错误率升高、接口变慢、消费积压等告警 |
| 任务规划 | 判断需要查哪些系统 |
| 工具调用 | 查询日志、指标、MySQL、Redis、Kafka |
| RAG 辅助 | 查询历史故障手册 |
| 根因排序 | 按证据强弱输出可能原因 |
| 止血方案 | 给出限流、回滚、扩容、降级建议 |
| 人工确认 | 写操作必须确认 |
| 审计 | 记录工具调用和输出 |

Kafka 在排障链路里的位置：

![Kafka Consumer Groups 图](https://kafka.apache.org/0110/images/consumer-groups.png)

图意说明：排障 Agent 需要理解 Kafka 消费组、分区和积压，否则看到延迟升高时很难判断是不是消费能力不足。图片来源：Apache Kafka 官方文档。

核心架构：

```text
告警事件
  |
  v
Agent Orchestrator
  |
  +--> Prometheus 工具：查 QPS / p95 / 错误率
  |
  +--> 日志工具：查 request_id / error stack
  |
  +--> MySQL 工具：查慢 SQL / 连接数 / 锁等待
  |
  +--> Redis 工具：查命中率 / 内存 / 热点 key
  |
  +--> Kafka 工具：查 Lag / 分区 / 消费者状态
  |
  +--> RAG 工具：查历史故障手册
  |
  v
证据汇总 -> 根因排序 -> 止血方案 -> 人工确认 -> 审计记录
```

关键指标：

| 指标 | 目标 |
|---|---|
| 排障任务成功率 | 60% 到 75% 起步 |
| 平均工具调用步数 | 3 到 6 步 |
| 平均响应时间 | 30 秒以内 |
| 危险操作拦截率 | 100% |
| 日志审计完整率 | 100% |

### 8.3 口述模板

讲 RAG 项目：

```text
背景：
公司内部知识分散在文档、故障手册和接口说明里，直接问大模型会不知道私有知识，也容易编造。

目标：
做一个企业知识库问答系统，要求回答可追溯、能拒答、质量可评测。

方案：
我把链路拆成文档解析、Chunk 切片、Embedding、向量库、混合检索、Rerank、Prompt 构造、模型回答和引用溯源。

难点：
主要难点不是调用模型，而是检索质量、切片策略、无答案判断、权限隔离和评测体系。

结果：
我用 Recall@5、引用准确率、p95 延迟和 Token 成本衡量效果，并通过混合检索和 Rerank 优化。

复盘：
RAG 不是万能的，文档质量、权限、评测和坏例迭代比单纯换模型更重要。
```

讲 Agent 项目：

```text
背景：
线上排障需要跨日志、指标、数据库、缓存和消息队列，信息分散，初级同学容易漏查。

目标：
做一个只读优先的排障 Agent，辅助定位问题，输出证据、根因排序和止血建议。

方案：
我设计了 Agent Orchestrator，通过计划、工具调用、观察、再计划的状态机工作，并接入 Prometheus、日志、MySQL、Redis、Kafka 和故障手册 RAG。

难点：
难点是工具权限、安全边界、失败恢复、审计日志和避免 Agent 无限循环。

结果：
系统能针对接口慢、错误率升高、Kafka 积压等场景输出排查路径，并保留每一步证据。

复盘：
Agent 适合作为辅助决策系统，不应该一开始追求完全自动化，生产环境要先做只读、可审计、可回滚。
```
