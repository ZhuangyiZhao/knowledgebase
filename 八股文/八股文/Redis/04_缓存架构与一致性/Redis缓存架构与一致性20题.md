## 1. 缓存的核心价值是什么？（基本难度）

### 1.1 最简练版

**缓存的核心价值是降低后端存储压力、缩短响应时间、提升系统吞吐。**  
Redis 常放在应用和数据库之间，承接高频读。  
但缓存会带来一致性、容量、热点和故障降级问题，不是简单加一层就结束。

### 1.2 详细解释版

缓存适合读多写少、数据可重建、允许短暂不一致的场景。设计缓存时要回答：缓存什么、key 怎么设计、TTL 多久、未命中怎么办、更新时如何失效、Redis 故障怎么降级。  
越靠近用户的缓存越快，但一致性越难；越靠近数据库的缓存越统一，但延迟和成本更高。

### 1.3 图示

```text
Client -> App -> Redis -> DB
              命中返回   未命中查询
```

## 2. 旁路缓存模式怎么工作？（基本难度）

### 2.1 最简练版

**旁路缓存是应用先读缓存，未命中再读数据库，然后把结果写回缓存。**  
写入时通常先更新数据库，再删除缓存。  
这是后端系统最常见的缓存模式，简单、通用、可控。

### 2.2 详细解释版

读流程：查 Redis，命中直接返回；未命中查 DB（Database，数据库），再回填 Redis。写流程：先落数据库，再使缓存失效，让下次读重新构建。  
旁路缓存不保证强一致，重点是通过删除缓存、消息补偿、短 TTL、版本号等手段实现最终一致。

### 2.3 图示

```text
读：Cache -> miss -> DB -> set Cache
写：DB update -> delete Cache
```

## 3. 先更新数据库还是先删除缓存？（高难度）

### 3.1 最简练版

**常见推荐是先更新数据库，再删除缓存。**  
如果先删缓存，再更新数据库，期间读请求可能把旧数据库值重新写入缓存。  
先更新数据库再删缓存也不是绝对无风险，所以通常配合重试、消息补偿和短 TTL。

### 3.2 详细解释版

先删缓存的问题更常见：线程 A 删除缓存后还没更新 DB，线程 B 读 DB 旧值并回填缓存，最后线程 A 更新 DB，新缓存却是旧值。  
先更新 DB 再删缓存，理论上也可能发生“读旧值回填后，写线程删除缓存失败”的问题，所以删除缓存必须可靠：本地重试、MQ（Message Queue，消息队列）补偿、订阅 binlog（Binary Log，二进制日志）异步删除。

### 3.3 图示

```text
推荐主链路：
更新 DB 成功 -> 删除 Cache 成功/重试 -> 后续读重建
```

## 4. 为什么一般不推荐先更新缓存？（中等难度）

### 4.1 最简练版

**不推荐先更新缓存，是因为缓存通常不是权威数据源，直接更新容易出现并发覆盖和脏缓存。**  
多个写请求乱序时，旧值可能覆盖新值。  
删除缓存让读请求重新从数据库构建，逻辑更简单。

### 4.2 详细解释版

缓存里可能是聚合数据、序列化对象、多表拼装结果，不一定容易精确更新。更新缓存还要处理并发顺序、失败重试、局部字段合并和多缓存副本一致。  
删除缓存虽然会带来一次未命中，但能降低一致性复杂度。对于极高读场景，可以使用版本号或 CAS（Compare And Swap，比较并交换）保护更新。

### 4.3 例子

```text
请求 A 更新 name=old
请求 B 更新 name=new
B 先写缓存，A 后写缓存 -> 缓存回退成 old
```

## 5. 延迟双删解决什么问题？（高难度）

### 5.1 最简练版

**延迟双删用于降低并发读把旧值回填缓存的概率。**  
流程是先删缓存，更新数据库，再延迟一小段时间删一次缓存。  
它是概率性补救方案，不是强一致方案。

### 5.2 详细解释版

延迟双删针对的是写过程中并发读读到旧 DB 并回填缓存的问题。第二次删除用于清掉这段时间可能被回填的旧缓存。  
延迟时间要大于典型读 DB 和回填耗时，但不能太长。更可靠的方式是“更新 DB 后删除缓存 + 删除失败重试 + binlog/MQ 补偿”。

### 5.3 图示

```text
del cache -> update DB -> sleep 200ms -> del cache again
```

## 6. 缓存和数据库如何做到最终一致？（高难度）

### 6.1 最简练版

**最终一致靠主链路失效缓存、失败重试、异步补偿和短 TTL 兜底。**  
数据库是权威数据源，缓存可以删除重建。  
关键是缓存删除失败不能静默丢，要有可靠补偿。

### 6.2 详细解释版

常见方案：应用更新 DB 后删除 Redis；删除失败写本地重试队列或 MQ；也可以监听 binlog，把变更事件投递给缓存清理服务；所有缓存设置合理 TTL 作为最终兜底。  
高要求场景可加版本号，读缓存时校验版本，避免旧缓存覆盖新数据。

### 6.3 图示

```text
DB 更新 -> 删除缓存
       -> 失败 -> MQ 补偿 -> 重试删除
       -> TTL 到期兜底
```

## 7. 缓存穿透是什么？（基本难度）

### 7.1 最简练版

**缓存穿透是请求的数据在缓存和数据库都不存在，导致每次都打到数据库。**  
典型场景是恶意请求不存在 ID，或业务参数异常。  
大量穿透会绕过缓存保护，直接压垮数据库。

### 7.2 详细解释版

普通未命中会查 DB 并回填缓存；穿透请求因为 DB 也没有数据，无法回填正常缓存，下次同样继续查 DB。  
需要在入口校验参数、缓存空值、使用布隆过滤器或风控限流。

### 7.3 例子

```text
GET /sku/-1
GET /sku/999999999999
```

## 8. 如何解决缓存穿透？（中等难度）

### 8.1 最简练版

**解决穿透常用参数校验、缓存空对象、布隆过滤器和限流。**  
缓存空对象适合不存在数据较少且可短期缓存的场景。  
布隆过滤器适合先判断 ID 是否可能存在，拦截大量非法请求。

### 8.2 详细解释版

空值缓存要设置短 TTL，避免数据库后来新增数据后长期查不到。布隆过滤器有误判但无漏判：可能把不存在判断为可能存在，但不会把存在判断为不存在。  
对攻击流量还要做 IP、用户、设备维度限流和黑名单。

### 8.3 表格

| 方案 | 优点 | 注意点 |
|---|---|---|
| 空值缓存 | 简单 | TTL 要短 |
| 布隆过滤器 | 内存低 | 有误判 |
| 参数校验 | 成本低 | 规则要完备 |

## 9. 布隆过滤器有什么优缺点？（中等难度）

### 9.1 最简练版

**布隆过滤器用位数组和多个哈希函数判断元素是否可能存在。**  
优点是内存占用很低、查询快；缺点是有误判，通常不支持直接删除。  
它适合拦截明显不存在的 key，保护数据库。

### 9.2 详细解释版

插入元素时，多个哈希函数把不同 bit 置 1；查询时这些 bit 都为 1 则“可能存在”，有任意 bit 为 0 则“一定不存在”。  
误判率受位数组大小、哈希函数数量、元素数量影响。删除场景可用计数布隆过滤器，但复杂度和内存会上升。

### 9.3 图示

```text
key -> hash1/hash2/hash3 -> bit array
查询：有 0 -> 一定不存在；全 1 -> 可能存在
```

## 10. 缓存击穿是什么？（基本难度）

### 10.1 最简练版

**缓存击穿是某个热点 key 失效瞬间，大量请求同时打到数据库。**  
它和穿透不同，击穿的数据本身存在，只是缓存突然没了。  
常见于热点商品、活动配置、明星内容。

### 10.2 详细解释版

击穿的关键是“热点 + 失效”。平时 Redis 扛住了请求，一旦 key 到期，所有并发都发现未命中并查 DB，造成数据库瞬时压力。  
要用互斥重建、逻辑过期、热点永不过期或后台刷新。

### 10.3 图示

```text
热点 key 到期
10000 个请求 miss -> DB 被同时查询
```

## 11. 如何解决缓存击穿？（中等难度）

### 11.1 最简练版

**解决击穿常用互斥锁、逻辑过期、热点 key 不设置物理过期和后台刷新。**  
互斥锁让只有一个请求重建缓存，其他请求等待或返回旧值。  
逻辑过期可以先返回旧值，再异步刷新，降低尾延迟。

### 11.2 详细解释版

互斥锁简单但要处理锁超时、释放和重试；逻辑过期适合允许短暂旧数据的场景；热点永不过期要靠主动更新和兜底降级。  
重建缓存一定要限并发，否则 Redis 未命中会把数据库打穿。

### 11.3 例子

```text
miss -> setnx rebuild_lock
拿到锁：查 DB 重建缓存
没拿到：短暂等待/返回旧值/降级
```

## 12. 缓存雪崩是什么？（基本难度）

### 12.1 最简练版

**缓存雪崩是大量 key 同时失效，或 Redis 整体不可用，导致请求集中打到数据库。**  
它影响面更大，不只是单个热点 key。  
雪崩会引发数据库、线程池和下游服务连锁故障。

### 12.2 详细解释版

雪崩有两类：一类是 TTL 同时到期，另一类是缓存集群故障。前者用 TTL 随机化、分批预热、热点保护；后者用高可用架构、本地缓存、限流、熔断和降级。  
核心原则是缓存失效时不要让所有流量直接打穿到数据库。

### 12.3 图示

```text
大量 key 过期 -> Redis miss 激增 -> DB QPS 激增 -> 服务超时
```

## 13. 如何解决缓存雪崩？（中等难度）

### 13.1 最简练版

**解决雪崩要从过期时间、预热、高可用、限流熔断和降级兜底入手。**  
TTL 加随机抖动，避免同时过期。  
Redis 故障时，应用要限制打到数据库的流量，并提供降级结果。

### 13.2 详细解释版

针对批量过期：分散 TTL、定时刷新热点、分批预热。针对 Redis 故障：主从或 Cluster 高可用、客户端超时控制、本地缓存兜底、数据库保护限流、返回默认值或旧值。  
雪崩治理不是单点技术，而是整条链路的容量保护。

### 13.3 例子

```text
TTL = baseTTL + random(0, jitter)
```

## 14. 热点 key 过期瞬间怎么保护数据库？（高难度）

### 14.1 最简练版

**热点 key 过期瞬间要限制只有少量请求能重建缓存。**  
可以用互斥锁、逻辑过期、后台刷新和本地缓存旧值。  
对核心热点，还应提前识别并主动续期或预热。

### 14.2 详细解释版

推荐方案是逻辑过期：缓存 value 中带 expireAt，物理 TTL 设置较长；读到逻辑过期后先返回旧数据，再异步加锁刷新。这样用户请求不直接阻塞在 DB 上。  
如果数据强一致要求高，就用互斥锁同步重建，但要控制等待时间和失败降级。

### 14.3 图示

```text
读缓存 -> 逻辑未过期 -> 返回
       -> 逻辑过期 -> 返回旧值 + 异步刷新
```

## 15. 本地缓存和 Redis 缓存怎么配合？（中等难度）

### 15.1 最简练版

**本地缓存适合极热点、读多写少、允许短暂不一致的数据；Redis 作为集中式缓存提供共享数据源。**  
本地缓存延迟最低，但多实例一致性更难。  
常见做法是本地短 TTL + Redis 长 TTL + 变更通知。

### 15.2 详细解释版

本地缓存能减少 Redis 热点压力，但每个应用实例都有一份副本。数据变更时，可以通过消息、Redis Pub/Sub、配置中心通知各实例失效。  
适合配置、字典、热点商品摘要，不适合高频写、强一致、用户私有敏感状态。

### 15.3 图示

```text
App 本地缓存 -> Redis -> DB
最快           共享缓存  权威数据
```

## 16. 缓存预热怎么做？（中等难度）

### 16.1 最简练版

**缓存预热是在流量到来前，把热点数据提前加载到 Redis。**  
它适合活动开始、服务重启、缓存清空、版本发布。  
预热要分批限速，避免把数据库一次性打满。

### 16.2 详细解释版

预热数据来源可以是历史访问排行、运营配置、活动商品列表、搜索热榜。流程通常是生成热点清单、批量查 DB、写 Redis、校验命中率。  
预热必须可重试、可限流、可观察，不能在上线瞬间全量扫库。

### 16.3 例子

```text
热榜 TOP 10000 -> 分批查 DB -> pipeline 写 Redis -> 验证命中率
```

## 17. 缓存降级怎么做？（高难度）

### 17.1 最简练版

**缓存降级是在 Redis 异常或数据库压力过高时，保护核心链路不被拖垮。**  
可以返回本地旧值、默认值、静态页，或关闭非核心功能。  
同时要限流到数据库，避免缓存故障变成数据库故障。

### 17.2 详细解释版

降级策略包括：Redis 超时快速失败、本地缓存兜底、热点数据快照、只保留核心字段、非核心接口熔断、数据库令牌桶限流。  
要提前定义哪些业务能降级、降级文案是什么、恢复条件是什么，并通过演练验证。

### 17.3 图示

```text
Redis 超时 -> 本地旧值
          -> 无旧值 -> 默认值/静态内容
          -> 限制 DB 回源
```

## 18. 缓存命中率低怎么分析？（高难度）

### 18.1 最简练版

**命中率低要从 key 设计、TTL、访问分布、容量淘汰、穿透和预热不足排查。**  
先区分是整体低、某业务低，还是某时间段低。  
再看未命中原因：不存在、过期、被淘汰、key 不一致或缓存未写入。

### 18.2 详细解释版

排查路径：看 Redis `keyspace_hits/misses`，按业务埋点统计命中率，抽样未命中 key，检查 TTL 和淘汰指标，确认写缓存流程是否失败。  
如果热点集中但命中低，可能是 key 拼接不稳定；如果 miss 后 DB 压力高，可能是穿透；如果高峰时命中下降，可能是内存淘汰或雪崩。

### 18.3 例子

```bash
info stats
info memory
info keyspace
```

## 19. 商品详情页缓存如何设计？（高难度）

### 19.1 最简练版

**商品详情缓存要拆分动静数据，热点商品做多级缓存，更新通过删除或消息失效。**  
基础信息、图片、描述可以长缓存；价格、库存、促销要短缓存或实时查询。  
还要防穿透、击穿、雪崩，并准备降级页面。

### 19.2 详细解释版

设计可以分层：CDN（Content Delivery Network，内容分发网络）缓存静态资源，本地缓存热点摘要，Redis 缓存详情聚合，DB 保存权威数据。  
key 按商品 ID 和版本设计，TTL 加抖动。更新商品时先更新 DB，再删除 Redis 和本地缓存；热点商品活动前预热。库存不要只依赖详情缓存，要有单独的库存链路。

### 19.3 图示

```text
CDN 静态资源
  |
App 本地热点摘要
  |
Redis 商品详情
  |
DB 权威数据
```

## 20. 库存扣减能完全依赖 Redis 吗？（高难度）

### 20.1 最简练版

**库存扣减可以用 Redis 做高并发前置扣减，但不能忽略数据库最终落账和对账。**  
Redis 适合快速判断和限流，数据库负责权威库存和订单状态。  
关键是防超卖、幂等、补偿和异常恢复。

### 20.2 详细解释版

常见方案是活动开始前把可售库存加载到 Redis，用 Lua 原子扣减；扣减成功后创建订单或发送 MQ 异步落库；落库失败要回补库存或关闭订单。  
如果 Redis 崩溃、消息丢失、重复消费、订单取消，都需要补偿链路。最终要以数据库订单和库存流水对账。

### 20.3 图示

```text
请求 -> Lua 扣 Redis 库存 -> MQ -> DB 落订单/库存流水
                         -> 失败补偿/对账
```
