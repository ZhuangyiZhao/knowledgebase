## 1. HTTP/1.1、HTTP/2、HTTP/3 最大的差异是什么？

### 1.1 面试题

HTTP（Hypertext Transfer Protocol，超文本传输协议）/1.1、HTTP/2、HTTP/3 的核心差异是什么？为什么协议要不断演进？

### 1.2 最简练版

**三者最核心的差异，在于数据传输组织方式和解决队头阻塞（Head-of-Line Blocking，队头阻塞）的能力不一样。**  
HTTP/1.1 是文本协议，通常一个连接里请求响应基本串行，靠长连接和管线化提升有限。  
HTTP/2 改成二进制分帧，并支持多路复用，解决了 HTTP 层面的并发效率问题，但底层还是 TCP，所以丢一个包仍可能卡住该连接上的所有流。  
HTTP/3 则把传输基础换成 QUIC，基于 UDP（User Datagram Protocol，用户数据报协议）在用户态实现可靠传输和多路复用，把不同流的丢包影响隔离开。  
所以演进路线可以概括成：**HTTP/1.1 优化连接复用，HTTP/2 优化应用层复用，HTTP/3 进一步优化传输层复用。**

### 1.3 详细解释版

面试可以这么说：

HTTP 协议演进，本质上是在解决三个问题：

1. **怎么减少连接建立和重复传输的开销。**
2. **怎么提高一个连接上的并发能力。**
3. **怎么降低网络抖动和丢包对整条连接的影响。**

HTTP/1.1 的主要改进是长连接和请求复用能力增强，但它仍然更像“一个连接里排队办事”，并发效率不高。

HTTP/2 往前走了一步，它把 HTTP 消息拆成更细粒度的 Frame（帧），允许多个 Stream（流）交错发送，所以从协议层看，请求不需要再一个个串行排队。

但 HTTP/2 的问题在于：它虽然在 **HTTP 层** 做到了多路复用，底层仍然依赖 TCP。TCP 需要保证字节流有序交付，只要其中一个 TCP 包丢了，后面的数据即使到了，也得等重传，结果就是多个 HTTP 流会一起被拖住。

HTTP/3 再往前走一步，不再把可靠传输完全交给 TCP，而是使用 QUIC：

- 基于 UDP 承载，避免被内核 TCP 协议栈限制演进速度。
- 在用户态实现重传、拥塞控制、流控、多路复用。
- 每个 Stream 独立确认和重传，降低单个丢包对其他流的影响。
- 把 TLS（Transport Layer Security，传输层安全协议）1.3 握手融合进连接建立过程，进一步减少时延。

所以你可以把三者理解成：

- HTTP/1.1：解决“少建连接”。
- HTTP/2：解决“一个连接里并发多个请求”。
- HTTP/3：解决“多路复用下 TCP 队头阻塞”。

### 1.4 图示

```text
演进主线：

HTTP/1.1
  |
  |  长连接 / Keep-Alive
  v
HTTP/2
  |
  |  二进制分帧 / 多路复用 / 头部压缩
  v
HTTP/3
  |
  |  QUIC / 基于 UDP / 传输层级别流隔离
  v
降低时延 + 降低队头阻塞影响 + 提升弱网体验
```

### 1.5 发布时间怎么记？

面试里可以直接这么说：

- `HTTP/1.1`：**1997 年**首次标准化，对应 `RFC 2068`；**1999 年** `RFC 2616` 成为更常被引用的 `HTTP/1.1` 标准。
- `HTTP/2`：**2015 年**发布标准，对应 `RFC 7540`。
- `HTTP/3`：**2022 年**发布标准，对应 `RFC 9114`。

速记版：

```text
HTTP/1.1：1997
HTTP/2：2015
HTTP/3：2022
```

### 1.6 一句话背诵

**HTTP/1.1 主要解决连接复用，HTTP/2 主要解决应用层多路复用，HTTP/3 主要解决 TCP 队头阻塞和弱网下的连接时延问题。**

## 2. HTTP/1.1 的特点和问题是什么？

### 2.1 面试题

HTTP/1.1 相比早期版本做了哪些优化？它的主要瓶颈是什么？

### 2.2 最简练版

**HTTP/1.1 的核心特点是文本协议、默认支持长连接、请求和响应语义简单，兼容性非常好。**  
它比 HTTP/1.0 少了频繁建连的成本，所以实际工程里普及很广。  
但 HTTP/1.1 的问题也很明显：一个连接上的请求处理能力有限，请求多了常常需要开多个 TCP 连接。  
另外它有队头阻塞问题，前一个请求慢，后面的请求就容易跟着等。  
所以 HTTP/1.1 的本质瓶颈是：**连接利用率低，并发能力弱。**

### 2.3 详细解释版

HTTP/1.1 的几个关键点：

1. **文本格式，可读性强。**  
   报文头和报文体结构清晰，调试方便，但编码体积更大，解析效率也不如二进制协议。

2. **默认长连接。**  
   通过 `Connection: keep-alive` 或默认持久连接减少反复 TCP 握手，降低延迟和服务端资源消耗。

3. **支持管线化（Pipelining）。**  
   客户端可以连续发多个请求，但响应仍要求按顺序返回。  
   这个特性理论上能提高吞吐，但因为代理兼容性差、服务端实现复杂、队头阻塞明显，生产环境使用并不广。

4. **通常通过多连接并发来弥补单连接不足。**  
   浏览器会对同一个域名建立多个 TCP 连接，让资源并发下载。

它的典型问题有三个：

1. **应用层队头阻塞。**  
   一个请求响应慢，后面的请求可能排队。

2. **需要更多 TCP 连接。**  
   为了提高并发，客户端常常不得不开多个连接，这会带来更多握手、更多内存占用、更多拥塞控制状态。

3. **Header 冗余传输。**  
   比如 Cookie、User-Agent、鉴权信息经常在每个请求里重复携带，浪费带宽。

线上角度可以补一句：

在高并发 Web 场景里，HTTP/1.1 的瓶颈常常不是业务逻辑本身，而是**连接数膨胀、TLS 握手成本、慢请求阻塞和重复 Header 带来的额外开销**。

### 2.4 图示

```text
HTTP/1.1 一个连接上的典型情况：

Client                          Server
  | ---- Request A ----------->   |
  | <--- Response A -----------   |
  | ---- Request B ----------->   |
  | <--- Response B -----------   |
  | ---- Request C ----------->   |
  | <--- Response C -----------   |

特点：整体更接近串行
```

如果为了并发只能这么做：

```text
Client
  |--- TCP Conn 1 ---> Request A
  |--- TCP Conn 2 ---> Request B
  |--- TCP Conn 3 ---> Request C

问题：连接数量上升，握手和维护成本变高
```

### 2.5 Header 冗余传输是什么意思？

**Header 冗余传输**，就是指多个 HTTP/1.1 请求里，经常会重复携带大量相同的请求头，每次都要原样再传一遍，浪费带宽和解析成本。

面试里可以这么说：

HTTP/1.1 是文本协议，请求头不会自动压缩。  
如果一个页面要加载很多资源，比如 HTML、CSS、JS、图片，这些请求往往都会重复带上差不多的 Header，比如：

- `Host`
- `Cookie`
- `User-Agent`
- `Authorization`
- `Accept`

这些字段在几十个请求里可能几乎一样，但 HTTP/1.1 还是每次都完整传输一次，这就叫 Header 冗余传输。

举个例子，用户访问首页时，浏览器可能连续发 3 个请求：

```text
GET /index.html HTTP/1.1
Host: example.com
Cookie: uid=1001; session=abcd1234; theme=dark
User-Agent: Chrome/...

GET /app.js HTTP/1.1
Host: example.com
Cookie: uid=1001; session=abcd1234; theme=dark
User-Agent: Chrome/...

GET /logo.png HTTP/1.1
Host: example.com
Cookie: uid=1001; session=abcd1234; theme=dark
User-Agent: Chrome/...
```

你会发现真正变的可能只有请求路径：

- `/index.html`
- `/app.js`
- `/logo.png`

但像 `Cookie`、`Host`、`User-Agent` 这些头部却反复传了很多次。  
如果 `Cookie` 很长，或者有鉴权信息，这种重复开销会更明显。

这也是为什么 HTTP/2 要引入 HPACK（Header Compression for HTTP/2，HTTP/2 头部压缩机制），HTTP/3 要引入 QPACK（HTTP/3 头部压缩机制）：  
**核心就是减少头部重复传输带来的带宽浪费和解析成本。**

### 2.6 常见误区

**误区 1：HTTP/1.1 没有长连接。**  
不对，HTTP/1.1 的一个重要改进就是默认持久连接。

**误区 2：HTTP/1.1 完全不能并发。**  
也不对，它可以通过多连接并发，也有管线化机制，只是整体效率和体验不如 HTTP/2。

**误区 3：HTTP/1.1 的队头阻塞和 TCP 队头阻塞是同一个问题。**  
不是。HTTP/1.1 更典型的是应用层请求响应顺序导致的阻塞；TCP 队头阻塞是传输层有序交付带来的阻塞。

### 2.7 一句话背诵

**HTTP/1.1 靠长连接降低建连成本，但单连接并发能力弱，通常要靠多 TCP 连接弥补，所以效率瓶颈主要在连接复用能力不足。**

## 3. HTTP/2 主要解决了什么问题？

### 3.1 面试题

HTTP/2 相比 HTTP/1.1 有哪些核心改进？它有没有彻底解决队头阻塞？

### 3.2 最简练版

**HTTP/2 的核心价值是把 HTTP 从文本协议升级成二进制分帧协议，并在一个 TCP 连接上实现多路复用。**  
这样多个请求和响应可以交错发送，不再像 HTTP/1.1 那样严重依赖多连接并发。  
同时它引入了头部压缩，减少重复 Header 传输开销，还支持服务端推送。  
但 HTTP/2 没有彻底解决阻塞问题，因为底层还是 TCP，只要一个包丢失，TCP 还是会卡住后续数据交付。  
所以它解决的是 **HTTP 层面的队头阻塞**，不是 **TCP 层面的队头阻塞**。

### 3.3 详细解释版

HTTP/2 的主要改进可以从四点来讲：

1. **二进制分帧。**  
   HTTP/1.1 的请求报文是整块文本；HTTP/2 会把消息拆成更小的 Frame。  
   常见帧包括 HEADERS、DATA、SETTINGS 等，解析效率更高，也更适合复用。

2. **多路复用。**  
   每个请求响应对应一个 Stream，不同 Stream 的帧可以在一个连接上交错发送。  
   这就避免了“必须等上一个请求彻底结束，后一个请求才能发”的限制。

3. **头部压缩。**  
   通过 HPACK（Header Compression for HTTP/2，HTTP/2 头部压缩机制）减少重复 Header 传输，尤其适合 Cookie 和鉴权信息很多的场景。

4. **服务端推送。**  
   服务端可以在客户端还没请求时主动推送资源。  
   但实际生产里使用并不算广，因为缓存命中判断复杂，也容易带来无效推送。

为什么它还是有问题？

因为 HTTP/2 的多路复用建立在 TCP 单连接之上，而 TCP 是字节流协议，必须保证有序交付。  
如果一个 TCP Segment（报文段）丢了，内核不会把后续字节直接交给上层，而是等丢失部分重传成功。  
这样一来，即使后面是另一个 Stream 的数据，也会一起被卡住。

这就是为什么很多人会说：

**HTTP/2 解决了应用层队头阻塞，但没有解决传输层队头阻塞。**

工程实践里，HTTP/2 很适合：

- 网关到客户端的大量小请求并发。
- 微服务之间大量短报文 RPC。
- TLS 普及场景下减少连接数和握手压力。

但在高丢包、弱网、移动网络切换场景下，它的收益会被 TCP 队头阻塞部分抵消。

### 3.4 图示

```text
HTTP/2 一个连接上多路复用：

TCP Connection
  |
  +-- Stream 1: HEADERS -> DATA -> DATA
  +-- Stream 3: HEADERS -> DATA
  +-- Stream 5: HEADERS -> DATA -> DATA

实际发送可能交错：

[S1-H][S3-H][S1-D][S5-H][S3-D][S5-D][S1-D]
```

### 3.5 为什么 HTTP/2 仍然会有队头阻塞？

因为 HTTP/2 的 Stream 只是 **HTTP 语义层的流**，不是 **传输层独立流**。

在 TCP 看来，这些数据最终仍然只是同一条有序字节流：

```text
HTTP/2 多个 Stream
        |
        v
   合并成一个 TCP 字节流
        |
        v
若中间某段丢失，则后续字节不能按序交付
```

这意味着：

- Stream 1 丢了一个包。
- Stream 3 的后续数据即使已经到达。
- 也可能因为 TCP 等待重传而暂时不能交付给 HTTP/2。

### 3.6 一句话背诵

**HTTP/2 的核心升级是二进制分帧、多路复用和头部压缩，它解决了 HTTP/1.1 单连接并发差的问题，但底层还是 TCP，所以仍然会受到传输层队头阻塞影响。**

## 4. HTTP/3 为什么要基于 QUIC？

### 4.1 面试题

HTTP/3 为什么不用 TCP，而要跑在 QUIC 上？它解决了什么实际问题？

### 4.2 最简练版

**HTTP/3 基于 QUIC，主要是为了绕开 TCP 的传输层队头阻塞，并进一步减少连接建立时延。**  
QUIC 跑在 UDP 之上，但自己在用户态实现了可靠传输、拥塞控制、流量控制和多路复用。  
这样每个流都能独立确认和重传，一个流丢包时，不会像 TCP 那样把同连接里的其他流一起卡住。  
同时 HTTP/3 通常结合 TLS 1.3，把安全握手和传输握手融合，首包时延更低，还支持连接迁移。  
所以它特别适合移动网络、弱网抖动和高并发多资源加载场景。

### 4.3 QUIC 是什么？

**QUIC 是一个基于 UDP 的传输层协议，用来提供可靠传输、多路复用、拥塞控制、流量控制、连接迁移等能力，HTTP/3 就是运行在 QUIC 之上的。**

你可以把它理解成：

- TCP 负责的很多事，比如可靠传输、重传、拥塞控制，QUIC 也做。
- TLS 负责的安全握手，QUIC 也和它深度结合。
- 但 QUIC 比 TCP 更适合多流并发和移动网络切换。

所以面试里一句话讲本质就是：

**QUIC 不是简单的 UDP 封装，而是在 UDP 之上重新实现了一套更适合现代网络场景的可靠传输机制。**

### 4.4 详细解释版

面试里可以先说结论：

**HTTP/3 不是简单给 HTTP/2 换个壳，而是把底层传输能力从 TCP 内核协议栈搬到了 QUIC 用户态协议栈里。**

为什么这么做？

#### 4.3.1 原因 1：解决 TCP 队头阻塞

TCP 以“字节流有序交付”为核心设计，一段丢了，后面的数据即使到了也要等。  
这对单流传输很合理，但对 HTTP/2 这种多流并发场景并不友好。

QUIC 的思路是：

- 连接里可以有多个独立 Stream。
- 每个 Stream 有自己的偏移和状态。
- 某个 Stream 的包丢了，只影响该 Stream 重传。
- 其他 Stream 可以继续被处理。

#### 4.3.2 原因 2：减少握手时延

传统 HTTPS 往往涉及：

1. TCP 三次握手。
2. TLS 握手。
3. 之后才开始传输 HTTP 数据。

而 QUIC 把传输握手和 TLS 1.3 握手做了更深融合，可以做到更低的首包时延，某些复用场景还能支持 0-RTT（Zero Round Trip Time，零往返时延）恢复。

#### 4.3.3 原因 3：协议演进更灵活

TCP 在内核里，升级和灰度成本高。  
QUIC 在用户态实现，客户端和服务端都更容易快速演进，比如调整拥塞控制算法、丢包恢复策略、ACK 行为等。

#### 4.3.4 原因 4：支持连接迁移

TCP 连接通常依赖四元组：

```text
源 IP + 源端口 + 目标 IP + 目标端口
```

移动端从 Wi-Fi 切到 4G，IP 一变，TCP 通常就断了。  
QUIC 使用 Connection ID（连接 ID）标识连接，网络路径变化时可以继续保持连接，这对移动场景很有价值。

线上价值可以这么讲：

- 页面资源多、首屏敏感时，HTTP/3 可能降低加载时延。
- 弱网和抖动环境里，HTTP/3 更能体现稳定性优势。
- 对音视频、实时互动、边缘接入这类业务，HTTP/3/QUIC 的收益通常更明显。

### 4.5 图示

```text
HTTP/3 / QUIC 结构：

HTTP/3
  |
QUIC
  |-- Stream 1
  |-- Stream 3
  |-- Stream 5
  |
UDP
  |
IP
```

丢包影响示意：

```text
HTTP/2 over TCP
  Stream 1 丢包
      |
      v
TCP 等待重传
      |
      v
Stream 3 / Stream 5 也可能一起受阻

HTTP/3 over QUIC
  Stream 1 丢包
      |
      v
只重传 Stream 1 对应数据
      |
      v
Stream 3 / Stream 5 可以继续推进
```

### 4.6 HTTP/3 带来的工程价值

1. **降低首包时延。**
2. **弱网下表现更稳。**
3. **多资源并发加载时更不容易相互拖累。**
4. **移动网络切换时更容易保持连接。**
5. **协议栈在用户态，升级更灵活。**

但也要补一句取舍：

- QUIC 实现复杂度更高。
- UDP 在部分中间设备上兼容性可能不如 TCP。
- CPU 开销和调优成本也可能更高。

所以不是所有业务都必须上 HTTP/3。

### 4.7 一句话背诵

**HTTP/3 通过 QUIC 把多路复用下的流隔离做到传输层，避免 TCP 队头阻塞，并把握手、安全、迁移能力一起优化了。**

## 5. 三者核心对比表

### 5.1 最简练版

如果面试官让你快速对比，可以直接说：

- HTTP/1.1：文本协议，长连接，单连接并发差，常靠多连接提升吞吐。
- HTTP/2：二进制分帧，多路复用，头部压缩，解决 HTTP 层并发问题，但仍受 TCP 队头阻塞影响。
- HTTP/3：基于 QUIC，传输层流隔离，握手更快，弱网体验更好。

### 5.2 详细对比表

| 对比项 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| 报文格式 | 文本 | 二进制分帧 | 二进制分帧 |
| 底层传输 | TCP | TCP | QUIC over UDP |
| 连接复用 | 长连接 | 长连接 | 长连接 |
| 多路复用 | 基本没有 | 支持 | 支持 |
| Header 压缩 | 无 | HPACK（HTTP/2 头部压缩机制） | QPACK（HTTP/3 头部压缩机制） |
| 队头阻塞 | 应用层明显 | 无 HTTP 层阻塞，但有 TCP 层阻塞 | 大幅缓解传输层阻塞 |
| 握手时延 | TCP + TLS | TCP + TLS | QUIC + TLS 1.3 融合更紧 |
| 连接迁移 | 不支持 | 不支持 | 支持 |
| 典型优势 | 简单、兼容性好 | 单连接并发强、头部更省 | 弱网更稳、首包更快、流隔离更好 |
| 典型问题 | 多连接成本高 | 仍受 TCP 丢包影响 | 实现复杂、UDP 兼容与调优更复杂 |

### 5.3 图示

```text
对比总览：

HTTP/1.1  = 文本 + 长连接 + 多连接并发补救
HTTP/2    = 二进制分帧 + 单连接多路复用 + 头部压缩
HTTP/3    = QUIC + 传输层流隔离 + 更低握手时延 + 连接迁移
```

## 6. 面试高频追问

### 6.1 追问 1：HTTP/2 已经多路复用了，为什么还要 HTTP/3？

#### 6.1.1 最简练版

**因为 HTTP/2 的多路复用只做到应用层，底层还是 TCP。**  
TCP 一旦丢包，后续字节都要等重传，多个流仍然可能一起卡住。  
HTTP/3 则把流隔离能力下沉到 QUIC，单个流丢包不会拖住整个连接。  
所以 HTTP/3 不是重复造轮子，而是进一步解决 HTTP/2 没解决完的问题。

#### 6.1.2 图示

```text
HTTP/2：多路复用在 HTTP 层
HTTP/3：多路复用 + 丢包隔离在传输层
```

### 6.2 追问 2：HTTP/3 为什么不用 TCP？

#### 6.2.1 最简练版

**因为 TCP 的核心语义是有序字节流交付，这和多流并发场景天然有冲突。**  
HTTP/3 需要的是“连接可靠，但流之间尽量独立”，这更适合由 QUIC 自己实现。  
另外 QUIC 在用户态，更容易快速迭代拥塞控制、重传和握手逻辑。  
所以它不是简单嫌弃 TCP，而是 TCP 的抽象边界已经不适合 HTTP/3 的目标了。

#### 6.2.2 图示

```text
TCP：可靠 + 有序字节流
QUIC：可靠 + 多流独立 + 用户态可演进
```

### 6.3 追问 3：HTTP/3 一定比 HTTP/2 快吗？

#### 6.3.1 最简练版

**不一定。**  
如果网络质量很好、链路稳定、连接复用充分，HTTP/2 也可能已经足够快。  
HTTP/3 的优势主要体现在高丢包、弱网、移动切网、多资源并发加载等场景。  
而且 QUIC 的 CPU 成本和运维复杂度更高，所以最终还是要看业务场景和观测数据。

#### 6.3.2 图示

```text
稳定内网 / 数据中心：
HTTP/2 可能已经够用

弱网 / 移动网络 / 高频切换：
HTTP/3 优势更明显
```

### 6.4 追问 4：线上怎么选择 HTTP/1.1、HTTP/2、HTTP/3？

#### 6.4.1 最简练版

**一般原则是兼容性优先、收益驱动升级。**  
HTTP/1.1 兼容性最好，很多基础设施默认支持。  
HTTP/2 适合绝大多数现代 Web 和内部 RPC 场景，通常是当前主流。  
HTTP/3 更适合对首包时延、弱网体验、移动切换敏感的业务，但要评估中间链路兼容性、CPU 和运维成本。  
所以不是新协议一定全量替换，而是按场景逐步灰度。

#### 6.4.2 图示

```text
选择思路：

先看兼容性
  |
  v
再看收益是否主要来自：
  - 连接时延
  - 丢包隔离
  - 弱网体验
  - 移动迁移
  |
  v
最后做灰度验证和指标对比
```

### 6.5 追问 5：字节流和文本协议、二进制分帧到底是什么关系？

#### 6.5.1 最简练版

**字节流说的是“底层怎么传”，文本协议和二进制分帧说的是“上层怎么组织数据”。**  
TCP 提供的是字节流，也就是它只保证一串字节按顺序到达，并不关心这些字节在业务上是一行文本还是一个帧。  
`HTTP/1.1` 是文本协议，你抓包能直接看到 `GET /index.html HTTP/1.1`、`Host: example.com` 这种文本内容。  
`HTTP/2` 虽然底层还是跑在 TCP 字节流上，但上层不再按文本报文组织，而是按二进制 Frame（帧）来组织。  
所以不要把“字节流”和“文本协议”当成一组对立概念，它们不在一个层面。

#### 6.5.2 详细解释版

面试可以这么说：

**字节流**，强调的是传输抽象。  
比如 TCP 只告诉上层：“我会把这串字节可靠、有序地给你”，但它不关心：

- 这串字节是不是一行 HTTP 文本。
- 是不是一个完整请求。
- 是不是某个业务对象。

所以 TCP 没有消息边界，应用层要自己定义边界。

**文本协议**，强调的是应用层协议内容是按可读文本来组织的。  
HTTP/1.1 就是典型例子，请求和响应头本质上都是字符串：

```text
GET /index.html HTTP/1.1
Host: example.com
Connection: keep-alive
```

人眼能直接看懂，所以调试方便，但体积偏大、解析效率也一般。

**二进制分帧**，强调的是应用层协议把消息拆成多个二进制帧来传。  
HTTP/2 就是这样，一个请求可能会被拆成：

- `HEADERS` 帧
- `DATA` 帧
- `SETTINGS` 帧

这些帧还能和别的请求的帧交错发送，所以更适合多路复用。

在 HTTP 里你可以这样理解差异：

```text
HTTP/1.1：
看到的是“完整文本报文”

HTTP/2：
看到的是“多个二进制帧组成的消息”
```

所以更准确的说法是：

- TCP：字节流传输。
- HTTP/1.1：运行在字节流之上的文本协议。
- HTTP/2：运行在字节流之上的二进制分帧协议。
- HTTP/3：运行在 QUIC 流之上的二进制分帧协议。

#### 6.5.3 图示

```text
分层理解：

HTTP/1.1
  文本协议
     |
     v
TCP
  字节流

HTTP/2
  二进制分帧
     |
     v
TCP
  字节流

HTTP/3
  二进制分帧
     |
     v
QUIC
  多流 + 可靠传输
```

## 7. 面试口述版

### 7.1 30 秒回答

面试可以这么说：

**HTTP/1.1、HTTP/2、HTTP/3 的核心差异，主要在并发传输和阻塞处理能力上。**  
HTTP/1.1 主要靠长连接降低建连成本，但单连接并发差，很多时候得靠多 TCP 连接补。  
HTTP/2 通过二进制分帧和多路复用，让多个请求可以复用一个 TCP 连接，但底层还是 TCP，所以丢包时仍会有传输层队头阻塞。  
HTTP/3 则基于 QUIC，把多流隔离做到传输层，单个流丢包不会拖住其他流，同时握手更快、弱网体验更好。

### 7.2 90 秒回答

面试可以这么说：

HTTP 协议的演进，本质上是在不断提高连接利用率和并发效率。  
HTTP/1.1 相比更早版本最大的进步是长连接，但它本质上还是文本协议，单连接上的请求响应能力有限，所以浏览器常常需要开多个 TCP 连接来并发拉资源，这会带来更多握手和连接维护成本。

HTTP/2 的升级比较大，它把 HTTP 改成了二进制分帧协议，并且支持多路复用，也就是多个请求响应可以在同一个 TCP 连接里交错发送，同时还通过 HPACK 做头部压缩，所以在大量小请求场景下效率高很多。

但 HTTP/2 没有彻底解决阻塞问题，因为底层还是 TCP。TCP 是有序字节流，只要丢一个包，后面的字节即使到了也得等重传，所以多个 HTTP 流仍可能一起被卡住。

HTTP/3 就是为了解决这个问题，它基于 QUIC，跑在 UDP 上，但自己实现了可靠传输、流控、拥塞控制和多路复用。这样每个流都能独立确认和重传，单个流丢包不会影响其他流。同时它还把 TLS 1.3 握手融合得更紧，能降低首包时延，并支持连接迁移，所以在移动网络和弱网场景里优势会更明显。

## 8. 容易踩坑

1. **不要把 HTTP/2 多路复用和“彻底没有阻塞”画等号。**  
   它只是解决了 HTTP 层面的串行问题，没有解决 TCP 层阻塞。

2. **不要把 HTTP/3 理解成“不可靠”。**  
   它底层虽然用 UDP，但 QUIC 自己提供了可靠传输能力。

3. **不要说 HTTP/3 一定全面替代 HTTP/2。**  
   实际上要看业务场景、终端支持、中间链路和成本收益。

4. **不要忽略工程取舍。**  
   新协议带来的不仅是性能收益，也有实现复杂度、监控体系和调优成本。
