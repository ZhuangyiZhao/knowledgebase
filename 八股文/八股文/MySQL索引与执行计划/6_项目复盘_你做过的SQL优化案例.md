## 1. SQL 优化案例应该怎么讲？（中等难度）

### 1.1 最简练版

**SQL 优化案例要讲清楚问题背景、定位证据、优化动作、上线方式和最终收益。**  
不要只说“加索引”，要说明原 SQL 为什么慢，比如扫描行数大、索引不匹配、额外排序、深分页或回表多。  
方案要体现取舍，比如为什么用联合索引、为什么没有直接分库分表、怎么控制写入成本。  
结果最好用 P95、P99、慢日志数量、扫描行数、CPU（Central Processing Unit，中央处理器）和 I/O（Input/Output，输入输出）变化来支撑。

### 1.2 详细解释版

项目类回答可以按这个结构：

| 模块 | 要讲什么 | 关键点 |
|---|---|---|
| 背景 | 业务链路和表规模 | 哪个接口、多少数据、多少 QPS |
| 问题 | 现象和影响范围 | RT（Response Time，响应时间）、P95（95th Percentile，第 95 百分位延迟）、慢日志 |
| 定位 | 用什么证据确认瓶颈 | 慢日志、`EXPLAIN`、索引、数据分布 |
| 方案 | 做了哪些改动 | SQL 改写、联合索引、覆盖索引、分页优化 |
| 上线 | 怎么降低风险 | 预发验证、低峰 DDL、灰度、开关回退 |
| 结果 | 指标改善多少 | 扫描行数、延迟、CPU、慢日志下降 |
| 复盘 | 怎么防止复发 | SQL 评审、巡检、压测门禁、索引规范 |

容易丢分的表达：

- 只说“加了个索引”，没有原 SQL 和执行计划。
- 只讲本地执行变快，没有线上指标。
- 不提新增索引的写入成本和 DDL（Data Definition Language，数据定义语言）风险。
- 不讲异常回退和灰度。

### 1.3 例子

一句话开场可以这样说：

> 我做过一个商家订单列表的慢查询治理。这个接口在活动高峰期 P95 从 `150ms` 涨到 `1.8s`，慢日志里主要是按租户、状态和创建时间查订单的 SQL。定位后发现原来只走了低选择性的单列索引，扫描行数大，还存在额外排序。最后我通过 SQL 改写、联合索引、列表字段收敛和游标翻页，把扫描行数从几十万降到几百以内，接口 P95 稳定回到 `100ms` 到 `200ms`。

## 2. 订单列表慢查询案例怎么讲？（高难度）

### 2.1 最简练版

**这个案例可以讲成一个高频列表接口的访问路径优化。**  
原查询按 `tenant_id`、`status` 过滤，并按 `created_at` 倒序取最新订单，但索引只覆盖了 `status`，导致扫描大量订单后再过滤和排序。  
我按 `where + order by + limit` 设计了 `(tenant_id, status, created_at, id)` 联合索引，并把深分页改成游标翻页。  
上线后扫描行数、慢日志数量、DB CPU 和接口高分位延迟都有明显下降。

### 2.2 详细解释版

背景：

| 项目项 | 内容 |
|---|---|
| 业务 | 商家后台订单列表 |
| 表规模 | 订单表约 `300` 万到 `500` 万行 |
| 写入 | 活动期每天新增 `10` 万级订单 |
| 流量 | 高峰 QPS（Queries Per Second，每秒查询数）约 `800` |
| 现象 | 大租户列表页打开慢，高峰期抖动明显 |

原 SQL：

```sql
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc
limit 20;
```

原索引：

```sql
create index idx_status on order_tab(status);
create index idx_created_at on order_tab(created_at);
```

问题在于：

| 问题 | 影响 |
|---|---|
| `status` 区分度低 | 同一状态下订单很多，扫描范围大 |
| `tenant_id` 不在索引前缀 | 多租户过滤不能提前完成 |
| `created_at` 没接在过滤条件后 | 排序无法稳定复用索引 |
| 没有稳定排序键 | 相同 `created_at` 下分页可能抖动 |

优化索引：

```sql
create index idx_tenant_status_time_id
on order_tab(tenant_id, status, created_at, id);
```

优化 SQL：

```sql
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc, id desc
limit 20;
```

如果存在深分页，把页码分页改成游标翻页：

```sql
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = ?
  and (
    created_at < ?
    or (created_at = ? and id < ?)
  )
order by created_at desc, id desc
limit 20;
```

### 2.3 例子

优化前执行计划可能是：

| type | key | rows | Extra |
|---|---|---:|---|
| ref | idx_status | 260000 | Using where; Using filesort |

解释：

- `idx_status` 用了索引，但 `status` 选择性太低。
- `tenant_id` 只能在取出记录后过滤。
- `order by created_at desc` 没有复用索引顺序。
- 扫描 `26` 万行只返回 `20` 条，资源浪费很大。

优化后执行计划期望是：

| type | key | rows | Extra |
|---|---|---:|---|
| ref | idx_tenant_status_time_id | 300 | Using where |

回答时可以补一句：**这不是简单地从“没索引”变成“有索引”，而是让索引顺序贴合业务访问路径，把大范围扫描变成小范围有序读取。**

## 3. 定位过程怎么体现工程经验？（中等难度）

### 3.1 最简练版

**定位过程要体现证据链，而不是凭感觉改索引。**  
我会先从监控确认接口慢，再从慢日志找到 SQL 和真实参数，然后用 `EXPLAIN` 看访问类型、命中索引、扫描行数和额外排序。  
接着看表结构、已有索引和数据分布，判断为什么优化器会选这个计划。  
最后通过预发压测和灰度指标验证收益。

### 3.2 详细解释版

定位路径：

| 步骤 | 工具或命令 | 目的 |
|---|---|---|
| 看接口指标 | 监控面板 | 确认 RT、P95、P99、错误率 |
| 找慢 SQL | 慢日志、APM（Application Performance Monitoring，应用性能监控） | 确认 SQL、参数、调用频率 |
| 看执行计划 | `EXPLAIN`、`EXPLAIN ANALYZE` | 看 `type`、`key`、`rows`、`Extra` |
| 看索引 | `show index from order_tab` | 判断索引是否匹配查询 |
| 看数据分布 | 聚合查询或统计报表 | 判断大租户、热点状态、时间倾斜 |
| 验证方案 | 预发、灰度、压测 | 避免只优化个例 |

常用 SQL：

```sql
explain
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = 10001
  and status = 1
order by created_at desc
limit 20;

show index from order_tab;

select status, count(*)
from order_tab
group by status;

select count(*)
from order_tab
where tenant_id = 10001
  and status = 1;
```

### 3.3 例子

假设数据分布是：

| status | 含义 | 行数 |
|---:|---|---:|
| 1 | 已支付 | 2600000 |
| 2 | 已发货 | 800000 |
| 3 | 已取消 | 300000 |

如果查询经常是 `status = 1`，那么单列索引 `idx_status` 的选择性很差。  
进一步看某个大租户：

```sql
select count(*)
from order_tab
where tenant_id = 10001
  and status = 1;
```

结果有 `18` 万行，说明慢点不是偶发，而是“大租户 + 热门状态 + 最新排序”的典型组合。  
所以联合索引先放 `tenant_id`，再放 `status`，最后放 `created_at, id`，逻辑是先缩小租户，再缩小状态，再复用时间排序。

## 4. 方案设计怎么体现取舍？（高难度）

### 4.1 最简练版

**方案设计要讲清楚为什么这么改，以及它带来的代价。**  
这个案例优先选择联合索引和 SQL 改写，因为核心瓶颈是访问路径不合理，单表规模还可控。  
没有直接分库分表，是因为容量和写入还没到单库瓶颈，贸然拆分会引入跨分片查询和运维复杂度。  
同时要说明新索引会增加写入和存储成本，所以只覆盖高频小字段。

### 4.2 详细解释版

方案拆解：

| 方案 | 解决问题 | 代价 |
|---|---|---|
| 联合索引 `(tenant_id, status, created_at, id)` | 减少扫描、支持排序 | 增加写入维护和存储 |
| 列表字段收敛 | 减少回表和网络传输 | 需要接口字段治理 |
| 游标翻页 | 避免深分页大 offset | 不支持任意跳页 |
| 延迟关联 | 兼容页码跳转，减少回表 | offset 扫描仍存在 |
| 异步导出 | 避免同步大查询 | 链路复杂度上升 |

不直接分库分表的理由：

- 慢点来自索引顺序不匹配，不是单库容量绝对不足。
- 核心查询还能通过联合索引把扫描行数控制住。
- 分库分表会带来跨租户查询、全局排序、事务和迁移成本。
- 当前阶段用低成本优化能快速降低线上风险。

### 4.3 例子

如果有人追问“为什么不把 `amount` 也放进覆盖索引”，可以这样回答：

> 我会先看列表页是否高频需要 `amount`。如果这个字段很小且每次都返回，可以考虑把它放进覆盖索引，减少回表。但覆盖索引不是越宽越好，因为订单表写入频繁，每多一个索引列都会增加写入维护成本和索引页大小。所以我会先对比两版执行计划和写入压测，如果回表成本不是主要瓶颈，就不盲目扩宽索引。

两种索引取舍：

| 索引 | 优点 | 风险 |
|---|---|---|
| `(tenant_id, status, created_at, id)` | 较窄，写入成本低 | 查询 `amount` 可能回表 |
| `(tenant_id, status, created_at, id, amount)` | 可能覆盖列表查询 | 索引更宽，写入和缓存成本更高 |

## 5. 结果指标怎么说才可信？（中等难度）

### 5.1 最简练版

**结果指标要同时说数据库指标和业务接口指标。**  
数据库侧看扫描行数、慢日志数量、CPU、I/O、锁等待和连接数。  
接口侧看平均 RT、P95、P99、错误率和超时数量。  
最好说明优化前后的对比，以及观察窗口是灰度、全量上线后一天还是一周。

### 5.2 详细解释版

推荐指标：

| 指标 | 为什么重要 |
|---|---|
| `EXPLAIN rows` | 直接反映预估扫描量 |
| 实际扫描行数 | 比预估更接近真实成本 |
| 慢日志数量 | 判断慢查询是否整体下降 |
| DB CPU | 判断排序、扫描、过滤压力 |
| I/O | 判断是否减少读放大和回表 |
| 接口 P95 / P99 | 判断用户体感 |
| 写入 RT | 判断新增索引是否拖慢写链路 |

表达时不要只说“快了很多”，要尽量量化：

- P95 从多少降到多少。
- 慢日志每分钟从多少条降到多少条。
- `rows` 从多少降到多少。
- CPU 从多少降到多少。
- 写入是否有副作用。

### 5.3 例子

可以这样描述结果：

| 指标 | 优化前 | 优化后 |
|---|---:|---:|
| `EXPLAIN rows` | 260000 | 300 |
| 接口平均 RT | 650ms | 80ms |
| 接口 P95 | 1.8s | 180ms |
| 慢日志数量 | 每分钟 120 条 | 每分钟 5 条以内 |
| DB CPU | 80% 左右 | 45% 左右 |
| 写入 RT | 20ms | 24ms |

回答时可以补一句：**我也观察了写入延迟，新索引确实有一点写入成本，但涨幅可控，核心读链路收益明显，所以最终保留了这个索引。**

## 6. 常见追问怎么准备？（高难度）

### 6.1 最简练版

**常见追问主要围绕为什么这样设计、有没有副作用、线上出问题怎么办。**  
要准备好索引列顺序、覆盖索引取舍、深分页处理、DDL 风险、优化器不选索引和回滚方案。  
回答时要从数据分布、访问路径和稳定性出发，不要只背结论。  
能讲出压测、灰度、监控和复盘，会更像真实项目经历。

### 6.2 详细解释版

高频追问：

| 追问 | 回答要点 |
|---|---|
| 为什么索引是这个顺序？ | `tenant_id` 缩小租户，`status` 等值过滤，`created_at/id` 支持排序和游标 |
| 为什么不用分库分表？ | 当前瓶颈是索引路径，不是容量和写入上限 |
| 新索引有什么副作用？ | 写入、存储、Buffer Pool 压力、DDL 风险 |
| 优化器不走新索引怎么办？ | 看统计信息、数据分布、成本估算，谨慎使用 `force index` |
| 深分页怎么处理？ | 游标翻页优先，页码跳转用延迟关联 |
| 上线出问题怎么回退？ | 开关切旧 SQL，必要时下线新入口，索引删除要评估低峰执行 |

### 6.3 例子

追问：“如果加了索引后写入变慢怎么办？”

可以这样答：

> 我会先确认写入变慢是不是由新索引引起的，比如看上线前后写入 RT、InnoDB 行锁、redo 写入、CPU 和 I/O 变化。如果确认是索引维护成本导致，就评估这个索引是否过宽、是否和已有索引重复、是否只服务低频查询。如果读收益只覆盖少量后台场景，我会考虑删除或收窄索引；如果它服务核心高频链路，就看能否通过批量写入、削峰、冷热拆分或异步化来平衡成本。

追问：“为什么 `force index` 不能随便用？”

可以这样答：

> `force index` 会把优化器选择空间压窄，短期能绕过误判，但数据分布变化后可能反而变成坏计划。我通常先更新统计信息、分析数据倾斜和索引选择性，再判断是否需要临时指定索引。即使用，也要配合监控和回滚，不能当作长期默认方案。
