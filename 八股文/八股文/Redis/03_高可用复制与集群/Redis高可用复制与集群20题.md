## 1. Redis 主从复制解决什么问题？（基本难度）

### 1.1 最简练版

**主从复制用于提升可用性、读扩展和数据冗余。**  
主库负责写入，从库复制主库数据，可以承担读请求，也可以在主库故障时被提升为新主。  
但复制是异步为主，所以不能默认认为主从数据实时一致。

### 1.2 详细解释版

主从复制常用于读写分离、备份节点、故障切换基础。它降低单点风险，但会引入复制延迟、读旧数据、故障切换丢数据等问题。  
如果业务对一致性敏感，读请求不能随意打到从库，要按用户维度、写后读策略或版本校验处理。

### 1.3 图示

```text
Client 写 -> Master
              |
              v
           Replica1
           Replica2
```

## 2. Redis 全量复制流程是什么？（中等难度）

### 2.1 最简练版

**全量复制是从库第一次同步或无法增量同步时，从主库拿一份完整数据。**  
主库生成 RDB 快照发给从库，从库加载快照，再接收快照期间的增量写命令。  
全量复制成本高，会消耗 CPU、内存、磁盘和网络。

### 2.2 详细解释版

典型流程是：从库发起同步，主库判断需要全量复制，执行 `bgsave` 生成 RDB，同时把新写命令写入复制缓冲；RDB 传给从库后，从库清空旧数据并加载；最后主库发送缓冲的增量命令。  
大实例做全量复制会影响延迟，也可能让网络带宽打满，所以要避免频繁全量同步。

### 2.3 图示

```text
Replica -> PSYNC -> Master
Master  -> RDB 快照 -> Replica 加载
Master  -> 增量命令 -> Replica 追平
```

## 3. Redis 增量复制靠什么实现？（中等难度）

### 3.1 最简练版

**增量复制依赖复制偏移量、主库运行 ID 和复制积压缓冲区。**  
从库短暂断线后，带着旧偏移量重新连接。  
如果主库缓冲区还保留缺失命令，就补发增量；否则只能全量复制。

### 3.2 详细解释版

主从双方都维护复制 offset（Offset，偏移量）。主库把最近写命令保存在 replication backlog buffer（复制积压缓冲区）里。  
从库重连时用 `PSYNC runid offset` 请求续传。主库发现 runid 匹配且缺失部分仍在缓冲区，就返回 `CONTINUE`；否则返回 `FULLRESYNC`。

### 3.3 例子

```text
从库已复制到 offset=1000
断线期间主库写到 offset=1300
backlog 仍有 1001-1300 -> 增量补发
```

## 4. replication backlog 有什么作用？（中等难度）

### 4.1 最简练版

**replication backlog 是主库保存最近写命令的环形缓冲区，用于支持断线后的增量复制。**  
缓冲区太小，从库稍微落后就只能全量同步。  
缓冲区大小要结合写入速率和最大可接受断线时间设置。

### 4.2 详细解释版

backlog 是环形结构，新的写命令会覆盖旧命令。假设写入流量是 10MB/s，希望从库断线 60 秒内都能增量恢复，backlog 至少要大于 600MB，并留出余量。  
设置过小会导致频繁全量复制；设置过大会占用更多内存。

### 4.3 例子

```text
推荐估算：backlog >= 写入速率 * 容忍断线时间 * 安全系数
```

## 5. 主从复制为什么可能有延迟？（中等难度）

### 5.1 最简练版

**主从延迟来自网络、从库处理能力、主库写入压力、大 key、持久化和全量同步。**  
Redis 复制是命令流，从库需要接收、解析、执行。  
只要从库消费速度低于主库写入速度，就会积压延迟。

### 5.2 详细解释版

常见原因包括跨机房 RTT（Round-Trip Time，往返时延）高、从库 CPU 忙、大 key 写入、Lua 阻塞、磁盘 fsync 抖动、全量复制加载 RDB、输出缓冲区堆积。  
排查先看 `master_repl_offset` 和 `slave_repl_offset` 差值，再结合网络、CPU、慢日志和持久化指标。

### 5.3 例子

```bash
info replication
slowlog get 10
info persistence
```

## 6. Redis 读写分离有什么风险？（中等难度）

### 6.1 最简练版

**读写分离最大风险是读到旧数据。**  
写入主库后，从库可能还没复制到最新状态。  
对写后立即读、库存、权限、余额等场景，要读主库或做一致性保护。

### 6.2 详细解释版

读写分离适合读多写少、短暂不一致可接受的缓存场景。对于用户刚修改资料后马上读取，可以按用户维度短时间读主；对于强一致校验，要读数据库或主库；对于列表页缓存，可接受秒级延迟。  
客户端还要处理主从切换期间连接变更和从库不可用。

### 6.3 例子

```text
写资料成功 -> 30 秒内该用户读主库
其他普通读 -> 可读从库
```

## 7. wait 命令能保证强一致吗？（高难度）

### 7.1 最简练版

**`wait` 只能提高写入被从库确认的概率，不等于强一致。**  
它等待指定数量从库确认复制到当前写入偏移量。  
但确认后仍可能发生故障切换、网络分区或从库未被选为新主导致数据丢失。

### 7.2 详细解释版

`WAIT numreplicas timeout` 能让客户端等待至少 N 个副本确认收到写入，减少主库单点崩溃导致的数据丢失窗口。  
但 Redis 复制和故障切换没有基于多数派提交日志的强一致协议。若确认的副本没有成为新主，或者确认只是写入内存还未持久化，仍有风险。它是“增强可靠性”，不是 CP（Consistency and Partition tolerance，一致性和分区容错）系统。

### 7.3 例子

```bash
set order:1001 paid
wait 1 100
```

## 8. 哨兵 Sentinel 解决什么问题？（基本难度）

### 8.1 最简练版

**Sentinel 用于监控 Redis 主从、自动故障转移和通知客户端新主地址。**  
它能在主库不可用时选一个从库提升为新主。  
Sentinel 本身也应部署多个节点，避免单点判断。

### 8.2 详细解释版

Sentinel 负责监控、通知、自动 failover（故障转移）和配置更新。多个 Sentinel 通过投票确认主库客观下线，再由 leader 执行切换。  
客户端需要支持 Sentinel 协议，能从 Sentinel 获取当前主库地址，否则切换后仍可能写旧主。

### 8.3 图示

```text
Sentinel 集群 -> 监控 Master/Replica
Master 故障 -> 选 Replica -> 提升新 Master
```

## 9. Sentinel 如何判断主库下线？（中等难度）

### 9.1 最简练版

**Sentinel 先做主观下线，再通过多个 Sentinel 投票形成客观下线。**  
单个 Sentinel 超时认为主库不可达，是主观下线。  
达到 quorum（法定票数）后，才认为主库客观下线并触发切换。

### 9.2 详细解释版

主观下线可能由网络抖动、主库忙、Sentinel 自身故障导致。为了避免误判，需要多个 Sentinel 交叉确认。  
`down-after-milliseconds` 太小会增加误切换，太大则故障恢复慢。`quorum` 和 Sentinel 数量要按可用性要求配置。

### 9.3 例子

```text
S1 认为 Master 超时 -> 主观下线
S1/S2/S3 多数确认 -> 客观下线
```

## 10. Sentinel 故障切换流程是什么？（高难度）

### 10.1 最简练版

**故障切换流程是确认主库客观下线、选举 Sentinel leader、选择新主库、让其他从库复制新主、通知客户端。**  
切换期间可能有短暂不可写。  
如果旧主恢复，会被降级为新主的从库。

### 10.2 详细解释版

Sentinel leader 会在候选从库中选一个最合适的提升为主，执行 `SLAVEOF NO ONE`，然后让其他从库复制它。之后更新配置纪元并通知客户端。  
如果旧主在网络分区中仍接受写入，就会形成脑裂风险。生产要配合 `min-replicas-to-write` 等配置降低旧主继续写的概率。

### 10.3 图示

```text
发现故障 -> 投票确认 -> 选 leader -> 选新主 -> 重配从库 -> 通知客户端
```

## 11. Sentinel 选新主库看哪些因素？（高难度）

### 11.1 最简练版

**选新主库会考虑从库优先级、复制偏移量、运行状态和 runid 排序等因素。**  
优先选择状态健康、优先级高、数据更接近旧主的从库。  
复制偏移量越大，通常表示数据越新。

### 11.2 详细解释版

`replica-priority` 为 0 的从库不会被提升。候选中会过滤下线、断连时间太久、复制状态差的节点。之后比较优先级，再比较复制 offset，最后用 runid 做稳定排序。  
从工程角度看，应把机房、机器规格、业务重要性和复制延迟纳入优先级规划。

### 11.3 例子

```text
优先级高 + offset 最大 + 状态健康 -> 更适合成为新主
```

## 12. Redis 脑裂是什么？（高难度）

### 12.1 最简练版

**脑裂是集群中同时出现两个主库对外提供写服务。**  
通常发生在网络分区或故障切换期间，旧主没有及时停止写，新主已经被选出。  
分区恢复后，旧主会被降级，它在分区期间接受的写入可能丢失。

### 12.2 详细解释版

Redis Sentinel 是 AP（Availability and Partition tolerance，可用性和分区容错）倾向的高可用方案，不能像强一致系统那样严格阻止所有脑裂写入。  
客户端如果仍连接旧主，旧主也没有足够从库确认限制，就可能继续写。恢复后旧主同步新主数据，旧写入被覆盖。

### 12.3 图示

```text
网络分区：
ClientA -> 旧 Master 写
Sentinel -> 选 Replica 为新 Master
ClientB -> 新 Master 写
```

## 13. 如何降低脑裂导致的数据丢失？（高难度）

### 13.1 最简练版

**可以通过限制主库最少从库数量和最大复制延迟，降低旧主在孤岛中继续写的概率。**  
常用配置是 `min-replicas-to-write` 和 `min-replicas-max-lag`。  
客户端也要及时感知新主，避免长期写旧主。

### 13.2 详细解释版

当主库发现可用从库数量不足，或从库延迟超过阈值时，拒绝写入。这样旧主被网络隔离后，往往无法满足写入条件，从而减少脑裂写。  
但这会牺牲可用性：网络抖动时即使主库还活着，也可能拒绝写。因此配置要结合业务对丢数据和不可写的容忍度。

### 13.3 例子

```conf
min-replicas-to-write 1
min-replicas-max-lag 10
```

## 14. Redis Cluster 是怎么分片的？（基本难度）

### 14.1 最简练版

**Redis Cluster 用 16384 个 hash slot（哈希槽）做分片。**  
每个 key 通过 CRC16 算法映射到某个 slot，slot 再分配给具体主节点。  
扩容迁移本质是迁移 slot，而不是按 key 全局重哈希。

### 14.2 详细解释版

Cluster 中每个主节点负责一部分 slot，客户端需要知道 slot 到节点的映射。访问错误节点时，Redis 返回重定向，客户端更新路由缓存。  
这种设计让扩缩容以 slot 为单位，降低全局迁移成本，但多 key 操作必须在同一个 slot。

### 14.3 图示

```text
key -> CRC16 -> slot 0..16383 -> Redis 节点
```

## 15. hash slot 和一致性哈希有什么区别？（中等难度）

### 15.1 最简练版

**hash slot 是固定槽位再分配给节点；一致性哈希是节点映射到哈希环上。**  
Redis Cluster 迁移的是 slot，路由更清晰。  
一致性哈希常依赖虚拟节点平衡数据，节点变化时迁移相邻区间。

### 15.2 详细解释版

固定 16384 slot 让 Redis 能明确知道每个 slot 的归属，并通过 gossip 传播集群拓扑。扩容时只需要把部分 slot 从旧节点迁到新节点。  
一致性哈希更通用，适合客户端分片或缓存代理，但多 key 约束和路由维护方式不同。

### 15.3 表格

| 维度 | hash slot | 一致性哈希 |
|---|---|---|
| 单位 | 固定槽位 | 哈希环区间 |
| 扩容 | 迁移部分 slot | 迁移相邻区间 |
| Redis Cluster | 原生支持 | 非原生方案 |

## 16. moved 和 ask 重定向有什么区别？（中等难度）

### 16.1 最简练版

**`MOVED` 表示 slot 已经稳定迁到新节点，客户端应更新路由缓存。**  
`ASK` 表示 slot 正在迁移中，本次请求临时去目标节点。  
一个是永久路由变更，一个是迁移过程的临时跳转。

### 16.2 详细解释版

Cluster 扩缩容迁移 slot 时，部分 key 可能已在新节点，部分仍在旧节点。此时旧节点可能返回 `ASK`，客户端要先发 `ASKING` 再执行命令。  
如果收到 `MOVED`，说明 slot 归属已经变更，客户端应刷新 slot map。

### 16.3 例子

```text
MOVED 3999 10.0.0.2:6379 -> 更新路由
ASK 3999 10.0.0.2:6379   -> 本次临时请求
```

## 17. Cluster 中 multi-key 命令有什么限制？（中等难度）

### 17.1 最简练版

**Cluster 中多 key 命令要求所有 key 落在同一个 slot。**  
否则 Redis 无法在单节点上原子执行，会返回跨槽错误。  
可以用 hash tag（哈希标签）让相关 key 落到同一 slot。

### 17.2 详细解释版

Redis Cluster 不支持跨节点事务和跨 slot Lua 原子执行。`mget a b`、`sunion k1 k2` 这类命令只有同 slot 才能执行。  
hash tag 使用 `{}` 中的内容计算 slot，例如 `user:{1001}:profile` 和 `user:{1001}:cart` 会落到同一 slot。但滥用会造成数据倾斜。

### 17.3 例子

```text
user:{1001}:profile
user:{1001}:orders
```

## 18. Cluster 扩容和迁移 slot 怎么做？（高难度）

### 18.1 最简练版

**Cluster 扩容通常是加入新节点，再从旧节点迁移部分 slot 到新节点。**  
迁移过程中客户端可能收到 `ASK`，迁移完成后收到 `MOVED`。  
迁移要控制速度，避免影响线上延迟。

### 18.2 详细解释版

流程是添加节点、分配迁移计划、逐个 slot 迁移 key、更新 slot 归属、观察负载均衡。迁移大 key 会拖慢过程并造成延迟尖刺，所以扩容前要先治理大 key。  
操作时要关注节点内存、网络、客户端重定向、slot 分布和业务低峰窗口。

### 18.3 图示

```text
旧节点 A/B/C -> 新节点 D
迁移部分 slot -> 数据重新均衡
```

## 19. Cluster 节点故障会怎样？（高难度）

### 19.1 最简练版

**主节点故障后，集群会从它的从节点中选一个提升为新主。**  
如果某个 slot 没有可用主节点，集群可能进入不可服务状态。  
故障恢复取决于副本配置、投票、超时和网络状态。

### 19.2 详细解释版

Cluster 节点通过 gossip 交换状态。某节点被多个主节点认为故障后，会触发 failover。对应从节点申请投票，获得多数主节点同意后提升。  
如果主节点没有从库，或者多数派不可达，负责的 slot 无法正常服务。生产要保证关键主节点至少一个副本，并跨机器、跨机架部署。

### 19.3 例子

```text
Master A 故障 -> Replica A1 发起选举 -> 成为新 Master -> 接管 slot
```

## 20. Redis 跨机房部署要注意什么？（高难度）

### 20.1 最简练版

**跨机房部署要重点关注延迟、一致性、故障切换边界和数据回流风险。**  
Redis 对低延迟网络更友好，跨机房复制延迟会明显增加。  
如果要多活，不能只靠原生主从复制，需要业务层冲突处理或专门方案。

### 20.2 详细解释版

单主跨机房从库适合容灾和异地读，但写后读一致性会变差。跨机房 Sentinel 或 Cluster 要谨慎，网络分区可能增加误判和脑裂风险。  
更稳的做法是同城多副本、异地备份或异地只读容灾；真正多活要设计数据分片归属、幂等、冲突解决、回放和降级策略。

### 20.3 例子

```text
同城：低延迟主从 + 自动切换
异地：备份/只读容灾 + 手动或半自动切换
```
