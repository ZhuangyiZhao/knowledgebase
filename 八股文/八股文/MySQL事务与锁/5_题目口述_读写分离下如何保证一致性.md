# 题目口述：读写分离下如何保证一致性

**适用场景：** MySQL 主从复制、读写分离、主从延迟、读己之写、订单状态一致性、架构设计面试题。

**回答模板：** 先承认读写分离天然有主从延迟；再按业务一致性要求分级；然后给出主库读、会话粘滞、等待位点、版本校验、缓存兜底、监控降级等组合方案；最后讲取舍。

## 1. 问题本质

### 1.1 最简练版

**读写分离的一致性问题，本质是主库写入成功后，从库还没来得及回放这次变更。**  
如果写后立刻读从库，就可能读到旧数据。  
主从复制通常是异步或半同步，不能默认认为从库数据实时等于主库。  
解决思路不是单一方案，而是按业务场景选择强一致、读己之写或最终一致。

### 1.2 主从复制简化链路

```text
应用写主库
  -> 主库提交事务
  -> 写 binlog
  -> 从库 IO 线程拉取 binlog
  -> 写 relay log
  -> 从库 SQL 线程回放
  -> 从库数据可见
```

延迟可能出现在：

1. 主库 binlog 传输到从库慢。
2. 从库 relay log 回放慢。
3. 从库执行大事务、DDL 或热点更新。
4. 从库负载高，CPU、IO、锁等待导致回放堆积。

### 1.3 典型错误

下单接口：

```text
写主库：订单创建成功
立刻查从库：订单不存在
用户看到：下单失败或订单消失
```

这类问题就是读写分离下最典型的“写后读不一致”。

## 2. 先做一致性分级

### 2.1 为什么要分级？

不是所有读都需要强一致。  
如果所有请求都读主库，读写分离的收益会大幅下降。  
所以架构上要先按业务重要性分级。

### 2.2 分级表

| 一致性要求 | 场景 | 推荐策略 |
|---|---|---|
| 强一致 | 支付结果、扣库存、权限校验 | 读主库或事务内完成 |
| 读己之写 | 用户刚提交后立刻查看 | 短时间会话读主、按位点等待 |
| 单调读 | 用户不能看到状态倒退 | 版本号、时间戳、会话粘滞 |
| 最终一致 | 列表、统计、非关键展示 | 读从库，允许延迟 |

### 2.3 面试加分点

可以这样说：

```text
我不会直接说所有读都切主库，因为那会打穿主库。
我会先把读请求按一致性要求分层：关键链路读主，写后短窗口读主，普通展示读从。
```

## 3. 常见解决方案

### 3.1 关键读直接走主库

**最简单可靠的方案是关键链路读主库。**

适用场景：

- 支付状态确认。
- 创建订单后的详情页。
- 库存扣减前校验。
- 权限、风控、余额等不能读旧的场景。

优点：

- 实现简单。
- 一致性强。
- 不依赖从库追赶速度。

缺点：

- 主库压力增加。
- 需要明确哪些读必须走主库。

### 3.2 写后短时间会话粘滞到主库

**用户发生写操作后，在一个短时间窗口内，该用户的读请求走主库。**

例子：

```text
用户提交订单
记录 user_id -> read_primary_until = now + 3s
3 秒内该用户订单详情和订单列表读主库
3 秒后恢复读从库
```

适用：

- 用户刚发布内容后查看。
- 刚修改资料后刷新页面。
- 刚创建订单后查订单详情。

优点：

- 能解决大部分读己之写。
- 不需要所有读都走主。

缺点：

- 窗口过短可能仍不一致。
- 窗口过长会增加主库压力。
- 多端、多服务要共享这个路由状态。

### 3.3 按 binlog 位点或 GTID 等待从库追上

**写主库后拿到提交位点，读从库前等待从库回放到该位点。**

MySQL 可以用 GTID 或 binlog position 类似思路判断从库是否追上。  
如果在超时时间内追上，就读从库；追不上就读主库或提示稍后。

简化流程：

```text
写主库成功
拿到本次写入对应的 GTID/位点
选择一个从库
等待从库执行到该位点，最多等 100ms
  -> 追上：读从库
  -> 未追上：读主库或降级
```

优点：

- 比固定时间粘滞更精确。
- 可以减少不必要的主库读。

缺点：

- 实现复杂。
- 等待会增加接口延迟。
- 从库延迟大时仍要降级。

### 3.4 根据延迟动态路由

**路由层实时监控从库延迟，延迟过大时摘掉从库或把关键读切主。**

常见指标：

- 主从延迟秒数。
- relay log 堆积。
- 回放位点差。
- 心跳表延迟。
- 从库 CPU、IO、锁等待。

注意：`Seconds_Behind_Master` 有时不够准确，生产上更推荐用心跳表或位点差辅助判断。

### 3.5 版本号或更新时间校验

**读请求携带期望版本，如果从库版本低于期望值，就切主或重试。**

例子：

```text
订单更新后版本 version = 8
前端或服务端上下文记录 expected_version = 8
读从库发现 version = 7
  -> 说明从库没追上
  -> 改读主库
```

适合状态机类业务，比如订单、工单、审批流。

### 3.6 缓存辅助

写成功后，可以把最新状态写入缓存或让缓存短时间承接读己之写。

但要注意：

1. 缓存不是天然强一致。
2. 需要处理缓存和数据库双写一致性。
3. 关键链路不能盲目依赖缓存里的旧值。

缓存更适合作为体验兜底，而不是替代数据库一致性设计。

### 3.7 半同步复制的边界

半同步复制可以提高数据不丢的概率，但不能简单等价为“读从库强一致”。  
因为半同步通常强调至少有从库收到日志确认，但不代表从库已经完成回放并可读。  
所以它不能单独解决写后立刻读从库的问题。

## 4. 推荐口述结构

### 4.1 四步回答

1. 定义问题：读写分离有主从延迟，写后读从可能读旧。
2. 业务分级：强一致、读己之写、最终一致分开处理。
3. 方案组合：关键读主，写后粘主，位点等待，延迟监控，版本校验。
4. 讲权衡：一致性、延迟、主库压力、实现复杂度之间取舍。

### 4.2 可以直接背的结构

```text
我会先判断这次读属于哪类一致性。
如果是支付、库存、权限这种强一致读，直接读主库。
如果是用户写完马上查看，做会话级短时间读主，或者用 GTID/位点等待从库追上。
如果是列表、推荐、统计这类允许延迟的读，就读从库。
同时路由层监控从库延迟，延迟超过阈值就摘从或切主。
对于订单这类状态机数据，再加 version/update_time 校验，避免状态倒退。
```

## 5. 典型业务场景

### 5.1 下单后查订单详情

推荐：

1. 创建订单写主库。
2. 返回订单 ID 和状态。
3. 订单详情页短时间读主库。
4. 订单列表可以短时间读主，也可以按订单 ID 读主、列表读从。
5. 从库延迟恢复后再回到读从。

不推荐：

```text
刚写完主库，立刻无脑读从库。
```

### 5.2 修改用户资料后刷新页面

推荐：

1. 写主库成功后更新用户会话标记。
2. 该用户未来几秒资料读取走主库。
3. 普通访客查看非关键展示可以读从库或缓存。

### 5.3 后台报表

报表通常可以接受分钟级延迟。  
可以读从库、离线库或数仓，不应该为了报表强一致打主库。

### 5.4 库存扣减

库存扣减前的校验和扣减必须在主库或同一个强一致存储内完成。  
不能读从库判断有库存，再去主库扣减，否则从库旧数据会放大超卖风险。

## 6. 监控和降级

### 6.1 必备监控

| 指标 | 说明 |
|---|---|
| 主从延迟 | 从库落后主库多久 |
| 位点差 | binlog/GTID 回放差距 |
| 心跳表延迟 | 更贴近业务可见延迟 |
| 从库 QPS 和 RT | 从库是否被读流量打满 |
| 复制线程状态 | IO/SQL 线程是否异常 |
| 锁等待 | 从库回放是否被锁阻塞 |

### 6.2 降级策略

当从库延迟超过阈值：

1. 摘掉延迟严重的从库。
2. 关键读切主库。
3. 非关键读返回缓存或提示稍后刷新。
4. 限制大查询，避免从库雪上加霜。
5. 告警并定位复制延迟原因。

### 6.3 防止切主打爆主库

切主不是万能药。  
如果大量读流量同时切主，可能把主库打挂。  
所以要配合：

- 只切关键读。
- 按用户或业务灰度切。
- 限流。
- 缓存兜底。
- 从库恢复后自动切回。

## 7. 面试口述版

### 7.1 30 秒回答

读写分离下的一致性问题，本质是主库写成功后从库还没回放到对应位点，写后立刻读从可能读旧数据。我会先按业务分级：支付、库存、权限这类强一致读直接走主库；用户写完马上看的读，做会话级短时间读主，或者等待 GTID/binlog 位点追上；普通列表、统计允许最终一致就读从库。同时监控从库延迟，延迟过大时摘从或关键读切主，并用版本号、更新时间避免状态倒退。

### 7.2 90 秒回答

读写分离不能默认保证强一致，因为 MySQL 主从复制存在传输和回放延迟。我的设计会先做一致性分级。第一类是强一致读，比如支付结果、库存扣减、权限校验，这些直接读主库，避免旧数据影响业务决策。第二类是读己之写，比如用户刚创建订单后查看详情，可以在写成功后给用户打一个短时间读主标记，或者记录写入位点，读从前等待从库追到这个 GTID 或 binlog position，超时就切主。第三类是允许最终一致的读，比如列表、推荐、报表，继续读从库。

此外，路由层要监控从库延迟，不能只看一个粗略指标，最好结合心跳表和位点差。延迟超过阈值时摘掉异常从库，关键读切主，非关键读可以缓存兜底或提示稍后刷新。对订单这类状态机数据，可以加版本号或更新时间校验，发现从库版本低于期望就改读主库。整体是在一致性、延迟、主库压力和实现复杂度之间做权衡。

## 8. 高频追问

### 8.1 全部读都走主库行不行？

技术上可以，但会削弱读写分离的意义，也可能把主库打成瓶颈。  
更合理的是关键读走主，普通读走从，写后短窗口读主。

### 8.2 主从延迟怎么发现？

可以看复制状态、位点差、relay log 堆积和心跳表延迟。  
生产上不要只依赖 `Seconds_Behind_Master`，它在一些场景下不够准确。

### 8.3 半同步复制能不能保证读从一致？

不能简单保证。  
半同步更关注事务提交时至少有从库收到日志确认，但从库收到不代表已经回放完成，也不代表应用立刻能从从库读到最新数据。

### 8.4 等待位点会不会拖慢接口？

会，所以要设置很短的超时。  
比如等待 50ms 或 100ms，追上就读从，追不上就切主或降级。  
不能无限等待。

### 8.5 读己之写和强一致有什么区别？

读己之写只要求用户能读到自己刚写的结果。  
强一致通常要求所有后续业务决策都基于最新正确数据。  
两者可以用不同成本的方案处理。

## 9. 容易踩坑

1. 不要说主从复制后读从库一定实时一致。
2. 不要把半同步复制说成从库已应用。
3. 不要所有读一刀切主库，要讲业务分级。
4. 不要只用固定 sleep 等从库追赶，应该有位点、版本或路由策略。
5. 不要忽略监控和降级，主从延迟是运行时问题，不是只靠代码能永久解决。
