## 1. MVCC 解决了什么问题？

### 1.1 最简练版

**MVCC 用多版本数据让普通读不用阻塞写，写也不用阻塞普通读。**  
它通过保存数据的历史版本，让事务按自己的 ReadView 读取可见版本。  
这样同一行数据被修改后，旧事务仍然可以读到旧版本，新事务可以读到新版本。  
它是 InnoDB 实现 RC 和 RR 隔离级别的重要机制。

### 1.2 详细解释版

如果没有 MVCC，数据库要保证读一致性，可能需要让读写互相加锁：

```text
事务 A 正在读一行
事务 B 想更新这一行
  -> 等

事务 B 正在更新一行
事务 A 想读这一行
  -> 等
```

这会让并发能力很差。

MVCC 的思路是：**不要只保留最新值，而是保留多个历史版本。**  
事务读数据时，不一定读最新版本，而是根据自己的快照规则选择一个可见版本。

```text
当前最新版本：余额 = 80，trx_id = 200
历史旧版本：余额 = 100，trx_id = 100

老事务的 ReadView 看不到 trx_id = 200
  -> 沿着版本链找到 trx_id = 100 的版本
  -> 读到余额 = 100
```

## 2. MVCC 依赖哪些结构？

### 2.1 核心结构

InnoDB 的 MVCC 主要依赖三类东西：

| 结构 | 作用 |
|---|---|
| 隐藏字段 | 标记这行数据由哪个事务修改，旧版本在哪里 |
| undo log | 保存数据被修改前的旧版本 |
| ReadView | 判断某个版本对当前事务是否可见 |

### 2.2 行记录隐藏字段

InnoDB 聚簇索引记录里有几个重要隐藏字段：

| 隐藏字段 | 含义 |
|---|---|
| `DB_TRX_ID` | 最近一次修改这行记录的事务 ID |
| `DB_ROLL_PTR` | 指向 undo log 中旧版本的回滚指针 |
| `DB_ROW_ID` | 没有显式主键时生成的隐藏行 ID |

面试重点通常是前两个：

```text
DB_TRX_ID   -> 这个版本是谁改出来的
DB_ROLL_PTR -> 上一个旧版本在哪里
```

### 2.3 为什么二级索引也能配合 MVCC？

MVCC 的版本链主要挂在聚簇索引记录上。二级索引查询时，如果涉及可见性判断，最终也需要通过主键回到聚簇索引记录判断版本是否可见。  
所以回答时可以重点讲聚簇索引记录的隐藏字段，不需要展开二级索引的内部细节。

## 3. undo log 和版本链怎么工作？

### 3.1 最简练版

**undo log 一方面支持事务回滚，另一方面为 MVCC 提供历史版本。**  
每次更新记录时，InnoDB 会把旧值写入 undo log，并通过回滚指针把多个旧版本串起来。  
普通查询如果看不到最新版本，就沿着版本链往前找，直到找到对当前 ReadView 可见的版本。  
长事务会导致旧版本长时间不能清理，从而造成 undo 膨胀。

### 3.2 版本链示意

假设一行订单金额被多次修改：

```text
聚簇索引当前记录
amount = 80
DB_TRX_ID = 300
DB_ROLL_PTR
   |
   v
undo 版本 1：amount = 100, DB_TRX_ID = 200
   |
   v
undo 版本 2：amount = 120, DB_TRX_ID = 100
```

如果某个事务的 ReadView 只能看到 `trx_id < 200` 的版本，那么：

```text
先看当前记录 trx_id = 300 -> 不可见
再看 undo 版本 trx_id = 200 -> 不可见
再看 undo 版本 trx_id = 100 -> 可见
最终读到 amount = 120
```

### 3.3 update 时发生了什么？

以更新订单状态为例：

```sql
update order_tab
set status = 'paid'
where order_id = 1001;
```

大致流程：

1. 找到需要更新的聚簇索引记录。
2. 对记录加锁，避免并发写冲突。
3. 把旧版本写入 undo log。
4. 修改当前记录的字段值。
5. 更新当前记录的 `DB_TRX_ID` 和 `DB_ROLL_PTR`。
6. 提交后，其他事务是否能看到新版本取决于它们的 ReadView。

### 3.4 undo log 什么时候清理？

undo log 不能在事务提交后立刻全部删除，因为可能还有老事务需要读旧版本。

```text
事务 A 很早开始，还没结束
事务 B 修改了很多数据并提交
事务 A 的快照可能还要读旧版本
  -> undo 不能清理
```

这就是长事务的风险之一：**它会拖住历史版本清理，导致 undo 空间增长，也可能让查询沿着很长的版本链回溯。**

## 4. ReadView 是什么？

### 4.1 最简练版

**ReadView 是事务做一致性读时生成的可见性快照。**  
它记录生成快照时哪些事务还活跃，以及事务 ID 的边界。  
查询某个版本时，就拿版本上的 `DB_TRX_ID` 和 ReadView 里的边界做判断。  
RC 下通常每条快照读语句生成新的 ReadView，RR 下通常同一事务复用第一次生成的 ReadView。

### 4.2 ReadView 里的关键字段

常见解释里会提到四个字段：

| 字段 | 含义 |
|---|---|
| `creator_trx_id` | 创建这个 ReadView 的事务 ID |
| `m_ids` | 创建 ReadView 时仍活跃的事务 ID 列表 |
| `min_trx_id` | `m_ids` 中最小的事务 ID |
| `max_trx_id` | 创建 ReadView 时系统将要分配的下一个事务 ID |

可以用一句话理解：

```text
ReadView 记录的是：我创建快照这一刻，哪些事务还没提交。
```

## 5. 可见性规则怎么判断？

### 5.1 最简练版

**判断一个版本是否可见，就是看这个版本的事务 ID 和 ReadView 的关系。**  
比 `min_trx_id` 小，说明早就提交了，通常可见。  
大于等于 `max_trx_id`，说明是快照之后才出现的，通常不可见。  
落在中间就看它是否还在活跃事务列表里，在列表里不可见，不在列表里可见。

### 5.2 规则表

设某个记录版本的事务 ID 是 `trx_id`。

| 判断条件 | 结果 | 含义 |
|---|---|---|
| `trx_id == creator_trx_id` | 可见 | 自己改的自己能看见 |
| `trx_id < min_trx_id` | 可见 | 生成快照前已经提交 |
| `trx_id >= max_trx_id` | 不可见 | 生成快照后才出现 |
| `min_trx_id <= trx_id < max_trx_id` 且在 `m_ids` 中 | 不可见 | 创建快照时该事务还没提交 |
| `min_trx_id <= trx_id < max_trx_id` 且不在 `m_ids` 中 | 可见 | 创建快照前已经提交 |

### 5.3 例子

ReadView 创建时：

```text
m_ids = [100, 101, 105]
min_trx_id = 100
max_trx_id = 110
creator_trx_id = 101
```

不同版本判断：

| 版本事务 ID | 是否可见 | 原因 |
|---|---|---|
| 90 | 可见 | 小于 `min_trx_id` |
| 100 | 不可见 | 在活跃事务列表中 |
| 101 | 可见 | 自己的修改 |
| 103 | 可见 | 不在活跃列表中，说明已提交 |
| 105 | 不可见 | 在活跃事务列表中 |
| 120 | 不可见 | 大于等于 `max_trx_id` |

## 6. RC 和 RR 的区别

### 6.1 最简练版

**RC 和 RR 的核心差异是 ReadView 创建时机不同。**  
RC 下，每条普通 `select` 都会生成新的 ReadView，所以能看到其他事务刚提交的数据。  
RR 下，同一事务内第一次普通 `select` 生成 ReadView 后通常复用，所以多次读取结果稳定。  
这就是 RC 可能不可重复读，而 RR 能做到可重复读的关键。

### 6.2 时间线例子

```text
初始值：balance = 100

事务 A begin
事务 A select balance -> 100

事务 B update balance = 80
事务 B commit

事务 A select balance
```

如果事务 A 是 RC：

```text
第二次 select 重新生成 ReadView
看到事务 B 已提交
结果 = 80
```

如果事务 A 是 RR：

```text
第二次 select 复用第一次 ReadView
事务 B 对事务 A 的快照不可见
结果 = 100
```

### 6.3 当前读不走旧快照

注意，`update`、`delete`、`select ... for update` 这类当前读读的是最新已提交版本并加锁，不是普通快照读。

```sql
select * from account where id = 1;
select * from account where id = 1 for update;
```

第一条是快照读，第二条是当前读。  
这也是很多面试题的坑：**RR 里的普通 select 可重复，不代表所有读都读同一个旧快照。**

## 7. 面试口述版

### 7.1 30 秒回答

MVCC 是 InnoDB 用来提升并发读写能力的机制，本质是给数据保留多个历史版本，让普通读按快照读取，不和写强互斥。它依赖行记录里的隐藏字段、undo log 版本链和 ReadView。更新数据时，旧版本会写入 undo log，并通过回滚指针串成版本链。查询时根据 ReadView 判断当前版本是否可见，如果不可见就沿着版本链找上一个版本。RC 和 RR 的主要区别是 ReadView 创建时机，RC 每条语句创建，RR 同一事务内通常复用。

### 7.2 90 秒回答

MVCC 解决的是并发读写互相阻塞的问题。如果写一行时普通读必须等，或者普通读一行时写必须等，数据库并发会很差。InnoDB 的做法是每行记录有隐藏字段，比如最近修改它的事务 ID 和指向 undo log 的回滚指针。每次更新时，旧值会进入 undo log，多个旧版本通过指针形成版本链。

普通 select 会生成 ReadView，ReadView 记录当前还活跃的事务 ID，以及事务 ID 的上下界。判断某个版本是否可见时，就拿记录版本上的 `DB_TRX_ID` 和 ReadView 比较。如果版本是快照前已提交的，就可见；如果是快照后出现的，或者创建快照时还没提交，就不可见，需要沿版本链继续找。RC 和 RR 最大差异是 RC 每条语句一个 ReadView，RR 同一事务内复用 ReadView，所以 RR 可以做到普通快照读可重复。

## 8. 高频追问

### 8.1 MVCC 能不能解决写写冲突？

不能完全解决。MVCC 主要优化普通读和写之间的并发。  
两个事务同时更新同一行时，仍然需要锁来控制，后来的事务要等待前一个事务释放行锁。

### 8.2 undo log 只用于回滚吗？

不是。undo log 至少有两个重要作用：

1. 支持事务回滚，保证原子性。
2. 保存历史版本，支持 MVCC 快照读。

### 8.3 为什么长事务危险？

长事务会长时间持有 ReadView，导致很多旧版本不能被 purge 清理。  
这会带来 undo 空间膨胀、版本链变长、查询变慢、锁持有时间过长等问题。

### 8.4 MVCC 下读到的一定是最新数据吗？

普通快照读不一定读最新数据，它读的是对当前 ReadView 可见的数据。  
如果必须读最新并加锁，要使用当前读，比如 `select ... for update`。

---

## 9. 容易踩坑

1. 不要说 MVCC 不加锁就能解决所有并发问题，写写冲突仍然靠锁。
2. 不要把 undo log 和 redo log 混淆：undo 主要用于回滚和历史版本，redo 主要用于崩溃恢复。
3. 不要忽略 ReadView 创建时机，RC 和 RR 的核心差异就在这里。
4. 不要认为事务提交后 undo 立刻删除，老事务还可能需要历史版本。
5. 不要把当前读也当作快照读，`for update`、`update`、`delete` 都是当前读。
