MySQL 慢 SQL 优化指南

没有走索引

SQL 没有走索引常见原因包括：查询条件对索引列使用函数或隐式类型转换、违反最左前缀原则、范围查询后面的列无法继续利用索引、统计信息不准确、返回数据比例过高导致优化器认为全表扫描更划算。排查时先看 EXPLAIN 的 type、key、rows、Extra，再结合表结构、索引基数和实际过滤率判断。

如果 type 是 ALL，通常表示全表扫描；如果 key 为空，表示优化器没有选择索引；如果 rows 很大，说明预估扫描行数高。优化方向包括改写 SQL、补充联合索引、避免隐式转换、减少回表、用覆盖索引降低随机 IO。

深分页优化

limit 100000, 20 这类深分页会扫描并丢弃大量数据，页数越深延迟越高。常见优化方式是基于上一页最大 ID 做 seek pagination，例如 where id > last_id order by id limit 20。如果必须随机跳页，可以先用覆盖索引查出主键，再回表查询完整数据。

联合索引设计

联合索引设计要把高频查询条件、等值条件、排序字段和覆盖字段综合考虑。一般把区分度高且常用于等值过滤的字段放前面，范围字段放后面。索引不是越多越好，过多索引会增加写入成本和优化器选择成本。
