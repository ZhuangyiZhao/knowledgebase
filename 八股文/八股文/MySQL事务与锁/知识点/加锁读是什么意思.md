# 加锁读是什么意思

**适用场景：** `select ... for update`、`select ... for share`、当前读、行锁、幻读、库存扣减、并发更新场景题。

**回答模板：** 先说加锁读是读取数据时主动给读到的记录或范围加锁；再说明它属于当前读，读取最新已提交版本；然后区分共享锁和排他锁；最后结合索引、隔离级别解释锁范围。

## 1. 一句话解释

**加锁读就是在读取数据的同时，对读到的记录或扫描到的索引范围加锁。**

普通 `select` 通常是快照读，只是读一个 MVCC 一致性视图，不主动加行锁。  
加锁读则是当前读，它要读最新版本，并通过锁保证后续操作期间数据不被其他事务随意修改或插入。

一句话背诵：

```text
加锁读 = 当前读 + 主动加锁，用来保证读完之后再更新或校验时，数据不会被并发事务破坏。
```

---

## 2. 为什么需要加锁读？

因为很多业务不是“只看一眼数据”就结束，而是：

```text
先查数据
再根据查询结果做判断
最后更新数据
```

如果查询阶段不加锁，多个事务可能同时读到同一份旧数据，然后都认为自己可以更新，导致并发问题。

典型场景：

- 秒杀扣库存。
- 账户余额扣减。
- 订单状态流转。
- 抢任务、抢优惠券、抢号段。

例如库存还剩 `1`，两个事务同时普通查询：

```text
事务 A select stock -> 1
事务 B select stock -> 1
事务 A update stock = stock - 1
事务 B update stock = stock - 1
```

如果没有合适的并发控制，就可能出现超卖、重复处理等问题。

## 3. 常见 SQL 写法

### 3.1 排他锁读

```sql
select *
from product
where id = 1001
for update;
```

`for update` 通常加排他锁，也就是 X Lock。  
它表示当前事务后续可能要修改这些数据，所以其他事务不能再对这些记录加排他锁或修改它们。

常见用法：

```text
先 select ... for update 锁住目标行
再根据业务条件 update
最后 commit 释放锁
```

### 3.2 共享锁读

MySQL 8.0 常用：

```sql
select *
from product
where id = 1001
for share;
```

旧写法：

```sql
select *
from product
where id = 1001
lock in share mode;
```

共享锁也叫 S Lock。  
它允许其他事务继续加共享锁读，但通常不允许其他事务修改这些记录。

### 3.3 隐式加锁的当前读

这些语句也属于当前读相关操作：

```sql
update product
set stock = stock - 1
where id = 1001;
```

```sql
delete from product
where id = 1001;
```

```sql
insert into product(id, stock) values(1001, 10);
```

`update` 和 `delete` 必须基于最新数据执行，所以不能只读历史快照。

## 4. 加锁读和快照读有什么区别？

| 对比项 | 快照读 | 加锁读 |
|---|---|---|
| 常见语句 | 普通 `select` | `select ... for update`、`select ... for share` |
| 读取版本 | MVCC 可见历史版本 | 最新已提交版本 |
| 是否加锁 | 通常不加行锁 | 会加行锁或范围锁 |
| 并发能力 | 读写不容易互相阻塞 | 可能阻塞其他事务 |
| 适用场景 | 查询展示、统计、普通读取 | 读后要改、强一致校验 |

核心区别：

```text
快照读追求读写并发，加锁读追求并发安全。
```

### 4.1 `begin; select * from xxx; commit;` 算什么读？

这种普通 `select` 算快照读。

```sql
begin;

select *
from xxx;

commit;
```

只要是普通 `select`，没有：

```sql
for update
for share
lock in share mode
```

它在 InnoDB 里通常就是快照读，也叫一致性非锁定读。  
它通过 MVCC 读取 ReadView 可见的数据版本，不主动加行锁。

所以是不是写在事务里，不决定它是快照读还是当前读。  
关键看 SQL 类型：

```text
普通 select -> 快照读
select ... for update -> 当前读
select ... for share -> 当前读
update/delete/insert -> 当前读相关操作
```

补一句：

```text
RC 下，普通 select 每次执行都会创建新的 ReadView；
RR 下，普通 select 如果是事务里的第一次快照读，就会创建 ReadView，后续普通 select 复用它。
```

## 5. 加锁读到底加什么锁？

加锁读的锁范围和 SQL 条件、索引、隔离级别、执行计划强相关。

### 5.1 主键等值命中

```sql
select *
from product
where id = 1001
for update;
```

如果 `id` 是主键且记录存在，通常锁住主键索引上 `id = 1001` 这一条记录，也就是 record lock。

### 5.2 唯一索引等值命中

```sql
select *
from user_tab
where email = 'a@test.com'
for update;
```

如果 `email` 是唯一索引且命中记录，通常也是锁住对应索引记录以及相关主键记录。

### 5.3 范围查询

```sql
select *
from product
where price between 100 and 200
for update;
```

范围加锁读可能加 next-key lock，也就是：

```text
record lock + gap lock
```

它不仅锁住已经存在的记录，还可能锁住索引间隙，防止其他事务插入新的满足条件的记录。

这里可以拆开理解：

| 锁类型 | 锁住什么 | 作用 |
|---|---|---|
| record lock | 已经存在的索引记录 | 防止这条记录被其他事务修改或删除 |
| gap lock | 两条索引记录之间的间隙 | 防止其他事务往这个间隙插入新记录 |
| next-key lock | 记录本身 + 记录前面的间隙 | 同时防止已有记录被改，也防止范围内插入新记录 |

假设 `price` 索引里已有这些值：

```text
10, 20, 30
```

那么索引间隙可以粗略理解为：

```text
(-∞, 10)
(10, 20)
(20, 30)
(30, +∞)
```

如果事务 A 做范围加锁读：

```sql
select *
from product
where price between 10 and 20
for update;
```

它可能不仅锁住 `price = 10`、`price = 20` 这些已有记录，还会锁住相关间隙。  
这样其他事务就不能随便插入 `price = 15` 这种落在范围里的新记录。

一句话背诵：

```text
record lock 管已有记录，gap lock 管记录之间的空隙，next-key lock 两个都管。
```

### 5.4 没有合适索引

```sql
select *
from product
where category = 'phone'
for update;
```

如果 `category` 没有合适索引，InnoDB 可能扫描大量记录并加锁。  
线上看起来就可能像锁了很多行，甚至接近锁表效果。

所以加锁读一定要关注：

- SQL 是否走索引。
- 走的是哪个索引。
- 扫描范围有多大。
- 是否出现全表扫描。

## 6. RC 和 RR 下有什么差异？

### 6.1 RC 隔离级别

在 RC 下，加锁读读取最新已提交版本。  
InnoDB 通常更偏向记录锁，gap lock 使用相对少一些，所以并发插入受到的限制较少。

### 6.2 RR 隔离级别

在 RR 下，加锁读仍然是当前读，不是快照读。  
为了避免范围当前读出现幻读，InnoDB 可能使用 next-key lock 锁住记录和间隙。

重点：

```text
RR 下普通 select 是快照读；
RR 下 select ... for update 是当前读；
当前读会读最新数据，并且可能加 next-key lock。
```

### 6.3 ReadView 创建时机怎么理解？

这句话：

```text
read committed：事务每次 select 时创建 ReadView
repeatable read：事务第一次 select 时创建 ReadView，后续一直使用
```

说的是普通 `select` 做快照读时，ReadView 什么时候生成。

ReadView 可以理解成一次查询时拍下的“可见性快照”。  
它决定当前这次 `select` 能看到哪些事务提交的数据，不能看到哪些还没提交或后来才提交的数据。

在 RC 下：

```text
每执行一次普通 select，就重新生成一个 ReadView。
```

所以同一个事务里，两次普通 `select` 可能看到不同结果：

```text
事务 A begin
事务 A select name where id = 1 -> 张三

事务 B update user set name = '李四' where id = 1
事务 B commit

事务 A select name where id = 1 -> 李四
```

因为事务 A 第二次 `select` 会重新创建 ReadView，所以能看到事务 B 已经提交的新数据。  
这就是 RC 下可能出现不可重复读的原因。

在 RR 下：

```text
事务内第一次普通 select 时创建 ReadView，后续普通 select 复用这个 ReadView。
```

所以同一个事务里，多次普通 `select` 看到的结果通常一致：

```text
事务 A begin
事务 A select name where id = 1 -> 张三

事务 B update user set name = '李四' where id = 1
事务 B commit

事务 A select name where id = 1 -> 张三
```

虽然事务 B 已经提交了，但事务 A 后续普通 `select` 仍然用第一次的 ReadView，所以看不到 `李四`，还是看到 `张三`。

一句话背诵：

```text
RC 是每次查询都拍新照片，所以能看到别人新提交的数据；
RR 是事务第一次查询拍一张照片，后面一直看这张照片，所以结果可重复。
```

注意，这里说的是普通 `select` 快照读。  
`select ... for update`、`select ... for share`、`update`、`delete` 属于当前读，不按“复用 ReadView”的逻辑走。

## 7. 典型例子

### 7.1 扣库存

```sql
begin;

select stock
from product
where id = 1001
for update;

-- 应用判断 stock > 0

update product
set stock = stock - 1
where id = 1001;

commit;
```

事务 A 执行 `select ... for update` 后，锁住商品行。  
事务 B 如果也想锁同一行，就要等待事务 A 提交或回滚。

这样可以避免两个事务同时基于同一个库存值做扣减判断。

### 7.2 抢任务

```sql
begin;

select id
from task
where status = 'pending'
order by id
limit 1
for update;

update task
set status = 'processing'
where id = ?;

commit;
```

这个模式用于让多个消费者竞争任务。  
谁先锁到任务，谁处理；其他事务要等待或跳过。

## 8. 面试口述版

加锁读指的是读取数据时主动加锁，比如 `select ... for update` 和 `select ... for share`。  
它和普通 `select` 不一样，普通 `select` 一般是快照读，通过 MVCC 读一致性视图，不会主动加行锁；加锁读属于当前读，会读取最新已提交版本，并对读到的记录或范围加锁。

`for update` 通常加排他锁，适合读完后要修改的场景，比如扣库存、扣余额；`for share` 或 `lock in share mode` 加共享锁，适合希望别人能读但不能改的场景。

加锁读具体锁住多少数据，取决于索引和隔离级别。主键等值命中一般锁单行；范围查询在 RR 下可能加 next-key lock，既锁记录也锁间隙，用来防止幻读。如果没有合适索引，锁范围可能非常大。

## 9. 高频追问

### 9.1 加锁读一定会阻塞普通 select 吗？

通常不会阻塞普通快照读。  
普通 `select` 可以通过 MVCC 读取历史可见版本。

但它可能阻塞其他加锁读、`update`、`delete` 等当前读操作。

### 9.2 加上 X Lock 的时候，其他事务可读吗？

可以读，但要看“怎么读”。

如果事务 A 执行：

```sql
begin;

select *
from product
where id = 1001
for update;
```

这时 `id = 1001` 这行被加了 X Lock。  
其他事务 B 执行普通查询：

```sql
select *
from product
where id = 1001;
```

通常仍然可以读，因为普通 `select` 是快照读，可以通过 MVCC 读取历史可见版本，不需要拿这行的锁。

但如果事务 B 执行下面这些当前读或写操作：

```sql
select *
from product
where id = 1001
for update;
```

```sql
select *
from product
where id = 1001
for share;
```

```sql
update product
set stock = stock - 1
where id = 1001;
```

就会被阻塞，直到事务 A 提交或回滚。  
因为 X Lock 和 S Lock、X Lock 都不兼容。

一句话背诵：

```text
X Lock 锁住的是当前读和写，不挡普通快照读。
```

### 9.3 加上 S Lock 的时候，其他事务可读吗？

可以读，也要看“怎么读”。

如果事务 A 执行：

```sql
begin;

select *
from product
where id = 1001
for share;
```

这时 `id = 1001` 这行被加了 S Lock。  
其他事务 B 执行普通查询：

```sql
select *
from product
where id = 1001;
```

可以读，因为普通 `select` 是快照读，走 MVCC，不需要和 S Lock 竞争。

如果事务 B 也执行共享锁读：

```sql
select *
from product
where id = 1001
for share;
```

也可以读，因为 S Lock 和 S Lock 兼容。

但如果事务 B 想修改数据：

```sql
update product
set stock = stock - 1
where id = 1001;
```

或者想加排他锁：

```sql
select *
from product
where id = 1001
for update;
```

就会被阻塞，直到事务 A 提交或回滚。  
因为 S Lock 和 X Lock 不兼容。

一句话背诵：

```text
S Lock 不挡普通读，也不挡共享锁读；但挡写和排他锁读。
```

### 9.4 `select ... for update` 和 `update` 有什么关系？

它们都属于当前读相关操作。  
区别是 `select ... for update` 是先显式锁定，再由应用决定后续怎么处理；`update` 是直接修改并加锁。

### 9.5 加锁读读到的是快照吗？

不是。  
加锁读是当前读，读取最新已提交版本，并尝试对目标记录或范围加锁。

### 9.6 为什么范围加锁读可能锁间隙？

因为只锁已有记录还不够。  
如果其他事务可以往范围里插入新记录，那么当前事务再次范围查询时就可能看到新增记录，也就是幻读。  
RR 下 InnoDB 通过 next-key lock 把记录和间隙一起锁住，减少这类问题。

## 10. 容易踩坑

### 10.1 以为 `for update` 只锁返回的行

不一定。  
范围查询、索引扫描、无索引扫描都可能导致更大的锁范围。

### 10.2 忽略索引

加锁读一定要配合合适索引。  
没有索引时，扫描范围变大，锁等待和死锁风险也会变大。

### 10.3 事务太长

锁要等事务提交或回滚才释放。  
如果事务里夹杂远程调用、复杂计算、用户交互，就会放大锁等待风险。

### 10.4 把快照读和当前读混在一起理解

同一个 RR 事务里，普通 `select` 可能看不到其他事务新提交的数据；  
但 `select ... for update` 是当前读，可能看到最新已提交数据。

这也是很多人疑惑的点：

```text
RR 不是可重复读吗，为什么 for update 又能看到新数据？
原因是普通 select 是快照读，for update 是当前读。
```
