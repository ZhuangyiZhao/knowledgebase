## 1. TCP 滑动窗口是什么？

### 1.1 面试题

TCP 为什么需要滑动窗口？滑动窗口怎么提升传输效率？

### 1.2 最简练版

**TCP 滑动窗口是为了在保证可靠性的同时提升传输效率。**  
如果每发一个包都等 ACK，**吞吐会被 RTT 严重限制**；有了窗口后，发送方可以连续发送多个未确认的数据段。  
窗口会随着 ACK 向前移动，**已经确认的数据从窗口中移出，新的数据进入窗口**。  
它本质上是在“可靠传输”和“批量发送”之间做平衡。

### 1.3 详细解释版

如果没有滑动窗口，TCP 只能这样工作：

```text
发送 1 个报文 -> 等 ACK -> 再发送下一个报文
```

这种方式叫**停止等待**，缺点是**链路利用率很低**。尤其在 RTT 比较大、带宽比较高的场景里，网络明明可以传更多数据，但发送方一直在等 ACK。

滑动窗口的思路是：发送方维护一个窗口，窗口内的数据可以先发出去，不必等每一个包都被确认。

发送方数据可以分成四段：

- 已发送且已确认。
- 已发送但未确认。
- 未发送但允许发送。
- 未发送且暂时不允许发送。

当接收方返回 ACK 后，发送方把已确认的数据移出窗口，窗口整体向右滑动，于是后面的数据又可以继续发送。

### 1.4 图示

```text
发送缓冲区：

| 已确认 | 已发送未确认 | 可发送但未发送 | 暂不可发送 |
|--------|--------------|----------------|------------|
         ^                              ^
         |                              |
      window left                   window right

收到 ACK 后：

| 已确认更多数据 | 已发送未确认 | 可发送但未发送 | 暂不可发送 |
|----------------|--------------|----------------|------------|
                 ^                              ^
                 |                              |
              window left                   window right
```

### 1.5 举例理解

假设窗口大小是 5 个报文段，发送方可以连续发送 1、2、3、4、5。  
如果收到了 `ACK=3`，表示 3 之前的数据都收到了，也就是 1、2 已确认。  
窗口就可以向后滑动两个位置，继续发送 6、7。

```text
初始窗口：

1  2  3  4  5  6  7  8
[-----------]

收到 ACK=3 后：

1  2  3  4  5  6  7  8
      [-----------]
```

### 1.6 一句话背诵

**滑动窗口让 TCP 不必每发一个包都等一次 ACK，而是在窗口范围内批量发送、按 ACK 推进，从而提高链路利用率。**

## 2. TCP 流量控制是什么？

### 2.1 面试题

TCP 流量控制解决什么问题？接收窗口 `rwnd` 是什么？

### 2.2 最简练版

**流量控制是为了防止发送方发得太快，把接收方缓冲区打满。**  
TCP 接收方会通过 ACK 报文里的窗口大小告诉发送方：**我现在还能接收多少数据**，这个窗口叫接收窗口 `rwnd`。  
发送方根据 `rwnd` 控制发送速度，**不能超过接收方的处理能力**。  
所以 *流量控制保护的是接收端*，*不是整个网络*。

### 2.3 详细解释版

TCP 是可靠传输，接收方收到数据后需要先放入接收缓冲区，再由应用程序读取。

如果发送方发送太快，而接收方应用读取太慢，就会出现：

```text
网络数据到达速度 > 应用程序读取速度
```

接收缓冲区会逐渐被填满。为了避免丢数据，接收方会在 ACK 中通告自己的**剩余接收窗口**，也就是 `rwnd`。

发送方根据 `rwnd` 控制最多还能发送多少未确认数据：

```text
可发送数据量 <= 接收方通告的 rwnd
```

如果接收方缓冲区满了，会通告窗口为 0，发送方暂停发送。之后发送方会启动零窗口探测，定期发一个很小的探测包，询问接收方窗口是否恢复。

### 2.4 图示

```text
接收方缓冲区：

| 已被应用读取 | 已收到未读取 | 剩余空间 |
|--------------|--------------|----------|
                              ^
                              |
                          rwnd 通告给发送方

发送方：

根据 rwnd 决定还能继续发多少数据
```

窗口变小过程：

```text
Client                              Server
  |                                   |
  | -------- data 1KB --------------> |  接收缓冲区剩余 8KB
  | <-------- ACK, rwnd=8KB --------- |
  |                                   |
  | -------- data 8KB --------------> |  应用读取慢，缓冲区满
  | <-------- ACK, rwnd=0 ----------- |
  |                                   |
  |      暂停发送，等待窗口恢复        |
```

### 2.5 和滑动窗口的关系

**滑动窗口是一种机制，流量控制是它解决的核心问题之一。**  
更严谨地说，*滑动窗口是手段*，*流量控制是目标*。  
接收方通过 `rwnd` 限制发送窗口大小，让发送方不要超过接收方处理能力。

### 2.6 常见追问：零窗口会不会一直卡住？

**可能。** 比如接收方应用一直不读取数据，窗口就一直是 0。  
TCP 会做**零窗口探测**，但应用层仍然可能表现为**请求卡住、连接堆积、延迟升高**。

线上排查时可以关注：

- 接收方应用是否阻塞。
- 读 socket 的协程或线程是否卡住。
- 连接的 Recv-Q / Send-Q 是否异常。
- 服务端是否出现慢请求、GC、锁等待、下游阻塞。

### 2.7 一句话背诵

**流量控制通过接收窗口 `rwnd` 告诉发送方“我还能接多少”，防止发送方压垮接收方。**

## 3. TCP 拥塞控制是什么？

### 3.1 面试题

TCP 拥塞控制解决什么问题？慢启动、拥塞避免、快重传、快恢复分别是什么？

### 3.2 最简练版

拥塞控制是为了防止发送方把网络链路打爆，保护的是整个网络。  
TCP 通过拥塞窗口 `cwnd`（Congestion Window，拥塞窗口）控制发送速率，真正能发送的数据量通常取接收窗口 `rwnd`（Receive Window，接收窗口）和 `cwnd` 的较小值。  
刚开始会用慢启动快速探测网络容量，达到阈值后进入拥塞避免，增长速度从指数变成线性。  
一旦出现超时或重复确认，TCP 会认为网络可能拥塞，然后通过快重传、快恢复或超时重传把发送速率降下来。  
你可以把它理解成“先大胆试探带宽，发现网络吃不消就立刻收敛”。

### 3.3 详细解释版

流量控制只关心接收方能不能接住，但它不知道网络中间链路是否拥堵。

举个例子：

```text
发送方很强，接收方也很强，但中间网络带宽不足
```

如果还一直猛发，路由器队列会堆积、丢包、重传增多，最后整个网络吞吐下降。拥塞控制就是为了解决这个问题。

TCP 发送方维护一个拥塞窗口 `cwnd`（Congestion Window，拥塞窗口）。实际可发送窗口通常是：

```text
send_window = min(rwnd, cwnd)
```

其中：

- `rwnd`：`Receive Window`，接收窗口，由接收方通告，体现接收方处理能力。
- `cwnd`：`Congestion Window`，拥塞窗口，由发送方自己维护，体现网络承载能力。

面试里可以先抓本质：

```text
流量控制看接收方有没有被打满
拥塞控制看网络链路有没有被打满
```

也就是说，哪怕发送方机器很强、接收方机器也很强，只要中间交换机、路由器、运营商链路扛不住，TCP 也必须主动降速。

### 3.4 一个很典型的线上例子

比如服务 A 机房向服务 B 机房同步日志：

- A 端网卡和 CPU 都很空闲。
- B 端消费也不慢，`rwnd` 一直不小。
- 但两地之间跨城专线带宽有限，而且高峰期还有别的业务共用。

这时如果 A 端只看 `rwnd`，会继续加速发送；结果中间链路队列越来越长，出现：

- RTT（Round-Trip Time，往返时延）变大。
- 丢包率升高。
- 重传增多。
- 最终有效吞吐反而下降。

这就是典型的网络拥塞。拥塞控制的价值不在于“把链路跑满”，而在于“在高吞吐和低丢包之间找到平衡点”。

### 3.5 图示

```text
发送方可发送量：

                 +-------------------+
接收方能力 rwnd ->|                   |
                 |  min(rwnd, cwnd)  | -> 实际发送窗口
网络能力 cwnd  -> |                   |
                 +-------------------+
```

拥塞窗口变化：

```text
cwnd
 ^
 |                         拥塞避免：线性增长
 |                       /
 |                     /
 |                   /
 |        慢启动：指数增长
 |      /
 |    /
 |  /
 +--------------------------------> time
        ssthresh
```

### 3.6 慢启动

慢启动 `Slow Start` 不是“慢慢发”，而是从一个较小的 `cwnd` 开始，按 ACK（Acknowledgment，确认应答）反馈快速增长。

典型表现：

```text
1 MSS -> 2 MSS -> 4 MSS -> 8 MSS -> 16 MSS
```

这里的 `MSS` 是 `Maximum Segment Size`，最大报文段长度，可以简单理解为“单个 TCP 数据段里实际负载数据的上限”。

它的目的不是保守到底，而是快速探测网络容量。

面试里经常会被追问：“既然叫慢启动，为什么还是指数增长？”

你可以这么答：

> 因为连接刚建立时并不知道链路容量，如果一开始线性增长，探测太慢；而指数增长能更快接近可用带宽，但一旦到达阈值或检测到丢包，就会立刻转入更保守的阶段。

### 3.7 拥塞避免

当 `cwnd` 增长到慢启动阈值 `ssthresh`（Slow Start Threshold，慢启动阈值）后，进入拥塞避免阶段。  
这时窗口不再指数增长，而是线性增长，避免过快冲击网络。

```text
16 MSS -> 17 MSS -> 18 MSS -> 19 MSS
```

为什么要线性增长？因为到这个阶段，TCP 已经认为自己接近网络承载上限了，再像慢启动那样翻倍，很容易把队列瞬间打满。

可以把它类比成开车：

- 慢启动像刚上高速，先尽快把速度提起来。
- 拥塞避免像已经接近车流极限了，只能小幅加速试探。

### 3.8 快重传

如果发送方连续收到 3 个重复 ACK，说明接收方一直在提醒“某个包没收到，但后面的包我收到了”。  
发送方不用等超时，立即重传丢失的数据段，这叫快重传。

```text
发送：1 2 3 4 5
丢失：3

接收方收到 4：回 ACK=3
接收方收到 5：回 ACK=3
接收方收到 6：回 ACK=3

发送方收到 3 个重复 ACK，立即重传 3
```

这里的关键点是：

- 重复 ACK 说明后续数据还能到达。
- 所以更像是“局部丢包”，不是“整条链路完全堵死”。
- 既然已经有证据说明某个段丢了，就没必要傻等超时。

这个机制的价值是显著缩短丢包恢复时间，避免吞吐量因为等待超时而大幅下降。

### 3.9 快恢复

快恢复认为：既然后续包还能到达，说明网络没有彻底不可用，只是发生了局部丢包。  
所以它不会像超时重传那样把窗口打回很小，而是适度降低窗口后继续传输。

简化理解：

```text
超时重传：拥塞可能严重，cwnd 大幅降低
快恢复：网络仍有吞吐，cwnd 适度降低
```

不同 TCP 实现细节会有差异，但面试里抓住核心就够了：

- 超时重传通常被认为拥塞更严重，反应更激烈。
- 3 个重复 ACK 触发的快恢复通常更温和，因为网络还在流动。

### 3.10 超时重传为什么更严重

如果一个数据段迟迟收不到 ACK，直到 RTO（Retransmission Timeout，重传超时时间）到期才重传，说明问题通常比“单个包偶发丢失”更重，可能是：

- 队列已经严重堆积。
- 某一段链路抖动很大。
- 突发流量把缓冲打满了。

所以超时后一般会把 `cwnd` 降得更狠，重新进入更保守的发送阶段。

### 3.11 面试里可以举的两个例子

#### 3.11.1 例子 1：大文件传输刚开始为什么速度越来越快

因为连接建立初期，`cwnd` 比较小，TCP 不敢一下子发太多；随着 ACK 持续返回，`cwnd` 在慢启动阶段快速扩大，所以你会感觉下载速度前面一小段时间是往上爬的。

#### 3.11.2 例子 2：为什么网络抖动时吞吐会突然掉下来

假设服务间通过 TCP 长连接传输批量数据，本来 `cwnd` 已经比较大。  
突然某个交换机队列打满，发生丢包：

- 如果是少量丢包，可能触发 3 个重复 ACK，进入快重传、快恢复。
- 如果抖动更严重，ACK 长时间回不来，就会触发超时重传。

一旦发生后者，`cwnd` 会明显收缩，吞吐会出现台阶式下降，然后再慢慢爬升。

### 3.12 经典阶段流程

```text
开始发送
  |
  v
慢启动 Slow Start
  |
  | cwnd >= ssthresh
  v
拥塞避免 Congestion Avoidance
  |
  | 收到 3 个重复 ACK
  |-----------------------> 快重传 Fast Retransmit
  |                            |
  |                            v
  |                       快恢复 Fast Recovery
  |                            |
  |                            v
  |---------------------- 回到拥塞避免
  |
  | 发生超时 RTO
  v
大幅缩小 cwnd，重新慢启动
```

### 3.13 流量控制和拥塞控制的区别

| 对比项 | 流量控制 | 拥塞控制 |
|---|---|---|
| 保护对象 | 接收方 | 整个网络 |
| 核心窗口 | `rwnd` | `cwnd` |
| 窗口来源 | 接收方通告 | 发送方维护 |
| 典型问题 | 接收缓冲区满、应用读取慢 | 链路拥堵、路由队列堆积、丢包 |
| 目标 | 不压垮接收端 | 不压垮网络 |

### 3.14 一句话背诵

拥塞控制用 `cwnd` 探测和约束网络承载能力，先通过慢启动快速摸高，再通过拥塞避免谨慎增长；一旦出现重复 ACK 或超时，就用快重传、快恢复或超时重传把速率降下来，避免把网络打爆。

## 4. TCP 粘包和拆包是什么？

### 4.1 面试题

TCP 为什么会粘包、拆包？怎么解决？

### 4.2 最简练版

TCP 是面向字节流的协议，不保留应用层消息边界，所以会出现粘包和拆包。  
粘包是多个应用消息被一次读到，拆包是一个应用消息被分多次读到。  
这不是 TCP 的 bug，而是 TCP 的正常语义。  
解决方式要在应用层协议里定义边界，比如固定长度、分隔符、长度字段，或者使用成熟协议如 HTTP、Protobuf + length prefix。

### 4.3 详细解释版

TCP 只保证字节有序、可靠到达，但不保证应用层一次 `write` 对应接收方一次 `read`。

应用层以为自己发送的是一条条消息：

```text
msg1 | msg2 | msg3
```

但 TCP 看到的只是连续字节流：

```text
abcdefg...
```

接收方读取时，可能一次读到多个消息，也可能一个消息分几次读到。

### 4.4 图示

发送方应用写入：

```text
write("hello")
write("world")
```

接收方可能读到：

```text
情况 1：粘包
read() -> "helloworld"

情况 2：拆包
read() -> "hel"
read() -> "loworld"

情况 3：既粘又拆
write("abc")
write("def")
write("ghi")

read() -> "abcd"
read() -> "efghi"
```

TCP 视角：

```text
应用消息：  [hello] [world] [tcp]
TCP 字节流： h e l l o w o r l d t c p
接收 read： [hellow] [orldtcp]
```

### 4.5 为什么会发生？

常见原因包括：

- TCP 是字节流协议，没有消息边界。
- 发送方可能启用 Nagle 算法，把小包合并发送。
- 接收方读取缓冲区大小不一定等于消息大小。
- 网络传输过程中会受 MSS、MTU、拥塞窗口等影响分段。
- 操作系统 socket 缓冲区会聚合或拆分数据。

### 4.6 怎么解决？

核心原则：应用层协议必须自己定义消息边界。

常见方案：

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| 固定长度 | 每条消息固定 N 字节 | 解析简单 | 浪费空间，不灵活 |
| 分隔符 | 用 `\n`、`\r\n` 等标记边界 | 简单直观 | 内容里要处理转义 |
| 长度字段 | 消息头写 body 长度 | 通用、高效 | 需要处理半包和异常长度 |
| 成熟协议 | HTTP、gRPC、Redis 协议等 | 生态成熟 | 有协议成本 |

长度字段方案示意：

```text
消息格式：

+-------------+----------------+
| length = 12 | body 12 bytes  |
+-------------+----------------+

接收方解析：

1. 先读固定长度消息头
2. 解析 body 长度
3. 再循环读取完整 body
4. 拼出一条完整应用消息
```

### 4.7 Go 中怎么处理？

在 Go 里使用 TCP 连接时，不能假设一次 `conn.Read` 就读到一条完整消息。  
通常要维护一个缓冲区，不断读数据，然后按协议边界解析完整消息。

伪代码：

```go
for {
    n, err := conn.Read(buf)
    if err != nil {
        return err
    }

    decoder.Append(buf[:n])

    for decoder.HasFullFrame() {
        msg := decoder.NextFrame()
        handle(msg)
    }
}
```

### 4.8 常见误区

- 误区 1：认为 TCP 粘包是 TCP 出错了。  
  实际上 TCP 本来就是字节流，没有消息边界。

- 误区 2：认为关闭 Nagle 就能彻底解决粘包。  
  关闭 Nagle 只能减少小包合并，不改变 TCP 字节流语义，仍然需要应用层协议 framing。

- 误区 3：认为一次 `write` 对应一次 `read`。  
  这是最常见的网络编程 bug，尤其在压测或线上高并发时容易暴露。

### 4.9 一句话背诵

粘包拆包的根因是 TCP 面向字节流、不保留消息边界，解决办法是在应用层协议里做 framing，比如长度字段、分隔符或固定长度。

## 5. 四者之间的关系

### 5.1 最简练版

滑动窗口是 TCP 提升传输效率的基础机制。  
流量控制用接收窗口 `rwnd` 约束发送方，避免压垮接收端。  
拥塞控制用拥塞窗口 `cwnd` 约束发送方，避免压垮网络。  
粘包拆包和窗口控制不是一类问题，它来自 TCP 字节流语义，需要应用层协议解决。

### 5.2 详细解释版

可以把 TCP 传输理解成三层约束：

```text
应用层消息边界：由应用层协议定义
TCP 可靠传输：通过序列号、ACK、重传、滑动窗口保证
网络承载能力：通过拥塞控制动态探测
```

真正发送数据时，发送方既要考虑接收方接不接得住，也要考虑网络扛不扛得住：

```text
实际发送窗口 = min(接收窗口 rwnd, 拥塞窗口 cwnd)
```

但无论窗口怎么控制，TCP 传给应用层的仍然是字节流，不会自动恢复应用层消息边界。

### 5.3 总图

```text
应用层：
  负责定义消息边界
  例如 length + body、分隔符、HTTP、gRPC
          |
          v
TCP 层：
  可靠传输：seq / ack / 重传
  传输效率：滑动窗口
  接收保护：rwnd 流量控制
  网络保护：cwnd 拥塞控制
          |
          v
IP / 网络层：
  尽力而为转发，可能延迟、丢包、乱序
```

窗口关系：

```text
                 +----------------+
接收方能力 rwnd ->|                |
                 | min(rwnd,cwnd) | -> 发送方当前最多可发的未确认数据
网络能力 cwnd  -> |                |
                 +----------------+

应用层消息边界：

[msg1][msg2][msg3]  ->  TCP 看成连续字节流  ->  接收方必须自行拆 frame
```

### 5.4 面试收口

面试时可以这样总结：

> 滑动窗口解决的是 TCP 可靠传输下的效率问题；流量控制通过 `rwnd` 防止接收端被打满；拥塞控制通过 `cwnd` 防止网络被打爆；粘包拆包不是 TCP 传输错误，而是字节流协议没有消息边界，必须由应用层协议做 framing。

## 6. 高频追问清单

### 6.1 追问 1：发送窗口到底由谁决定？

### 6.1.1 最简练版

发送窗口通常由 `rwnd` 和 `cwnd` 共同决定，取二者较小值。  
`rwnd` 来自接收方，表示接收能力；`cwnd` 来自发送方，表示网络承载能力。  
所以 TCP 既不能压垮接收端，也不能压垮网络。

### 6.1.2 图示

```text
send_window = min(rwnd, cwnd)

rwnd 小：接收方慢，流量控制生效
cwnd 小：网络拥塞，拥塞控制生效
```

### 6.2 追问 2：UDP 有粘包问题吗？

### 6.2.1 最简练版

UDP 一般不说粘包，因为 UDP 是面向报文的，保留消息边界。  
一次发送一个 UDP datagram，接收方也是按 datagram 接收。  
但 UDP 有丢包、乱序、重复、超过 MTU 可能分片等问题。

### 6.2.2 图示

```text
TCP：字节流
[msg1][msg2] -> bytes -> read 可能读到 [msg1 + msg2 的一部分]

UDP：报文
[datagram1] -> recvfrom 读一个报文
[datagram2] -> recvfrom 读一个报文
```

### 6.3 追问 3：粘包拆包线上会造成什么问题？

### 6.3.1 最简练版

如果协议解析没处理好粘包拆包，线上会出现消息解析错乱、请求阻塞、读到半包、把多条请求当一条请求处理等问题。  
这类问题在低并发本地测试不明显，压测或网络抖动时更容易暴露。

### 6.3.2 图示

```text
错误假设：
一次 Read == 一条消息

真实情况：
一次 Read 可能是：
  半条消息
  一条消息
  多条消息
  半条 + 多条 + 半条
```

### 6.4 追问 4：Nagle 算法和粘包是什么关系？

### 6.4.1 最简练版

Nagle 算法会把小数据合并发送，可能让粘包现象更容易出现。  
但粘包的根因不是 Nagle，而是 TCP 是字节流协议。  
即使关闭 Nagle，也必须在应用层协议里处理消息边界。

### 6.4.2 图示

```text
小包：
"a" "b" "c"

Nagle 可能合并：
"abc"

但关闭 Nagle 后：
TCP 仍然是字节流，read 边界依然不等于 write 边界
```
