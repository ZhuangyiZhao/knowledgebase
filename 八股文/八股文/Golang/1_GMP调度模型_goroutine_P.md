## 1. goroutine 为什么比线程轻量？（基本难度）

### 1.1 最简练版

**goroutine 比线程轻量，核心原因是它由 Go 运行时调度，而不是直接交给操作系统内核调度。**  
线程通常有更大的固定栈，并且创建、销毁、切换都要更多内核参与；`goroutine` 的栈通常从几 KB 起步，可以按需增长和收缩。  
大量 `goroutine` 会复用少量操作系统线程，形成 M:N 调度关系，所以创建成本和切换成本都更低。  
但轻量不代表没有成本，过量 `goroutine` 仍然会带来内存、调度和排查压力。

### 1.2 详细解释版

可以从四个维度理解：

| 维度 | 线程 | goroutine |
|---|---|---|
| 调度者 | 操作系统内核 | Go 运行时 |
| 栈空间 | 通常 MB 级起步，平台相关 | 通常几 KB 起步，按需增长 |
| 数量级 | 太多线程会显著增加内存和调度成本 | 可以支撑大量并发任务 |
| 切换成本 | 往往涉及内核调度 | 多数情况下在用户态完成 |

`goroutine` 适合表达大量并发任务，比如网络请求、后台异步任务、定时任务、流水线处理。  
不过它不是“无限资源”：如果每个 `goroutine` 都阻塞在未释放的 channel、锁、网络调用或定时器上，一样会造成泄漏和内存上涨。

常见追问可以这样补：

- **轻量来自运行时调度，不是来自语法糖。**
- **并行度不等于 goroutine 数量。** 真正同时在 CPU（Central Processing Unit，中央处理器）上运行的数量主要受 `GOMAXPROCS` 和 `P` 数量影响。
- **IO（Input/Output，输入输出）密集型任务更适合大量 goroutine。** CPU 密集型任务开再多 `goroutine` 也不能突破 CPU 核心限制。

### 1.3 图示和例子

```text
很多 goroutine
G1 G2 G3 G4 G5 G6 ...
 \  |  /     \  |  /
  \ | /       \ | /
   M1          M2      少量操作系统线程
    \          /
     \        /
      CPU 核心
```

一句话记忆：

```text
线程是操作系统调度单位，goroutine 是 Go 运行时调度单位；
goroutine 用更小的栈和 M:N 复用，把并发任务表达得更便宜。
```

## 2. GMP 调度模型是什么？（中等难度）

### 2.1 最简练版

**GMP 是 Go 运行时调度 goroutine 的核心模型。**  
`G` 表示 goroutine，也就是待执行任务；`M` 表示 Machine，本质上是操作系统线程；`P` 表示 Processor，是运行 Go 代码所需的执行资源和调度上下文。  
一个 `M` 必须绑定一个 `P`，才能从本地队列或全局队列取出 `G` 执行。  
这个模型通过本地队列、全局队列、网络轮询和 work stealing（工作窃取）提高并发调度效率。

### 2.2 详细解释版

三个对象可以这样理解：

| 对象 | 含义 | 作用 |
|---|---|---|
| `G` | Goroutine | 代表一个可运行或阻塞的任务 |
| `M` | Machine | 代表一个操作系统线程 |
| `P` | Processor | 代表运行 Go 代码需要的执行令牌和调度上下文 |

`M` 和 `P` 最容易混淆，可以重点这样区分：

| 对比点 | `M` | `P` |
|---|---|---|
| 本质 | 操作系统线程 | Go 运行时的执行资源和调度上下文 |
| 谁调度它 | 操作系统内核调度 `M` 上 CPU | Go 运行时管理 `P` 的绑定和队列 |
| 主要职责 | 真正执行代码 | 管理本地运行队列、调度状态、内存分配缓存，并控制 Go 代码并行度 |
| 数量特点 | 运行时按阻塞和调度需求动态创建 | 通常由 `GOMAXPROCS` 决定 |
| 关系 | `M` 必须绑定 `P` 才能执行 Go 代码 | `P` 像执行许可证，交给某个 `M` 后才能跑 `G` |

可以用一个类比记：**`M` 是工人，`P` 是工位，`G` 是任务。工人必须坐到工位上，才能处理 Go 的任务。**  
所以，`M` 负责“实际跑”，`P` 负责“允许跑并提供本地调度资源”。

调度大致过程：

1. 新建 `goroutine` 后，它会成为一个 `G`，通常先放进当前 `P` 的本地运行队列。
2. `M` 绑定 `P` 后，从 `P` 的本地队列取 `G` 执行。
3. 如果本地队列为空，会尝试从全局队列、网络轮询器或其他 `P` 的队列中找任务。
4. 如果某个 `G` 发生阻塞，比如 channel 等待、锁等待、网络等待，运行时会把当前 `M` 腾出来继续调度其他 `G`。
5. 如果 `M` 进入阻塞系统调用，运行时可以把它绑定的 `P` 分离出来，交给其他 `M` 继续跑 Go 代码。

这里的关键点是：**Go 不是一个 goroutine 对应一个线程，而是大量 goroutine 复用少量线程。**

### 2.3 图示和例子

```text
                 全局运行队列
              +----------------+
              | G7 G8 G9 ...   |
              +----------------+
                      ^
                      |
        +-------------+-------------+
        |                           |
  +-----+-----+               +-----+-----+
  |    P1     |               |    P2     |
  | G1 G2 G3 |               | G4 G5 G6 |
  +-----+-----+               +-----+-----+
        |                           |
      bind                        bind
        |                           |
  +-----+-----+               +-----+-----+
  |    M1     |               |    M2     |
  +-----------+               +-----------+
```

如果 `P1` 的本地队列空了，而 `P2` 很忙，`P1` 可能从 `P2` 队列偷一部分 `G` 过来执行，这就是 work stealing。

## 3. P 的作用是什么？为什么需要 P？（中等难度）

### 3.1 最简练版

**P 是运行 Go 代码的执行令牌，也是调度器的本地资源容器。**  
`M` 只有拿到 `P` 才能执行 Go 代码，`P` 的数量通常由 `GOMAXPROCS` 决定，因此它控制了 Go 代码的并行度上限。  
`P` 还维护本地运行队列、调度缓存、定时器等运行时资源，让调度减少全局锁竞争。  
没有 `P`，大量线程直接争抢全局队列，调度成本会更高。

### 3.2 详细解释版

`P` 的价值主要有三点：

| 作用 | 解释 |
|---|---|
| 控制并行度 | 同一时刻最多有多少个 `P` 在运行 Go 代码，通常就对应 Go 层面的并行上限 |
| 降低竞争 | 每个 `P` 有本地运行队列，减少所有 `M` 抢同一个全局队列 |
| 承载运行时资源 | 调度缓存、定时器、内存分配相关缓存等都和 `P` 有关系 |

为什么 `M` 不能直接执行 `G`？  
因为如果只有 `G` 和 `M`，所有线程都去抢全局队列和全局资源，线程一多就会产生严重锁竞争。  
引入 `P` 后，运行时把调度状态分散到每个 `P` 上，既能限制并行度，也能让绝大多数调度发生在本地队列里。

还要注意两个边界：

- `GOMAXPROCS` 控制的是可同时执行 Go 代码的 `P` 数量，不是 `goroutine` 数量。
- `GOMAXPROCS` 也不是直接控制 `M` 的数量。比如 `GOMAXPROCS=4`，表示最多有 4 个 `P`，同一时刻最多有 4 个 `M` 能绑定 `P` 执行 Go 代码。
- `M` 数量可以大于 `P` 数量，因为系统调用、cgo、网络阻塞等场景可能需要额外线程。阻塞的 `M` 可能暂时不执行 Go 代码，但运行时可以把它的 `P` 交给别的 `M` 继续跑任务。

### 3.3 图示和例子

```text
GOMAXPROCS = 2

可同时执行 Go 代码的 P：

P1 -> M1 -> 正在运行 G1
P2 -> M2 -> 正在运行 G2

其他 G：
G3/G4/G5 在运行队列中等待
```

口述时可以压缩成：

```text
M 是操作系统线程，负责实际执行；P 是 Go 运行时的执行许可证和本地调度器。
M 必须拿到 P 才能跑 G。
P 的数量决定 Go 代码的并行度，M 的数量由运行时按阻塞和调度需求动态创建。
```

## 4. goroutine 切换为什么成本低？（中等难度）

### 4.1 最简练版

**goroutine 切换通常由 Go 运行时在用户态完成，不一定需要进入内核调度。**  
切换时主要保存和恢复少量运行上下文，比如栈指针、程序计数器和寄存器状态，成本比线程上下文切换低。  
`goroutine` 栈小，也更容易让调度和内存访问保持较好的缓存局部性。  
但如果涉及系统调用、cgo 或线程阻塞，成本会明显上升。

### 4.2 详细解释版

线程切换通常涉及：

- 从用户态进入内核态；
- 操作系统调度器选择下一个线程；
- 保存和恢复线程上下文；
- 可能带来缓存、TLB（Translation Lookaside Buffer，转译后备缓冲器）等状态失效。

`goroutine` 切换更轻，是因为常见阻塞点由运行时接管：

| 场景 | 运行时行为 |
|---|---|
| channel 阻塞 | 当前 `G` 挂起，`M` 继续找其他 `G` |
| mutex 等待 | 当前 `G` 进入等待队列，释放执行机会 |
| 网络 IO 等待 | 交给 netpoller，等事件就绪后再唤醒 |
| 函数调用安全点 | 运行时可以做协作式或异步抢占 |

所以很多时候不是“线程被切走”，而是“当前 `G` 暂停，当前 `M` 继续跑别的 `G`”。  
这就是 `goroutine` 在高并发网络服务里非常好用的原因。

#### 4.2.1 如何理解“保存和恢复少量运行上下文”

这里的**运行上下文**，可以理解成一个 `goroutine` 暂停后，未来还能从原位置继续执行所必须保存的现场。  
比如代码执行到某个函数中间，如果此时被切走，运行时至少要知道：

| 上下文内容 | 含义 |
|---|---|
| 栈指针 | 当前函数调用栈在哪里 |
| 程序计数器 | 下一条要执行的指令在哪里 |
| 寄存器状态 | 一些临时计算值、参数和中间结果 |

可以用一个类比记：**代码像读书，切换时要保存书签和草稿纸。**  
书签对应“执行到哪一行”，草稿纸对应“临时变量和中间状态”。恢复时把这些现场拿回来，就能继续往下跑。

线程切换和 `goroutine` 切换的差异在于：

```text
线程切换：
操作系统保存线程 A 现场 -> 内核调度线程 B -> 恢复线程 B 现场

goroutine 切换：
Go runtime 保存 G1 现场 -> 同一个 M 继续跑 G2
```

所以 `goroutine` 切换更轻，不是因为它什么都不用保存，而是因为：**保存的现场更小，调度多数发生在用户态，通常不需要完整的内核线程切换。**

不过要避免两个误区：

- **低成本不等于零成本。** 频繁创建短生命周期 `goroutine`、大量定时器、海量阻塞等待，仍然会增加调度压力。
- **切换低不代表业务一定快。** 如果瓶颈在锁竞争、下游调用、内存分配或 GC（Garbage Collection，垃圾回收），光增加 `goroutine` 数量可能会更慢。

### 4.3 图示和例子

```text
G1 执行中
  |
  | channel 接收阻塞
  v
G1 park
  |
  v
M 不退出，继续从 P 的队列取 G2
  |
  v
G2 执行
```

线程切换和 `goroutine` 切换可以这样对比：

| 对比点 | 线程切换 | goroutine 切换 |
|---|---|---|
| 调度层级 | 操作系统内核 | Go 运行时 |
| 是否常进内核 | 常见 | 多数场景不用 |
| 上下文规模 | 线程上下文更重 | `G` 上下文更轻 |
| 常见触发 | 时间片、阻塞、优先级 | channel、锁、网络、抢占点 |
