## 1. Redis 延迟升高怎么排查？（高难度）

### 1.1 最简练版

**Redis 延迟升高要先区分是服务端慢、网络慢、客户端慢，还是下游依赖慢。**  
服务端重点看慢日志、延迟诊断、CPU、内存、持久化、复制和大 key。  
客户端重点看连接池耗尽、超时配置、重试风暴和单次返回数据过大。

### 1.2 详细解释版

排查路径：先确认现象范围，是单业务、单实例、单命令还是全局；再看 `SLOWLOG GET`、`LATENCY DOCTOR`、`INFO commandstats`、`INFO memory`、`INFO persistence`、`INFO replication`。  
如果慢日志没有异常但客户端超时，可能是网络、连接池排队、输出缓冲区或客户端 GC（Garbage Collection，垃圾回收）。如果延迟和 `bgsave`、AOF 重写重合，要重点看 fork 和磁盘。

### 1.3 例子

```bash
slowlog get 20
latency doctor
info commandstats
info memory
```

## 2. Redis CPU 打满怎么排查？（高难度）

### 2.1 最简练版

**CPU 打满通常来自高 QPS、慢命令、大 key、复杂 Lua、过期删除、内存碎片整理或集群迁移。**  
先看命令统计和慢日志，再看业务流量是否突增。  
如果是单个热点 key，要做热点拆分、本地缓存或限流。

### 2.2 详细解释版

`INFO commandstats` 可以看各命令调用次数和耗时，`MONITOR` 信息最全但线上慎用。CPU 高还可能来自大量 key 同时过期、`active defrag`、AOF 重写、slot 迁移。  
治理方式包括限制高风险命令、分页扫描、拆大 key、热点副本、本地缓存、扩容分片和客户端限流。

### 2.3 例子

```bash
info commandstats
slowlog get 10
top -Hp $(pidof redis-server)
```

## 3. Redis 连接数打满怎么处理？（中等难度）

### 3.1 最简练版

**连接数打满要先确认是正常流量增长、连接泄漏、短连接风暴还是阻塞导致连接堆积。**  
临时处理可以扩连接上限和限流，根因治理要优化连接池复用。  
不能只盲目调大 `maxclients`，还要看系统 fd（File Descriptor，文件描述符）和内存。

### 3.2 详细解释版

排查 `connected_clients`、`blocked_clients`、客户端来源和连接年龄。短连接风暴通常来自未使用连接池或健康检查过于频繁；连接泄漏来自未释放连接；阻塞堆积来自 Redis 慢或下游线程池卡住。  
处理时要设置合理连接池、超时、最大空闲、最大等待时间，并按服务维度限流。

### 3.3 例子

```bash
info clients
client list
config get maxclients
```

## 4. 客户端连接池怎么设置？（中等难度）

### 4.1 最简练版

**连接池要根据并发、单请求 Redis 次数、平均耗时和服务实例数设置。**  
太小会排队超时，太大会把 Redis 连接打满。  
核心是复用连接、限制等待、设置超时和监控池状态。

### 4.2 详细解释版

估算时可以用 `并发请求数 * 每请求 Redis 调用数 * 平均耗时 / 请求耗时` 粗略判断需要连接数，再乘以安全系数。  
还要控制每个应用实例的最大连接数，避免多实例扩容后 Redis 总连接数爆炸。监控包括活跃连接、空闲连接、等待队列、获取连接耗时和超时次数。

### 4.3 例子

```text
单实例 200 并发，每请求 2 次 Redis，Redis 平均 2ms，请求 100ms
估算活跃连接约 200 * 2 * 2 / 100 = 8，再留余量
```

## 5. Redis 超时应该怎么设置？（中等难度）

### 5.1 最简练版

**Redis 超时要短而可控，不能让请求长时间卡死。**  
连接超时、读超时、写超时和连接池等待超时都要分别设置。  
超时后要有降级和限流，避免重试风暴。

### 5.2 详细解释版

缓存访问通常在毫秒级，超时应明显低于接口总超时预算。比如接口 SLA（Service Level Agreement，服务等级协议）是 200ms，Redis 读超时不能设成几秒。  
重试要谨慎，只适合幂等读或短暂网络抖动，并加退避和熔断。写操作超时要考虑是否已经执行成功，避免重复扣减。

### 5.3 例子

```text
连接超时：50ms
读超时：100ms
连接池等待：20ms
失败后：本地缓存/默认值/限流
```

## 6. 慢日志怎么用？（基本难度）

### 6.1 最简练版

**慢日志记录执行时间超过阈值的命令，用来定位慢命令。**  
它只统计命令执行耗时，不包含网络传输和客户端排队。  
常用 `SLOWLOG GET` 查看，`slowlog-log-slower-than` 配置阈值。

### 6.2 详细解释版

慢日志能发现 `hgetall`、`smembers`、大范围 `zrange`、复杂 Lua 等服务端耗时命令。  
如果客户端超时但慢日志没有记录，要继续查网络、连接池、客户端序列化、输出缓冲区和系统调度。阈值不宜太高，否则抓不到问题；也不宜太低，否则噪声过多。

### 6.3 例子

```bash
config get slowlog-log-slower-than
slowlog len
slowlog get 10
```

## 7. 如何避免阻塞 Redis 主线程？（中等难度）

### 7.1 最简练版

**避免阻塞主线程的关键是控制单命令处理数据量。**  
不要在线上使用全量扫描、大集合全量读取、复杂 Lua 和同步删除大 key。  
改用 scan 分批、分页、拆 key、`unlink` 和异步任务。

### 7.2 详细解释版

Redis 是快速处理大量短命令的系统，不适合单条命令搬运大量数据。集合操作要限制范围，聚合尽量前置到写入时维护。  
删除大 key 用 `UNLINK` 异步释放；清理业务数据用游标分批；Lua 只做小而确定的原子逻辑。

### 7.3 例子

```bash
hscan big:hash 0 count 500
unlink old:big:key
```

## 8. Redis 分布式锁的基本实现是什么？（中等难度）

### 8.1 最简练版

**Redis 分布式锁通常用 `SET key value NX PX ttl` 实现加锁。**  
`NX` 保证不存在才设置，`PX` 设置过期时间，value 用唯一随机值标识持有者。  
释放锁要用 Lua 校验 value 后删除，避免误删别人的锁。

### 8.2 详细解释版

加锁必须是“判断不存在 + 设置值 + 设置过期时间”的原子操作，因此不能用 `setnx` 后再单独 `expire`。value 通常用 UUID（Universally Unique Identifier，通用唯一识别码）或请求 ID。  
锁适合保护短临界区，不适合长事务和强一致资源协调。业务本身还要做幂等和状态校验。

### 8.3 例子

```bash
set lock:order:1001 request-uuid nx px 30000
```

## 9. 为什么释放锁要校验 value？（中等难度）

### 9.1 最简练版

**校验 value 是为了确认锁还是自己持有，防止误删别人的锁。**  
如果业务执行超过锁过期时间，锁可能已被其他请求重新获得。  
此时旧请求直接 `del` 会删除新请求的锁。

### 9.2 详细解释版

正确释放锁需要把“读取 value、比较、删除”放到 Lua 中原子执行。否则客户端先 `get` 看到是自己的值，随后锁过期并被别人加锁，再执行 `del` 仍会误删。  
value 是锁持有者身份，TTL 是故障兜底，两者缺一不可。

### 9.3 例子

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
  return redis.call('del', KEYS[1])
end
return 0
```

## 10. 分布式锁过期时间怎么设置？（高难度）

### 10.1 最简练版

**锁过期时间要大于临界区正常执行时间，并留出 GC、网络和下游抖动余量。**  
太短会导致锁提前过期并发进入，太长会降低故障恢复速度。  
长任务要考虑续期，但续期也必须和持有者身份绑定。

### 10.2 详细解释版

设置 TTL 前要统计 P99（Percentile 99，第 99 百分位）耗时，而不是只看平均值。比如临界区 P99 是 800ms，可以设置 3s 或 5s，并限制最大执行时间。  
如果使用 watchdog（看门狗）续期，要确保任务结束能停止续期，异常时不会无限持锁。对关键业务，锁只是并发控制，还需要数据库唯一约束或状态机兜底。

### 10.3 例子

```text
锁 TTL = 临界区 P99 * 3 + 网络抖动余量
```

## 11. Redlock 是否一定可靠？（高难度）

### 11.1 最简练版

**Redlock 通过多个独立 Redis 节点加锁来提高可靠性，但不能简单理解为绝对可靠。**  
它依赖时钟、网络延迟和多数节点成功。  
强一致资源协调更推荐使用数据库事务、ZooKeeper、etcd 等带一致性协议的系统。

### 11.2 详细解释版

Redlock 的思路是在 N 个独立节点上尝试加锁，超过多数成功且耗时小于锁有效期才认为成功。它比单实例锁抗单点故障更好。  
争议点在于网络分区、进程暂停、时钟漂移和锁过期后业务仍执行等场景。实践中要根据业务风险选择：普通防重复可用 Redis 锁，金融级互斥要用更强一致的协调方案。

### 11.3 表格

| 场景 | 建议 |
|---|---|
| 防重复提交 | Redis 锁可接受 |
| 定时任务抢占 | Redis 锁 + 幂等 |
| 资金状态变更 | 数据库约束或一致性协调系统 |

## 12. Redis 如何做接口限流？（中等难度）

### 12.1 最简练版

**Redis 可以用计数器、滑动窗口、令牌桶等方式做分布式限流。**  
简单场景用 `incr + expire` 固定窗口。  
更平滑的限流可以用 ZSet 滑动窗口或令牌桶算法。

### 12.2 详细解释版

固定窗口实现简单但边界会突刺；滑动窗口更精确但存储和计算更贵；令牌桶允许一定突发，适合接口保护。  
限流维度可以是用户、IP、设备、接口、租户。核心链路要用 Lua 保证判断和更新原子。

### 12.3 例子

```text
limit:user:1001:/pay -> 每分钟最多 10 次
```

## 13. 固定窗口和滑动窗口限流有什么区别？（中等难度）

### 13.1 最简练版

**固定窗口按固定时间片计数，实现简单但边界可能放大流量。**  
滑动窗口按最近一段时间统计，更平滑更准确。  
代价是滑动窗口需要保存请求时间或多个子窗口，成本更高。

### 13.2 详细解释版

固定窗口中，用户可能在上一分钟最后 1 秒和下一分钟最初 1 秒各打满配额，形成瞬时双倍流量。  
滑动窗口用 ZSet 记录时间戳，每次清理窗口外请求并统计窗口内数量。它适合关键接口，但高 QPS 场景要控制存储量或改用令牌桶。

### 13.3 图示

```text
固定窗口：| 60s | 60s |
滑动窗口：任意时刻向前看最近 60s
```

## 14. Redis 如何实现排行榜？（基本难度）

### 14.1 最简练版

**排行榜通常用 ZSet 实现，member 是用户或对象 ID，score 是分数。**  
`zadd` 更新分数，`zrevrange` 查榜单，`zrevrank` 查排名。  
ZSet 天然适合按分数排序和范围查询。

### 14.2 详细解释版

排行榜要先定义维度：总榜、日榜、周榜、活动榜。key 可以按榜单和周期拆分，例如 `rank:game:daily:20260702`。  
查询前 N 名、用户周围排名、分页榜单都可以基于 ZSet。超大榜单要注意内存、分页深度和历史榜单清理。

### 14.3 例子

```bash
zadd rank:game:daily:20260702 100 user1
zrevrange rank:game:daily:20260702 0 99 withscores
zrevrank rank:game:daily:20260702 user1
```

## 15. 排行榜如何处理并列和分页？（中等难度）

### 15.1 最简练版

**ZSet score 相同的成员会按字典序排序，如果业务要严格并列，需要额外设计规则。**  
分页可以用 rank 范围，但深分页成本和结果稳定性要关注。  
常见做法是 score 编码时间或用二级字段补充排序。

### 15.2 详细解释版

如果希望同分先达到者靠前，可以把 score 设计成 `业务分数 * 放大倍数 + 时间反向因子`，但要避免精度问题。也可以把 ZSet 只作为排序索引，详情和并列展示在业务层处理。  
分页时，榜单实时变化会造成翻页重复或漏项。高要求场景要用榜单快照。

### 15.3 例子

```text
score = 积分 * 1_000_000 + (999999 - 秒级时间偏移)
```

## 16. Redis 如何实现延迟队列？（中等难度）

### 16.1 最简练版

**延迟队列常用 ZSet 实现，score 存执行时间戳，member 存任务 ID。**  
消费者轮询取 score 小于当前时间的任务，用 Lua 原子获取并删除。  
它适合轻量延迟任务，不适合超大规模可靠调度。

### 16.2 详细解释版

基本流程：`zadd delay_queue execute_at task_id`；消费者 `zrangebyscore` 找到到期任务；用 Lua 确认并 `zrem`，再执行业务。  
要处理重复消费、执行失败重试、死信队列、任务内容存储和消费者并发。关键任务建议使用专业 MQ 延迟消息或调度系统。

### 16.3 图示

```text
ZSet(score=执行时间)
  |
到期扫描 -> 原子领取 -> 执行 -> 失败重试/死信
```

## 17. Redis 做幂等去重怎么设计？（中等难度）

### 17.1 最简练版

**Redis 做幂等去重通常用唯一业务 key 加 `set nx ex`。**  
第一次请求设置成功就执行业务，重复请求设置失败则直接返回或查询结果。  
TTL 要覆盖业务可能重复提交的时间窗口。

### 17.2 详细解释版

幂等 key 可以由业务类型、用户 ID、请求号、订单号组成。Redis 去重适合防重复点击、消息消费去重、接口幂等入口保护。  
但关键数据仍要有数据库唯一约束兜底，因为 Redis key 可能过期、丢失或被误删。对于异步消费，处理结果也要可查询，避免重复请求不知道上次结果。

### 17.3 例子

```bash
set idem:pay:order123 request-id nx ex 86400
```

## 18. Redis 作为会话存储要注意什么？（中等难度）

### 18.1 最简练版

**Redis 存会话要关注 TTL、续期、安全、容量和高可用。**  
会话 key 通常按 token 或 session ID 存储，并设置过期时间。  
敏感信息不要明文存太多，退出登录要主动删除。

### 18.2 详细解释版

会话访问频繁，可以按活跃度滑动续期，但续期频率要控制，避免每次请求都写 Redis。多端登录、踢人、权限变更要有版本号或用户态字段。  
Redis 故障时要明确是拒绝登录态、短期使用本地缓存，还是降级为只读访问。持久化和主从切换也会影响会话可靠性。

### 18.3 例子

```text
session:{token} -> user_id、scope、version、expire_at
```

## 19. Redis 故障演练应该覆盖什么？（高难度）

### 19.1 最简练版

**故障演练要覆盖 Redis 不可用、延迟升高、主从切换、数据丢失、连接池耗尽和缓存雪崩。**  
目标不是证明不会出事，而是验证降级、限流、告警和恢复是否有效。  
演练后要沉淀阈值、预案和自动化脚本。

### 19.2 详细解释版

典型演练项：杀主库观察切换时间；注入延迟观察客户端超时；模拟大 key 阻塞；清空热点缓存观察 DB 保护；调低连接数观察连接池行为；断开从库观察复制恢复。  
关键指标包括接口成功率、P99 延迟、DB QPS、Redis 命中率、错误率、切换耗时和人工介入时间。

### 19.3 清单

```text
不可用 -> 是否降级
慢请求 -> 是否超时熔断
缓存失效 -> DB 是否被保护
主从切换 -> 客户端是否自动恢复
```

## 20. 你做过 Redis 优化项目可以怎么讲？（高难度）

### 20.1 最简练版

**可以按背景、问题、方案、结果、复盘来讲。**  
比如某接口高峰 Redis 延迟升高，定位到大 key 和热点 key，改成拆分存储、本地缓存、分页访问和异步删除。  
最后用量化指标说明收益，比如 P99 降低、DB QPS 降低、错误率下降。

### 20.2 详细解释版

表达时不要只说“加了缓存”，要讲清楚：业务背景是什么，瓶颈指标是什么，如何定位到 Redis，发现了哪些慢命令和 key，为什么选择当前方案，有什么风险和兜底。  
结果最好量化：命中率从多少提升到多少，P99 从多少降到多少，大 key 数量下降多少，故障恢复时间缩短多少。复盘要补充规范：key 设计、命令白名单、容量告警、压测和演练。

### 20.3 模板

```text
背景：活动页高峰延迟升高
定位：slowlog + bigkeys 发现大 Hash 和热点配置
方案：拆 key、hscan 分页、本地缓存、TTL 抖动、unlink 清理
结果：P99 下降，Redis CPU 下降，DB 回源下降
复盘：接入 key 规范和大 key 巡检
```
