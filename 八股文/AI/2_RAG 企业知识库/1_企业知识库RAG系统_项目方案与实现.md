## 1.2 最简练版

企业知识库 RAG（Retrieval-Augmented Generation，检索增强生成）系统的本质是：**先从企业私有文档里找证据，再让大语言模型基于证据回答**。它解决的是通用模型不知道内部知识、容易编造、答案无法追溯的问题。这个项目最适合做成“后端工程化作品”，因为它不只考模型调用，还覆盖文档导入、切片、索引、混合检索、Rerank、权限、引用溯源、质量评测和线上观测。建议你选“后端故障手册 + 项目接口文档 + 数据库规范”的知识库场景，既贴合你的后端经验，也容易讲出性能、稳定性和工程取舍。

## 1.3 详细解释版

### 1.3.1 这个项目要解决什么问题？（基本难度）

可以这样回答：

> 这个系统面向企业内部知识问答。用户问一个业务、接口、故障或规范问题时，系统不会直接把问题丢给模型，而是先做权限校验、问题改写和知识检索，从内部文档中召回相关片段，再把片段作为上下文交给模型生成答案。答案必须带引用来源，如果检索证据不足就拒答。这样可以降低幻觉，提高可追溯性，也方便后续用评测集持续优化质量。

核心价值：

| 问题 | RAG 解决方式 |
|---|---|
| 模型不知道企业内部知识 | 把内部文档切片、向量化、建索引 |
| 模型容易编造 | Prompt 约束“只基于引用材料回答” |
| 答案不可追溯 | 每个答案返回文档名、章节、Chunk ID |
| 知识更新慢 | 支持增量导入和重建索引 |
| 质量不可衡量 | 建评测集，统计 Recall@5、引用准确率、拒答率 |
| 企业权限复杂 | 检索前按用户、部门、项目过滤 metadata |

### 1.3.2 有哪些适合做作品集的好点子？（中等难度）

| 方向 | 项目点子 | 亮点 | 难度 | 推荐指数 |
|---|---|---|---|---|
| 后端稳定性 | **后端故障手册 RAG 助手** | 能结合日志、告警、排障流程，后续可升级为 Agent | 中等 | 高 |
| 数据库治理 | **MySQL 规范与慢 SQL 分析知识库** | 贴合索引、执行计划、SQL 优化，可展示工程深度 | 中等 | 高 |
| 研发效能 | **项目接口文档智能问答系统** | 可解析接口文档、错误码、变更记录，业务感强 | 中等 | 高 |
| 客服运营 | 智能客服知识库 | 场景常见，但容易像通用 demo | 低 | 中 |
| 合规风控 | 内部制度与权限问答系统 | 权限、审计、拒答空间大 | 中等 | 中 |
| 代码知识库 | 代码仓库 RAG 助手 | 有技术含量，但解析代码和依赖关系更复杂 | 高 | 中 |

最建议你做这三个组合：

1. **后端故障手册 RAG 助手**：把 MySQL、Redis、Kafka、Linux、网络排障文档导入，回答“线上延迟升高怎么排查”。
2. **接口与业务规则知识库**：导入接口说明、错误码、字段含义、灰度规则，回答“某个订单状态为什么不能退款”。
3. **SQL 优化知识库**：导入慢 SQL 案例、索引规范、Explain 说明，回答“这个 SQL 为什么没走索引”。

### 1.3.3 推荐选哪个题目？（中等难度）

推荐题目：

> **面向后端团队的故障手册与工程规范 RAG 系统**

推荐原因：

| 维度 | 说明 |
|---|---|
| 和你的背景匹配 | 你可以把后端排障、数据库、缓存、消息队列经验放进去 |
| 容易做出差异化 | 不只是知识问答，还能回答“怎么排查、看哪些指标、先止血还是定位” |
| 能自然扩展 Agent | 第 3 个月可以让 Agent 调用这个 RAG 系统查询故障手册 |
| 可量化 | 能统计命中率、拒答率、延迟、引用准确率 |
| 好讲取舍 | Chunk 粒度、混合检索、Rerank、权限过滤、缓存、降级都能展开 |

项目一句话：

> 做一个给后端研发使用的内部知识库系统，支持导入故障手册、接口文档和工程规范，用户提问后返回带引用的答案，并在证据不足时拒答。

### 1.3.4 系统边界怎么定？（中等难度）

第一版不要做太大，重点把 RAG 主链路做完整。

| 阶段 | 必做 | 暂缓 |
|---|---|---|
| MVP | Markdown / TXT 导入、切片、索引、检索、引用、拒答、评测 | 多租户复杂管理、漂亮前端 |
| 强化版 | PDF 解析、混合检索、Rerank、metadata 过滤、缓存、SSE（Server-Sent Events，服务端事件流） | 复杂工作流、自动写操作 |
| 生产版 | PostgreSQL + pgvector、对象存储、异步任务、Prometheus 观测、审计日志 | 从零训练模型 |

MVP 的验收标准：

| 能力 | 验收方式 |
|---|---|
| 文档导入 | 放入 30 篇后端文档，一条命令完成索引 |
| 检索 | 输入问题能返回 Top 5 相关 Chunk |
| 问答 | 回答必须带引用来源 |
| 拒答 | 问知识库外的问题时明确说明证据不足 |
| 评测 | 用 30 到 50 条问题统计 Recall@5 |

### 1.3.5 核心链路怎么设计？（中等难度）

离线链路：

1. 读取 Markdown / TXT / PDF 文档。
2. 清洗文本，去掉噪声、目录、重复空行。
3. 按标题和长度切成 Chunk。
4. 生成 Embedding（Embedding，向量表示）。
5. 写入向量库和关键词索引。
6. 记录 metadata，包括文档名、章节、权限标签、更新时间。

在线链路：

1. 用户发起问题。
2. 鉴权，确定可访问的项目、部门、文档类型。
3. 问题改写，补齐上下文。
4. 向量检索召回语义相近片段。
5. 关键词检索召回关键词匹配片段。
6. 融合排序，取 Top N 候选。
7. Rerank 重排，取 Top K 证据。
8. 分数低于阈值时拒答。
9. 构造 Prompt，让模型基于证据回答。
10. 返回答案、引用、耗时、检索分数和请求 ID。

### 1.3.6 存储模型怎么设计？（中等难度）

| 表 | 关键字段 | 说明 |
|---|---|---|
| `documents` | `id`、`title`、`source_uri`、`doc_type`、`owner`、`updated_at` | 文档元信息 |
| `chunks` | `id`、`document_id`、`section_path`、`content`、`embedding`、`metadata` | 文档切片和向量 |
| `queries` | `id`、`user_id`、`question`、`answer`、`latency_ms`、`hit_count` | 问答日志 |
| `query_citations` | `query_id`、`chunk_id`、`score`、`rank` | 引用溯源 |
| `eval_cases` | `question`、`expected_chunk_ids`、`expected_answer` | 离线评测集 |

MVP 可以先用 JSON 文件存索引，方便理解链路；生产版建议用 PostgreSQL + pgvector 或 Milvus。

### 1.3.7 检索质量怎么优化？（高难度）

检索质量通常不是靠“换模型”一次性解决，而是持续调参和坏例迭代。

| 问题 | 表现 | 优化方向 |
|---|---|---|
| Chunk 太大 | Top 5 命中但上下文噪声多 | 按标题切片，控制 300 到 800 字 |
| Chunk 太小 | 召回片段缺上下文 | 加 overlap，保留标题路径 |
| 只用向量检索 | 专有名词、错误码、字段名召回差 | 加 BM25（Best Matching 25，经典关键词相关性算法） |
| 只用关键词检索 | 同义表达召回差 | 加 Embedding 语义检索 |
| Top N 噪声多 | 模型引用错材料 | 引入 Rerank |
| 文档质量差 | 答案不稳定 | 清洗、去重、补标题、补 FAQ |

混合检索融合公式可以先简单做：

```text
final_score = 0.6 * vector_score + 0.4 * keyword_score
```

后续根据评测结果调权重，而不是凭感觉调。

### 1.3.8 如何处理无答案和幻觉？（高难度）

无答案处理要靠三层约束：

| 层次 | 做法 |
|---|---|
| 检索层 | Top 1 分数低于阈值或 Top K 分数整体偏低时拒答 |
| Prompt 层 | 明确要求“引用材料没有依据时，不要猜测” |
| 后处理层 | 检查答案是否包含引用，没有引用则降级为“证据不足” |

常见坑：

- *不能只靠模型自己判断有没有答案*，因为模型会倾向于补全一个看似合理的回答。
- *不能把无关 Chunk 硬塞给模型*，否则模型会从噪声中编出关系。
- *不能只看相似度分数*，专有名词命中、章节路径、文档类型也要作为判断依据。

### 1.3.9 如何做权限隔离？（高难度）

企业知识库一定要先过滤权限，再做答案生成。

| 位置 | 控制点 |
|---|---|
| 文档导入 | 给文档写入 `tenant_id`、`project_id`、`department`、`security_level` |
| 检索前 | 根据用户身份生成 metadata filter |
| 检索中 | 向量库和关键词索引都必须带过滤条件 |
| Prompt 构造 | 只允许拼接用户有权限访问的 Chunk |
| 日志审计 | 记录用户、问题、命中文档、引用、耗时 |

可以这样表达：

> 权限不能放在模型回答之后做脱敏，因为模型可能已经看到不该看的上下文。正确做法是在检索前和检索中完成权限过滤，保证进入 Prompt 的材料本身就是用户可见的。

### 1.3.10 如何做评测和观测？（高难度）

离线评测指标：

| 指标 | 含义 | 目标 |
|---|---|---|
| Recall@5 | Top 5 是否包含标准命中 Chunk | 70% 到 85% 起步 |
| MRR（Mean Reciprocal Rank，平均倒数排名） | 正确 Chunk 排名越靠前越好 | 持续提升 |
| 引用准确率 | 答案引用是否真能支撑结论 | 80% 以上 |
| 拒答率 | 无答案问题是否能拒答 | 看业务要求 |
| 坏例分类 | 切片问题、召回问题、排序问题、Prompt 问题 | 每周复盘 |

线上观测指标：

| 指标 | 说明 |
|---|---|
| QPS（Queries Per Second，每秒查询数） | 系统吞吐 |
| p95 延迟 | 关注检索、Rerank、模型生成分别耗时 |
| 错误率 | 模型超时、索引失败、解析失败 |
| Token 成本 | 每次请求输入、输出 Token 和费用 |
| 缓存命中率 | 高频问题是否减少模型调用 |
| 低分检索占比 | 发现知识库缺口或切片问题 |

### 1.3.11 如何分阶段落地？（中等难度）

| 周期 | 目标 | 产出 |
|---|---|---|
| 第 1 周 | 文档导入和切片 | 支持 Markdown / TXT，生成 Chunk JSON |
| 第 2 周 | 索引和检索 | 支持本地向量检索和关键词检索 |
| 第 3 周 | 问答和引用 | 返回答案、引用、拒答 |
| 第 4 周 | 评测和优化 | 评测集、Recall@5、坏例分析 |
| 第 5 周 | 工程化升级 | FastAPI、审计日志、缓存、配置化 |
| 第 6 周 | 生产版替换 | 接入真实 Embedding、pgvector、LLM |

### 1.3.12 代码实现说明

当前目录下的 `enterprise_rag_system` 是一个可运行的轻量版本，故意少依赖，方便先理解完整链路。

| 文件 | 作用 |
|---|---|
| `rag_app/ingest.py` | 读取文档、按标题和长度切片、写入索引 |
| `rag_app/retriever.py` | 关键词分、向量分、混合检索和简单重排 |
| `rag_app/generator.py` | 基于引用材料生成可追溯回答 |
| `rag_app/evaluator.py` | 读取评测集，计算 Recall@5 |
| `rag_app/main.py` | 命令行入口 |
| `data/docs` | 示例知识库文档 |
| `data/eval/eval_cases.jsonl` | 示例评测集 |

运行方式：

```bash
cd "AI/2_RAG 企业知识库/enterprise_rag_system"
python3 -m rag_app.main ingest
python3 -m rag_app.main ask "Redis 缓存击穿怎么处理？"
python3 -m rag_app.main search "慢 SQL 为什么没有走索引？"
python3 -m rag_app.main eval
```

## 1.5 高频追问

| 问题 | 回答要点 |
|---|---|
| 为什么 RAG 能降低幻觉？ | 因为它把回答约束在外部证据上，并通过引用和拒答降低编造概率 |
| Chunk 切太大和太小有什么问题？ | 太大噪声多，太小缺上下文；要结合标题路径和 overlap |
| 为什么要混合检索？ | 向量适合同义语义，关键词适合错误码、字段名、专有名词 |
| 为什么要 Rerank？ | 初召回追求召回率，Rerank 追求排序精度 |
| 没有答案怎么办？ | 分数阈值、Prompt 约束、引用校验三层拒答 |
| 权限怎么做？ | 进入 Prompt 前必须按 metadata 过滤，不能回答后再脱敏 |
| 如何降低延迟？ | 缓存、并行检索、减少 Top N、Rerank 只处理候选、流式输出 |
| 如何证明系统变好了？ | 用固定评测集比较 Recall@5、MRR、引用准确率和坏例数量 |
