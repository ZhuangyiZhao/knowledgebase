# 幻读、不可重复读、当前读 / 快照读

**适用场景：** 隔离级别细节、RR 是否解决幻读、普通 `select` 和 `for update` 区别、并发读写场景题。

**回答模板：** 先区分不可重复读和幻读；再说明快照读读历史可见版本，当前读读最新并加锁；然后解释 InnoDB RR 为什么普通快照读结果稳定，以及范围当前读如何通过 next-key lock 控制幻读。

## 1. 不可重复读是什么？

### 1.1 最简练版

**不可重复读是同一个事务内，两次读取同一行数据，结果不一样。**  
它关注的是“同一行内容被别的事务修改并提交”。  
RC 隔离级别下可能出现不可重复读。  
RR 隔离级别下，普通快照读通常可以避免不可重复读。

### 1.2 例子

初始数据：

```text
account(id=1, balance=100)
```

时间线：

```text
事务 A begin
事务 A select balance where id = 1 -> 100

事务 B update account set balance = 80 where id = 1
事务 B commit

事务 A select balance where id = 1 -> 80
```

事务 A 两次读同一行，余额从 `100` 变成 `80`。这就是不可重复读。

### 1.3 本质

不可重复读关注的是：

```text
同一条记录
  -> 被其他事务 update/delete
  -> 当前事务再次读取时内容变化
```

## 2. 幻读是什么？

### 2.1 最简练版

**幻读是同一个事务内，两次按同一条件查询，结果集合发生变化。**  
它关注的是“满足条件的记录变多或变少”，尤其是别的事务插入了新记录。  
和不可重复读相比，幻读更强调范围查询和记录集合变化。  
InnoDB RR 下，快照读和当前读分别用 MVCC 与 next-key lock 处理大部分幻读问题。

### 2.2 例子

初始数据：

```text
order_tab 中 status = 'created' 的订单有 10 条
```

时间线：

```text
事务 A begin
事务 A select count(*) from order_tab where status = 'created' -> 10

事务 B insert into order_tab(status) values('created')
事务 B commit

事务 A select count(*) from order_tab where status = 'created' -> 11
```

事务 A 第二次查询时，像出现了“幻影行”。这就是幻读。

### 2.3 和不可重复读的区别

| 类型 | 关注点 | 常见触发 |
|---|---|---|
| 不可重复读 | 同一行内容变化 | update/delete |
| 幻读 | 结果集合变化 | insert/delete |

一句话背诵：

```text
不可重复读看的是行内容变了，幻读看的是结果集合变了。
```

## 3. 快照读是什么？

### 3.1 最简练版

**快照读是普通 `select` 通过 MVCC 读取某个历史一致性视图。**  
它不一定读最新数据，而是读对当前 ReadView 可见的数据版本。  
快照读通常不加行锁，所以读写并发能力好。  
在 RR 下，同一事务内普通快照读通常复用第一次 ReadView。

### 3.2 常见 SQL

普通查询通常是快照读：

```sql
select *
from order_tab
where order_id = 1001;
```

范围普通查询也是快照读：

```sql
select count(*)
from order_tab
where status = 'created';
```

### 3.3 快照读特点

| 特点 | 说明 |
|---|---|
| 读历史版本 | 根据 ReadView 找可见版本 |
| 通常不加锁 | 不阻塞普通写 |
| 不一定最新 | 可能读到旧快照 |
| RR 下稳定 | 同一事务内多次普通读通常一致 |

## 4. 当前读是什么？

### 4.1 最简练版

**当前读读取的是最新已提交版本，并且通常会加锁。**  
`update`、`delete`、`insert`、`select ... for update`、`select ... for share` 都属于当前读相关操作。  
当前读要面向最新数据做修改或锁定，所以不能只读历史快照。  
在 RR 下，范围当前读常通过 next-key lock 防止幻读。

### 4.2 常见 SQL

```sql
select *
from order_tab
where order_id = 1001
for update;
```

```sql
update order_tab
set status = 'paid'
where order_id = 1001;
```

```sql
delete from order_tab
where created_at < '2026-01-01';
```

这些都要面对最新数据，并且可能加锁。

### 4.3 为什么 update 不能读旧快照？

假设事务 A 的快照里库存是 `10`，但其他事务已经把库存扣到 `0` 并提交。  
如果事务 A 的 `update` 还按旧快照扣库存，就会造成严重错误。  
所以更新必须基于当前最新可修改版本，并通过锁控制并发写。

## 5. RR 到底如何处理幻读？

### 5.1 最简练版

**InnoDB RR 处理幻读要分快照读和当前读。**  
普通快照读复用 ReadView，因此同一事务内不会看到其他事务后来插入的新记录。  
当前读不能读旧快照，所以范围当前读会通过 next-key lock 锁住记录和间隙，阻止其他事务插入幻影行。  
因此更准确的表述是：InnoDB RR 在大部分常见场景下能避免幻读。

### 5.2 快照读下的幻读

时间线：

```text
事务 A begin
事务 A 普通 select count(*) where status = 'created' -> 10

事务 B insert status = 'created'
事务 B commit

事务 A 普通 select count(*) where status = 'created' -> 10
```

在 RR 下，事务 A 第二次普通查询复用 ReadView，看不到事务 B 后来提交的新行。

### 5.3 当前读下的幻读

如果事务 A 是范围当前读：

```sql
select *
from order_tab
where amount between 100 and 200
for update;
```

InnoDB 在 RR 下会对扫描到的索引范围加 next-key lock。  
如果事务 B 想插入 `amount = 150` 的订单，就会被阻塞，因为这个插入落在事务 A 锁住的间隙里。

### 5.4 混用快照读和当前读的坑

同一个事务里，普通 select 和 `for update` 可能看到不同结果：

```text
事务 A begin
事务 A 普通 select -> 看到旧快照

事务 B update 并 commit

事务 A select ... for update -> 读当前最新并加锁
```

这不是 MVCC 失效，而是两种读语义不同。

## 6. 典型时间线例子

### 6.1 RC 下不可重复读

```text
事务 A begin
A: select balance where id = 1 -> 100

事务 B begin
B: update account set balance = 80 where id = 1
B: commit

A: select balance where id = 1 -> 80
A: commit
```

原因：RC 每条语句生成新 ReadView。

### 6.2 RR 下普通快照读可重复

```text
事务 A begin
A: select balance where id = 1 -> 100

事务 B begin
B: update account set balance = 80 where id = 1
B: commit

A: select balance where id = 1 -> 100
A: commit
```

原因：RR 下事务 A 复用第一次快照读的 ReadView。

### 6.3 RR 下当前读看到最新

```text
事务 A begin
A: select balance where id = 1 -> 100

事务 B begin
B: update account set balance = 80 where id = 1
B: commit

A: select balance where id = 1 for update -> 80
```

原因：`for update` 是当前读。

## 7. 面试口述版

### 7.1 30 秒回答

不可重复读是同一事务内两次读同一行，行内容变了；幻读是同一事务内两次按条件查，结果集合变了。快照读是普通 select 通过 MVCC 读一致性视图，通常不加锁，也不一定读最新。当前读是读最新已提交版本并加锁，比如 update、delete、select for update。InnoDB 在 RR 下，普通快照读通过复用 ReadView 避免结果变化，范围当前读通过 next-key lock 阻止其他事务插入新记录，所以大部分场景能避免幻读。

### 7.2 90 秒回答

不可重复读和幻读的区别在于关注对象不同。不可重复读关注同一行数据内容变化，比如第一次读余额 100，别人提交修改后第二次读成 80。幻读关注结果集合变化，比如第一次查满足条件的订单有 10 条，别人插入一条后第二次查有 11 条。

InnoDB 里还要区分快照读和当前读。普通 select 是快照读，通过 ReadView 和 undo 版本链读取可见版本，所以在 RR 下同一事务内普通查询结果通常稳定。`select ... for update`、`update`、`delete` 是当前读，要读最新数据并加锁。RR 下范围当前读会用 next-key lock 锁住记录和间隙，阻止其他事务插入满足条件的新记录。因此说 RR 解决幻读时要讲清楚是 InnoDB 的实现，以及快照读和当前读分别靠什么机制。

## 8. 高频追问

### 8.1 RR 下普通 select 会加锁吗？

通常不会。普通 select 是快照读，通过 MVCC 读取历史可见版本。  
如果显式加 `for update` 或 `for share`，就变成当前读并加锁。

### 8.2 为什么当前读要读最新数据？

因为当前读通常要修改或锁定数据。  
如果基于旧快照修改，就可能覆盖其他事务已经提交的结果，造成丢失更新或业务错误。

### 8.3 RR 是否完全等价于 Serializable？

不等价。  
RR 通过 MVCC 和 next-key lock 在很多场景避免幻读，但它仍然允许较高并发。  
Serializable 更严格，普通读也可能加锁，并发能力更差。

### 8.4 delete 导致结果集合变少算幻读吗？

标准定义里幻读常强调插入导致出现新行。  
面试口述时可以说：广义上结果集合变化也包括删除，但核心要表达的是“范围查询的集合变化”，不要和同一行内容变化混淆。

---

## 9. 容易踩坑

1. 不要把不可重复读和幻读都说成“数据变了”，要区分行内容和结果集合。
2. 不要说 RR 下所有读都一定读旧快照，当前读读最新。
3. 不要说普通 select 会加 next-key lock，普通 select 通常是快照读。
4. 不要说 next-key lock 是 MVCC 的一部分，它属于锁机制。
5. 不要在回答 RR 避免幻读时忽略 InnoDB 这个前提。
