## 1. new 和 make 的区别是什么？（基本难度）

### 1.1 最简练版

**`new` 用来分配某个类型的零值并返回指针，`make` 用来初始化 slice、map、channel 这三类运行时结构。**  
`new(T)` 返回 `*T`，得到的是一个指向零值的指针。  
`make(T, ...)` 返回的是 `T` 本身，不是指针。  
如果 map 或 channel 只用 `new` 得到零值，内部结构没有初始化，通常不能直接当正常容器使用。  
slice 比较特殊：`new([]T)` 得到的是 `*[]T`，它指向一个 nil slice；nil slice 可以 `append`，但长度和容量都是 0，不能直接按下标读写。

### 1.2 详细解释版

| 对比点 | `new` | `make` |
|---|---|---|
| 适用类型 | 任意类型 | slice、map、channel |
| 返回值 | `*T` | `T` |
| 初始化能力 | 只得到零值 | 初始化运行时结构 |
| 常见用途 | 创建结构体、基础类型指针 | 创建可用的 slice、map、channel |

几个关键细节：

- `new([]int)` 返回 `*[]int`，指向一个 nil slice，`len=0`、`cap=0`、没有底层数组。
- nil slice 可以 `append`，但要把 `append` 的结果重新赋回去，比如 `*p = append(*p, 1)`。
- nil slice 不能直接按下标写入，比如 `(*p)[0] = 1` 会 panic，因为长度是 0。
- `make([]int, 0, 10)` 返回一个可 append 的 slice，并提前准备了容量。
- `make([]int, 3, 10)` 返回一个长度为 3 的 slice，可以直接写 `s[0] = 1`。
- `new(map[string]int)` 返回 `*map[string]int`，里面的 map 仍是 nil，直接写入会 panic。
- `make(map[string]int)` 返回可写入的 map。
- `new(chan int)` 返回 `*chan int`，里面的 channel 仍是 nil，对它发送或接收会永久阻塞。
- `make(chan int)` 返回可正常通信的 channel。

nil slice、nil map、nil channel 的区别：

| 类型 | `new` 后的零值 | 可以做什么 | 不能直接做什么 |
|---|---|---|---|
| `[]T` | nil slice | `len`、`cap`、`range`、`append` | 按下标读写 |
| `map[K]V` | nil map | 读取、`len`、`range` | 写入，写入会 panic |
| `chan T` | nil channel | 和 `nil` 比较 | 发送、接收会永久阻塞，`close` 会 panic |

### 1.3 图示和例子

```go
p := new(int)
*p = 10

s0 := new([]int)
*s0 = append(*s0, 1) // 可以 append，但要解引用并接住返回值

s := make([]int, 0, 4)
s = append(s, 1)

s2 := make([]int, 3, 4)
s2[0] = 1 // len=3，可以按下标写

m := make(map[string]int)
m["a"] = 1

ch := make(chan int, 1)
ch <- 1
```

一句话记忆：

```text
new 负责分配零值并返回指针；make 负责返回可直接使用的 slice/map/channel。
slice 的零值 nil slice 可以 append；map/channel 的零值不能当正常容器使用。
```

### 1.4 为什么 new 返回指针，make 不返回指针？

核心原因是二者的语义不同：

```text
new(T)  -> *T  ：分配一个 T 类型的零值变量，返回这个变量的地址。
make(T) -> T   ：初始化 slice/map/channel，返回已经可直接使用的容器值。
```

`new` 的语义是“我要一块新的变量存储空间”。  
如果 `new(int)` 返回的是 `int` 本身，那它和直接声明零值变量区别不大：

```go
var x int
```

所以 `new(int)` 返回 `*int`，调用方拿到地址后可以通过指针修改这块新分配的变量：

```go
p := new(int)
*p = 10
```

结构体也是类似：

```go
u := new(User)
u.Name = "Tom"
```

`make` 不返回指针，是因为 slice、map、channel 这三类值本身就是运行时描述符或句柄。  
它们的值里已经包含了指向底层数据结构的信息，拿到返回值后就可以直接使用。

slice 可以粗略理解成：

```text
slice = {
    ptr, // 指向底层数组
    len,
    cap,
}
```

所以：

```go
s := make([]int, 0, 10)
s = append(s, 1)
```

如果 `make` 返回 `*[]int`，反而会让常见用法变得别扭：

```go
// 假设 make 返回 *[]int，就得这样写，反而多一层没必要的间接访问
*s = append(*s, 1)
```

map 和 channel 也是类似：

```go
m := make(map[string]int)
m["a"] = 1

ch := make(chan int, 1)
ch <- 1
```

`m` 和 `ch` 变量本身已经是指向运行时内部结构的句柄，不需要再额外拿 `*map` 或 `*chan` 才能修改内部状态。

注意：不是说 `make` 不分配内存。  
`make([]T, 0, n)` 可能分配底层数组，`make(map[K]V)` 会初始化哈希表结构，`make(chan T, n)` 会初始化 channel 运行时结构。  
区别在于：`make` 返回的是初始化好的容器值，而不是这个变量本身的指针。

### 1.5 引用和地址有什么区别？

在 Go 里可以这样区分：

```text
地址：某个变量在内存里的位置，比如 &x。
指针：保存地址的变量，比如 *int。
引用语义：拷贝一个值以后，多个变量仍然共享同一份底层数据。
```

指针最直接：

```go
x := 10
p := &x

*p = 20
fmt.Println(x) // 20
```

这里 `&x` 是 `x` 的地址，`p` 是保存这个地址的指针。通过 `*p` 可以改到原来的 `x`。

而 Go 里常说 slice、map、channel 是“引用语义”，不是说 Go 有 C++ 那种引用变量，而是说这些值内部间接指向底层运行时数据。  
拷贝这些值时，表层描述符被拷贝了，但底层数据可能仍然共享。

普通值拷贝后互不影响：

```go
x := 1
y := x
y = 2

fmt.Println(x) // 1
```

slice 拷贝后，slice 头是新的，但底层数组仍然可能是同一个：

```go
s1 := []int{1}
s2 := s1
s2[0] = 2

fmt.Println(s1) // [2]
```

map 拷贝后，两个 map 变量指向同一个运行时哈希表：

```go
m1 := map[string]int{"a": 1}
m2 := m1
m2["a"] = 2

fmt.Println(m1["a"]) // 2
```

channel 拷贝后，两个 channel 变量也指向同一个运行时 channel：

```go
ch1 := make(chan int, 1)
ch2 := ch1

ch2 <- 1
fmt.Println(<-ch1) // 1
```

所以面试里可以这样回答：

```text
地址是内存位置，指针是保存地址的变量。
引用语义不是一种单独语法，而是某些类型拷贝后仍共享底层数据的表现。
Go 里 slice、map、channel 都有引用语义，但它们本身仍然是值，赋值和传参都会拷贝这个值。
```

## 2. slice 的底层结构是什么？（基本难度）

### 2.1 最简练版

**slice 底层是一个描述符，包含数组指针、长度和容量。**  
长度 `len` 表示当前可访问元素个数，容量 `cap` 表示从起始位置到底层数组末尾还能容纳多少元素。  
多个 slice 可以共享同一个底层数组，因此修改一个 slice 可能影响另一个 slice。  
当 `append` 超过容量时，会分配新数组并复制旧数据，这时共享关系可能断开。

### 2.2 详细解释版

slice 不是数组本身，而是对数组某一段的视图。可以抽象成：

| 字段 | 含义 |
|---|---|
| pointer | 指向底层数组的起始位置 |
| len | 当前 slice 的长度 |
| cap | 当前 slice 可用容量 |

常见坑：

- **共享底层数组。** 子切片修改元素，会影响原切片。
- **append 可能原地追加，也可能扩容换数组。** 是否影响原切片取决于容量是否足够。
- **大数组被小切片引用会造成内存无法释放。** 需要 `copy` 出真正需要的小片段。

### 2.3 图示和例子

```text
底层数组: [10][20][30][40][50]
            ^
            |
slice: ptr --+
       len = 3   -> [10][20][30]
       cap = 5   -> 从 ptr 到数组末尾
```

共享底层数组：

```go
a := []int{1, 2, 3, 4}
b := a[:2]
b[0] = 100

// a 变成 [100 2 3 4]
```

## 3. slice 扩容规则大致是什么？（中等难度）

### 3.1 最简练版

**slice 在 `append` 后长度超过容量时会扩容，扩容会分配新底层数组并复制旧元素。**  
小容量时通常增长较快，接近翻倍；容量变大后增长会放缓，避免内存浪费。  
实际新容量还会受元素大小、内存对齐和运行时分配规格影响，所以不要死记精确数字。  
工程上更重要的是能预估大小时提前 `make` 指定容量，减少扩容和拷贝。

### 3.2 详细解释版

扩容逻辑关注三件事：

1. 新容量至少要能放下目标长度。
2. 小 slice 通常采用较激进的增长策略，减少频繁扩容。
3. 大 slice 增长更保守，避免一次性浪费太多内存。

扩容的成本：

| 成本 | 说明 |
|---|---|
| 分配新数组 | 需要向运行时申请新内存 |
| 复制旧元素 | 元素越多，复制成本越高 |
| 旧数组等待回收 | 如果没有引用，后续由 GC 回收 |
| 指针变化 | 原来共享底层数组的关系可能断开 |

优化建议：

- 知道大致数量时，用 `make([]T, 0, n)`。
- 大对象切片尽量避免频繁 append。
- 需要截取大切片中的小片段长期保存时，用 `copy` 断开对大数组的引用。

### 3.3 图示和例子

```go
s := make([]int, 0, 3)
s = append(s, 1, 2, 3)

// 再 append 一个元素，超过 cap=3，需要扩容
s = append(s, 4)
```

过程：

```text
旧数组: [1][2][3]
              append 4
新数组: [1][2][3][4][ ][ ]...
```

## 4. map 为什么并发不安全？（基本难度）

### 4.1 最简练版

**Go 内置 map 不是并发安全的，因为读写和写写并发可能破坏内部哈希表状态。**  
map 写入时可能触发扩容、桶迁移和元素重排，如果另一个 `goroutine` 同时读写，就可能看到不一致状态。  
运行时经常能检测到并报 `fatal error: concurrent map read and map write`，但不能依赖它兜底。  
并发场景要用 `sync.Mutex`、`sync.RWMutex`、`sync.Map` 或分片 map。

### 4.2 详细解释版

map 底层大致由哈希桶组成，写入时可能发生：

- 找桶、找槽位；
- 插入或覆盖元素；
- 桶溢出；
- 触发扩容；
- 增量迁移旧桶到新桶。

这些过程都要求内部状态一致。  
如果同时有另一个 `goroutine` 读或写，可能读到正在迁移的桶、半更新状态或错误指针。

选择方案：

| 方案 | 适用场景 |
|---|---|
| `sync.Mutex` + map | 读写都不少，逻辑简单 |
| `sync.RWMutex` + map | 读多写少，读临界区较长 |
| `sync.Map` | key 稳定、读多写少、缓存类场景 |
| 分片 map | 高并发读写，需要降低单锁竞争 |
| copy-on-write | 读极多、写极少、可接受整体替换 |

### 4.3 图示和例子

错误写法：

```go
m := map[string]int{}

go func() {
    m["a"] = 1
}()

go func() {
    _ = m["a"]
}()
```

更稳的写法：

```go
var mu sync.RWMutex
m := map[string]int{}

mu.Lock()
m["a"] = 1
mu.Unlock()

mu.RLock()
v := m["a"]
mu.RUnlock()
_ = v
```

## 5. Go 的逃逸分析在解决什么问题？（中等难度）

### 5.1 最简练版

**逃逸分析解决的是对象应该分配在栈上还是堆上的问题。**  
如果编译器能证明对象只在当前函数内使用，就倾向于放栈上；如果对象生命周期可能超过当前函数，就必须放到堆上。  
栈分配便宜，函数返回后自动回收；堆分配更灵活，但会增加 GC 压力。  
逃逸分析的目标不是消灭堆分配，而是在保证正确性的前提下尽量减少不必要的堆分配。

### 5.2 详细解释版

常见逃逸原因：

| 场景 | 原因 |
|---|---|
| 返回局部变量指针 | 函数返回后对象仍被外部使用 |
| 被闭包捕获 | 对象生命周期可能跟闭包一样长 |
| 传给接口 | 编译器可能难以确定具体使用方式 |
| 启动 goroutine 使用局部变量 | 当前函数返回后 goroutine 仍可能访问 |
| 对象过大 | 放栈上可能导致栈增长成本过高 |

逃逸影响：

- 堆分配次数增加；
- GC 扫描和回收压力变大；
- 高频路径延迟可能上升；
- 但代码可读性和正确性通常比微小分配更重要。

### 5.3 图示和例子

会逃逸的例子：

```go
func build() *int {
    x := 10
    return &x
}
```

`x` 的地址被返回，生命周期超过 `build`，所以不能放在 `build` 的栈帧里。

```text
build 栈帧
  x = 10
  |
  return &x
  |
  v
外部还要使用 x，所以 x 需要放到堆上
```

再看几个常见逃逸原因。

1. **返回局部变量地址**

```go
type User struct {
    Name string
}

func newUser() *User {
    u := User{Name: "Tom"}
    return &u
}
```

原因：`u` 是函数局部变量，但函数返回后调用方还要通过 `*User` 使用它。  
如果 `u` 放在 `newUser` 的栈上，函数返回后栈帧销毁，外部指针就悬空了。所以 `u` 需要逃逸到堆上。

```text
newUser 栈帧结束
  |
  v
调用方仍持有 &u
  |
  v
u 不能只活在 newUser 的栈里
```

2. **闭包捕获局部变量**

```go
func counter() func() int {
    n := 0

    return func() int {
        n++
        return n
    }
}
```

原因：返回的匿名函数捕获了 `n`。  
`counter` 返回后，闭包还可能继续调用，并继续读写 `n`。因此 `n` 的生命周期跟闭包绑定，可能超过 `counter` 本身。

```go
next := counter()
fmt.Println(next()) // 1
fmt.Println(next()) // 2
```

一句话记忆：**闭包带走了变量，变量就不能只留在原函数栈上。**

3. **启动 goroutine 使用局部变量**

```go
func start() {
    msg := "hello"

    go func() {
        fmt.Println(msg)
    }()
}
```

原因：`start` 可能很快返回，但新启动的 goroutine 可能之后才执行。  
匿名函数捕获了 `msg`，所以 `msg` 不能只依赖 `start` 的栈帧存活。

```text
start 返回
  |
  v
goroutine 之后才读取 msg
  |
  v
msg 的生命周期可能超过 start
```

注意：不是所有 goroutine 里的变量都会逃逸，关键看它是否引用了外层局部变量，以及编译器能否证明生命周期安全。

4. **传给 interface 导致装箱**

```go
func printAny() {
    x := 10
    fmt.Println(x)
}
```

原因：`fmt.Println` 的参数类型是 `...any`，`x` 会被装箱成 interface。  
接口值需要保存动态类型和动态值，编译器有时难以完全证明这个值只在当前栈帧内使用，因此可能发生逃逸。

更明显的例子：

```go
func toAny() any {
    x := 10
    return x
}
```

这里返回的是 `any`，调用方在函数外还要使用这个接口值，所以 `x` 装箱后的值可能需要放到堆上。

一句话记忆：**传给接口不一定必然逃逸，但接口装箱是逃逸分析里很常见的触发点。**

5. **指针放进切片或 map 后被外部持有**

```go
func buildUsers() []*User {
    u := User{Name: "Tom"}
    users := []*User{&u}
    return users
}
```

原因：`users` 被返回到函数外，里面保存了 `&u`。  
只要外部还持有 `users`，就可能继续通过 `users[0]` 访问 `u`，所以 `u` 需要逃逸。

map 也是类似：

```go
func buildUserMap() map[string]*User {
    u := User{Name: "Tom"}
    return map[string]*User{
        "tom": &u,
    }
}
```

一句话记忆：**局部变量的地址只要被放进会返回到外部的容器里，局部变量通常就要逃逸。**

6. **大对象可能被放到堆上**

```go
func big() {
    buf := [1 << 20]byte{}
    _ = buf
}
```

原因：栈空间会按需增长，但把很大的对象放在栈上可能带来较高的栈增长和拷贝成本。  
编译器可能选择把大对象分配到堆上。

注意：这个判断和 Go 版本、对象大小、使用方式有关。面试时不要死背某个具体阈值，重点说清楚：**过大的局部对象可能因为栈成本而逃逸到堆上。**

7. **看起来用了 `new`，不代表一定逃逸**

```go
func local() int {
    p := new(int)
    *p = 10
    return *p
}
```

这段代码里 `new(int)` 不一定分配到堆上。  
如果编译器能证明 `p` 只在当前函数内使用，底层的 `int` 仍然可能放在栈上，甚至被优化掉。

所以不要把规则记成：

```text
new 一定上堆，局部变量一定上栈
```

更准确的说法是：

```text
是否上堆由逃逸分析决定，核心看对象是否可能在当前函数之外继续被访问。
```

## 6. 如何判断对象发生了逃逸？（基本难度）

### 6.1 最简练版

**最直接的方法是看编译器逃逸分析输出。**  
可以用 `go build -gcflags='-m -m'` 查看哪些变量 `escapes to heap` 或 `moved to heap`。  
如果要看运行时影响，可以结合 benchmark 的 `-benchmem`、pprof 的 heap profile 和 alloc profile。  
判断时不要只看有没有逃逸，还要看它是不是在高频路径、是否造成明显分配和 GC 压力。

### 6.2 详细解释版

常用命令：

```bash
go build -gcflags='-m -m' ./...
```

基准测试：

```bash
go test -bench=. -benchmem ./...
```

线上或压测环境：

```bash
go tool pprof http://127.0.0.1:6060/debug/pprof/heap
go tool pprof http://127.0.0.1:6060/debug/pprof/allocs
```

看输出时重点关注：

| 信息 | 含义 |
|---|---|
| `escapes to heap` | 对象逃逸到堆 |
| `moved to heap` | 变量被移动到堆上 |
| `allocs/op` | 每次操作分配次数 |
| `B/op` | 每次操作分配字节数 |

### 6.3 图示和例子

```go
func toIface() any {
    x := 10
    return x
}
```

可能出现的分析方向：

```text
变量 x
  |
  v
装箱到 interface
  |
  v
编译器判断是否需要堆分配
```

优化时优先看热点：

```text
逃逸分析输出
  |
  v
benchmark 确认分配
  |
  v
pprof 确认线上热点
  |
  v
再决定是否改代码
```

## 7. Go 的 GC 基本流程是什么？（高难度）

### 7.1 最简练版

**Go GC 的核心是并发标记清扫。**  
它会先短暂停顿开启标记和写屏障，然后并发扫描可达对象，再短暂停顿完成标记，最后清扫不可达对象。  
业务 `goroutine` 在大部分标记阶段可以继续运行，但需要配合写屏障和辅助标记。  
GC 的主要目标是在吞吐和延迟之间折中，避免长时间 STW（Stop The World，暂停用户代码执行）。

### 7.2 详细解释版

典型流程：

1. **触发 GC。** 堆增长达到目标、手动调用 `runtime.GC()` 或运行时认为需要回收。
2. **标记开始。** 短暂停顿，开启写屏障，准备根对象扫描。
3. **并发标记。** 从根对象出发，扫描可达对象，业务代码继续运行。
4. **辅助标记。** 分配过快的 `goroutine` 可能参与一部分标记工作，避免 GC 追不上分配速度。
5. **标记终止。** 再次短暂停顿，完成剩余标记工作并关闭写屏障。
6. **清扫。** 回收未标记对象占用的内存，后续分配时也可能顺带清扫。

常见观察方式：

```bash
GODEBUG=gctrace=1 ./server
```

也可以通过 `runtime.MemStats`、`pprof heap` 和服务指标观察堆大小、GC 次数、暂停时间和分配速度。

### 7.3 图示和例子

```text
触发 GC
  |
  v
短暂 STW：开启写屏障
  |
  v
并发标记：业务继续运行
  |
  v
短暂 STW：标记终止
  |
  v
并发清扫：回收不可达对象
```

## 8. 三色标记解决了什么问题？（高难度）

### 8.1 最简练版

**三色标记用白、灰、黑三种状态组织对象扫描，解决“如何判断对象是否可达”的问题。**  
白色表示尚未确认可达，灰色表示已发现但子对象还没扫描完，黑色表示自己和子对象都扫描完。  
GC 从根对象开始，把可达对象逐步从白变灰再变黑，最后仍然白色的对象就是不可达对象。  
并发标记时，业务代码还在修改指针，所以需要写屏障维持正确性，避免可达对象被漏标。

### 8.2 详细解释版

三种颜色：

| 颜色 | 含义 |
|---|---|
| 白色 | 还没被发现，最终如果仍是白色就会被回收 |
| 灰色 | 已发现，但它引用的对象还没扫描完 |
| 黑色 | 已扫描完成，它引用的对象也已处理 |

核心过程：

1. 所有对象初始看作白色。
2. 根对象入队，标为灰色。
3. 取出灰色对象，扫描它引用的对象，把被引用白色对象标为灰色。
4. 当前对象扫描完后变黑。
5. 没有灰色对象时，标记结束。
6. 剩余白色对象不可达，可以清扫。

并发难点：  
如果黑色对象在业务代码运行时新增了对白色对象的引用，而 GC 不知道，就可能把这个白色对象误回收。  
写屏障就是为了记录这类指针变化，维持标记正确性。

可以把 GC 想成一个“边走边查”的扫描者，而业务代码还在旁边不断修改对象之间的引用关系。  
三色标记里最重要的不变式之一是：

```text
黑色对象不能直接指向白色对象
```

原因是：黑色表示这个对象已经扫描完成，GC 后面通常不会再回头扫描它。  
如果它后来又偷偷指向一个白色对象，GC 可能就永远发现不了这个白色对象。

举个例子：

```text
一开始：

黑色 A        白色 B
A 没有指向 B
```

此时 GC 已经扫描过 `A`，认为 `A` 没有新的子对象需要处理。  
然后业务代码执行了一次指针赋值：

```go
A.child = B
```

对象关系变成：

```text
黑色 A  --->  白色 B
```

危险点在于：`A` 已经是黑色，GC 不会再回头扫描 `A`；`B` 还是白色，最后可能被当成垃圾回收。  
但从业务代码角度看，`A.child` 已经指向 `B`，`B` 明明还是活对象。

写屏障就是在指针写入时插入一段小逻辑，类似：

```go
// 伪代码，不是 Go 源码
if 正在 GC && 发生指针写入 {
    通知 GC：这里出现了新的引用，别漏标
}
A.child = B
```

这样 GC 就可以把 `B` 标记成灰色，后续继续扫描：

```text
黑色 A  --->  灰色 B
```

一句话记忆：

```text
业务代码改指针，GC 可能看漏；写屏障负责提醒 GC。
```

更短一点：

```text
写屏障 = 指针赋值时的“打报告”机制。
```

### 8.3 图示和例子

```text
初始：
root -> A -> B
        |
        v
        C

标记中：
白色：未发现
灰色：待扫描
黑色：已扫描
```

扫描过程：

```text
root 发现 A：A 变灰
A 扫描 B/C：B、C 变灰，A 变黑
B 扫描完成：B 变黑
C 扫描完成：C 变黑
```

最后还保持白色的对象，就是不可达对象。

## 9. STW 在 Go GC 里扮演什么角色？（高难度）

### 9.1 最简练版

**STW 是 GC 在关键阶段为了获得一致状态而短暂停顿用户代码。**  
Go 的 GC 不是完全没有 STW，而是把长时间停顿拆成很短的阶段，主要出现在标记开始和标记终止。  
标记开始需要开启写屏障并建立标记状态，标记终止需要完成剩余标记并切换阶段。  
如果 STW 或 GC 频率异常，通常要看分配速度、堆大小、对象存活率和是否有大量指针对象。

### 9.2 详细解释版

STW 的作用：

| 阶段 | 作用 |
|---|---|
| 标记开始 | 切换到标记阶段，开启写屏障，准备扫描 |
| 标记终止 | 完成剩余标记任务，关闭写屏障，进入清扫 |

为什么不能完全没有 STW？  
因为 GC 和业务代码并发运行时，堆对象引用关系不断变化。某些阶段必须短暂停下业务代码，让运行时切换全局状态，保证标记和屏障规则一致。

导致 GC 影响变大的常见原因：

- 瞬时分配量太大；
- 大量短生命周期对象；
- 大量指针对象，扫描成本高；
- 堆目标设置过小，GC 太频繁；
- 容器内存限制导致堆增长空间不足；
- 业务代码频繁创建大切片、大 map 或大量临时对象。

### 9.3 图示和例子

```text
时间线：

业务运行 | STW | 并发标记 + 业务运行 | STW | 并发清扫 + 业务运行
          ^                         ^
          |                         |
       标记开始                  标记终止
```

小白类比：老师检查教室里谁还在。

```text
GC       = 老师
对象     = 学生
引用关系 = 谁牵着谁
白色     = 还没确认在教室
灰色     = 确认在教室，但他牵着的人还没查
黑色     = 他和他牵着的人都查完了
写屏障   = 班长记录谁新牵了谁
清扫     = 清理没人坐的空座位
```

完整过程：

```text
第 1 次 STW：
老师喊“大家先别动一下”，开始检查，并安排班长记录变化。

并发标记：
老师继续检查学生，学生也继续活动。
如果有人新牵手，班长记录下来，防止老师漏查。

第 2 次 STW，也就是标记终止：
老师喊“大家再停一下”，把班长记录的小纸条全部处理完。
确认没有漏掉的人，再开始清理没人坐的座位。
```

为什么标记终止很重要？

```text
并发标记阶段，学生还在动，班长还在记小纸条。
老师快检查完时，必须停一下，把小纸条核对完。
否则可能把还在教室的人误当成不在的人清走。
```

对应到 GC：

```text
标记终止 = GC 最后暂停一下，处理剩余灰色对象和写屏障记录，
确认没有漏标后，才进入清扫阶段。
```

排查时可以看：

```bash
GODEBUG=gctrace=1 ./server
go tool pprof http://127.0.0.1:6060/debug/pprof/heap
```

关注：

```text
分配速度 -> GC 频率
存活对象 -> 堆回落情况
暂停时间 -> 尾延迟影响
```
