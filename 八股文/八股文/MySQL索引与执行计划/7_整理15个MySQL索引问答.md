## 1. MySQL 为什么常用 B+ 树索引？（中等难度）

### 1.1 最简练版

**B+ 树适合数据库索引，因为树高低、磁盘 I/O（Input/Output，输入输出）少，并且叶子节点有序。**  
它不仅支持等值查询，还支持范围查询、排序和前缀匹配。  
非叶子节点主要存键值和指针，一个页能容纳更多分支，查询时访问的页更少。  
相比哈希索引，B+ 树对数据库常见查询更通用。

### 1.2 详细解释版

数据库按页读写，InnoDB 页大小常见是 `16KB`。B+ 树的高扇出能让树高保持较低，点查一般只需要访问少量页。  
叶子节点之间按顺序连接，因此范围扫描、`order by` 和分页都可以顺着叶子节点走。  
哈希结构等值查询很快，但不适合范围和排序；二叉树在内存里好用，但树高和磁盘页模型不匹配。

| 结构 | 优点 | 短板 |
|---|---|---|
| 哈希 | 等值查询快 | 不支持范围和排序 |
| 红黑树 | 内存查找方便 | 树高较高，不适合磁盘页 |
| B+ 树 | 点查、范围、排序都均衡 | 维护成本更高 |

### 1.3 例子

```sql
select *
from order_tab
where id between 1000 and 2000
order by id;
```

B+ 树可以先定位到 `id = 1000` 附近的叶子节点，再顺着叶子节点扫描到 `2000`。这比每一行都全表检查要高效得多。

## 2. 聚簇索引和二级索引有什么区别？（中等难度）

### 2.1 最简练版

**InnoDB 的聚簇索引通常就是主键索引，叶子节点存整行数据；二级索引叶子节点存索引列和主键值。**  
通过二级索引查不到所需列时，需要再拿主键回到聚簇索引取整行，这叫回表。  
一张表通常只有一个聚簇索引，但可以有多个二级索引。  
主键越长，二级索引也会越大，因为二级索引叶子节点要保存主键。

### 2.2 详细解释版

InnoDB 会优先使用主键作为聚簇索引；如果没有主键，会选择唯一非空索引；再没有才生成隐藏 row id。  
聚簇索引决定数据行的物理组织方式，二级索引用来加速非主键字段查询。

| 维度 | 聚簇索引 | 二级索引 |
|---|---|---|
| 数量 | 通常一个 | 可以多个 |
| 叶子节点 | 整行数据 | 索引列 + 主键值 |
| 查询路径 | 主键直接定位整行 | 可能需要回表 |
| 受主键影响 | 主键决定数据组织 | 每个二级索引都保存主键 |

### 2.3 例子

```sql
create table user_tab (
  id bigint primary key,
  phone varchar(32),
  name varchar(64),
  key idx_phone(phone)
);
```

查询 `phone` 时，`idx_phone` 的叶子节点保存的是 `phone + id`。如果查询 `name`：

```sql
select name from user_tab where phone = '13800000000';
```

执行路径通常是：先走 `idx_phone` 找到 `id`，再走主键索引取 `name`。

## 3. 回表是什么？（基本难度）

### 3.1 最简练版

**回表是指先通过二级索引找到主键，再回到聚簇索引读取完整行。**  
回表会多走一次索引树，返回行数多时成本会明显放大。  
它不是绝对不能接受，关键看回表次数、查询频率和缓存命中情况。  
减少回表的常见方式是覆盖索引、减少返回字段和先查主键再取详情。

### 3.2 详细解释版

二级索引不保存整行，只保存索引列和主键值。  
如果查询需要的列不在二级索引里，就需要拿主键去聚簇索引查整行，这个过程就是回表。  
回表成本高的场景通常是：返回行数多、查询频率高、二级索引选择性差、聚簇索引页不在缓存中。

### 3.3 例子

```sql
create index idx_user_status
on order_tab(user_id, status);
```

这条 SQL 需要回表，因为 `address` 不在索引里：

```sql
select id, user_id, status, address
from order_tab
where user_id = ?
  and status = ?;
```

如果改成只查索引内字段：

```sql
select id, user_id, status
from order_tab
where user_id = ?
  and status = ?;
```

就更可能直接通过索引返回。

## 4. 覆盖索引是什么？（基本难度）

### 4.1 最简练版

**覆盖索引是指查询需要的列都在同一个索引里，执行时不用回表。**  
它常用于高频列表页、统计接口和深分页子查询。  
执行计划里 `Extra` 出现 `Using index`，通常说明可能用了覆盖索引。  
但覆盖索引不是越宽越好，索引列越多，写入和存储成本越高。

### 4.2 详细解释版

覆盖索引的收益是少读聚簇索引页，减少随机 I/O，提高高频查询稳定性。  
代价是索引变宽，单页能放的索引项变少，写入、更新、删除都要维护更多内容。  
所以覆盖索引适合高频、返回字段少、字段较小的查询，不适合把大字段硬塞进去。

### 4.3 例子

```sql
create index idx_tenant_status_time_id
on order_tab(tenant_id, status, created_at, id);
```

下面查询可以被覆盖：

```sql
select id, tenant_id, status, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc
limit 20;
```

如果查询多了 `address`，而 `address` 不在索引里，就需要回表。

## 5. 联合索引为什么遵循最左前缀？（中等难度）

### 5.1 最简练版

**联合索引按照定义列顺序组成一个有序 key，只有从最左列开始连续匹配，才能形成有效定位区间。**  
比如 `(a,b,c)` 可以支持 `a`、`a,b`、`a,b,c`，但通常不适合直接查 `b`。  
本质是 B+ 树先按 `a` 排，再在相同 `a` 内按 `b` 排，最后才按 `c` 排。  
跳过左侧列后，后面的列只在局部有序，不能作为全局入口。

### 5.2 详细解释版

最左前缀不是死记硬背的规则，而是多列排序方式决定的访问路径。  
索引 `(tenant_id, status, created_at)` 的顺序是：先按 `tenant_id`，再按 `status`，再按 `created_at`。  
如果没有 `tenant_id` 条件，只查 `status`，数据库很难直接定位到所有 `status = ?` 的连续区间。

### 5.3 例子

```sql
create index idx_tenant_status_time
on order_tab(tenant_id, status, created_at);
```

| 查询 | 利用情况 |
|---|---|
| `where tenant_id = ?` | 可以利用最左列 |
| `where tenant_id = ? and status = ?` | 可以连续利用前两列 |
| `where tenant_id = ? and status = ? order by created_at` | 可能同时利用排序 |
| `where status = ?` | 通常不适合直接使用这个联合索引 |

## 6. 范围查询后面的列还能用索引吗？（中等难度）

### 6.1 最简练版

**范围查询后面的列通常很难继续用于精确定位，但不代表完全没用。**  
例如 `(a,b,c)` 中 `a = ? and b > ? and c = ?`，`b` 已经把访问变成范围，`c` 很难继续收窄索引边界。  
但 `c` 仍可能参与 ICP（Index Condition Pushdown，索引条件下推）过滤，或者用于覆盖索引。  
准确说法是：范围后的列定位能力下降，但可能还能过滤或覆盖。

### 6.2 详细解释版

范围条件包括 `>`、`<`、`between`、部分 `like 'abc%'` 和大量离散 `in`。  
一旦前面的列形成范围，后续列就不再是一个连续有序的小区间，优化器很难用它继续精确定位。  
不过后续列如果在索引里，仍可减少回表或在索引层做条件判断。

### 6.3 例子

```sql
create index idx_a_b_c on t(a, b, c);
```

| SQL | 说明 |
|---|---|
| `where a = 1 and b = 2 and c = 3` | 三列都能较好定位 |
| `where a = 1 and b > 2 and c = 3` | `a,b` 定范围，`c` 难继续做边界 |
| `where a = 1 and b between 2 and 8 and c = 3` | `c` 可能用于过滤或覆盖 |

## 7. 为什么不建议 select 星号？（基本难度）

### 7.1 最简练版

**不建议 `select *`，因为它会读取不必要字段，增加 I/O、网络传输和内存开销。**  
它还容易破坏覆盖索引，让原本只查二级索引的查询变成回表查询。  
字段变更时，`select *` 也可能让接口返回内容和序列化成本不可控。  
列表页尤其应该只查必要字段。

### 7.2 详细解释版

`select *` 的风险：

| 风险 | 说明 |
|---|---|
| 读放大 | 大字段、冷字段也被读出来 |
| 回表增加 | 查询列超过索引覆盖范围 |
| 网络开销 | 返回数据包变大 |
| 代码耦合 | 表结构变更影响接口行为 |

### 7.3 例子

```sql
select *
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc
limit 20;
```

如果订单表里有 `address`、`remark`、`extra_json` 等大字段，列表页并不需要这些字段，却会被一起读出。  
改成：

```sql
select id, order_no, status, amount, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc
limit 20;
```

更容易配合覆盖索引，查询也更稳定。

## 8. 索引失效常见原因有哪些？（中等难度）

### 8.1 最简练版

**索引失效常见原因包括函数包列、隐式类型转换、左模糊匹配、违反最左前缀、低选择性条件和优化器成本判断。**  
不是所有“没走索引”都叫失效，有时是优化器认为全表扫更便宜。  
排查时要看 `EXPLAIN` 的 `key`、`rows`、`Extra`，再结合数据分布。  
修复方式通常是 SQL 改写、调整联合索引顺序、更新统计信息或改变查询模式。

### 8.2 详细解释版

| 原因 | 例子 | 处理 |
|---|---|---|
| 函数包列 | `date(created_at) = ...` | 改成时间范围 |
| 隐式转换 | 字符串列用数字查 | 类型保持一致 |
| 左模糊 | `like '%abc'` | 改搜索系统或反向索引 |
| 跳过最左列 | 索引 `(a,b)` 只查 `b` | 调整索引或查询 |
| 低选择性 | `status = 1` 命中大部分行 | 加更高选择性前缀 |
| 统计信息不准 | 预估行数偏差大 | `analyze table` |

### 8.3 例子

坏写法：

```sql
select id
from order_tab
where date(created_at) = '2026-05-06';
```

改写：

```sql
select id
from order_tab
where created_at >= '2026-05-06 00:00:00'
  and created_at <  '2026-05-07 00:00:00';
```

这样可以让 `created_at` 形成明确范围，更容易使用索引。

## 9. 区分度高的列一定放联合索引最前面吗？（中等难度）

### 9.1 最简练版

**区分度高很重要，但不一定永远放最前面。**  
联合索引顺序要综合查询频率、等值条件、排序、范围条件和分页方式。  
如果高频查询都先按租户过滤，即使 `tenant_id` 区分度不是最高，也可能适合放前面。  
核心是让索引匹配最重要的访问路径，而不是机械按区分度排序。

### 9.2 详细解释版

设计联合索引要看：

| 因素 | 影响 |
|---|---|
| 高频查询条件 | 优先服务核心 SQL |
| 等值条件 | 通常适合放在前面 |
| 排序字段 | 接在连续等值条件后更容易复用 |
| 范围字段 | 放太前可能截断后续列定位 |
| 选择性 | 影响扫描范围，但不是唯一因素 |

### 9.3 例子

查询：

```sql
select id, order_no, created_at
from order_tab
where tenant_id = ?
  and status = ?
order by created_at desc
limit 20;
```

即使 `created_at` 区分度高，也不适合放在最前面做 `(created_at, tenant_id, status)`。  
更常见的索引是：

```sql
create index idx_tenant_status_time
on order_tab(tenant_id, status, created_at);
```

因为这个索引先按租户和状态缩小范围，再按时间顺序取最新记录。

## 10. MySQL 为什么有时不走索引？（中等难度）

### 10.1 最简练版

**MySQL 不走索引通常是因为优化器认为走索引成本更高。**  
如果条件选择性差、需要回表很多、统计信息不准或索引不匹配排序，优化器可能选择全表扫描。  
所以不能只看有没有索引，要看这个索引能不能减少实际成本。  
排查时看 `EXPLAIN`、数据分布、统计信息和真实执行耗时。

### 10.2 详细解释版

常见原因：

| 原因 | 说明 |
|---|---|
| 返回比例太高 | 走索引再回表不如全表扫 |
| 回表成本高 | 二级索引筛出太多行 |
| 统计信息不准 | 优化器估算错误 |
| 索引不匹配排序 | 走索引仍要额外排序 |
| 条件写法不友好 | 函数、转换、模糊匹配 |

`force index` 只能作为谨慎手段，不能把优化器不选索引的问题都用强制索引盖住。

### 10.3 例子

```sql
select *
from order_tab
where status = 1;
```

如果 `status = 1` 覆盖了全表 `80%` 的数据，走 `idx_status` 后还要回表读取大量整行，优化器可能认为全表扫描更便宜。  
这时更合理的做法是增加更具体的条件，比如 `tenant_id` 或时间范围，而不是强行走单列状态索引。

## 11. 前缀索引适合什么场景？（中等难度）

### 11.1 最简练版

**前缀索引适合长字符串字段，既想加速查询，又不想让索引太大。**  
比如邮箱、URL（Uniform Resource Locator，统一资源定位符）、长编码可以只取前 N 个字符建索引。  
它能节省空间，但区分度可能下降，也不能完全支持覆盖索引返回完整字段。  
使用前要评估前缀长度的选择性。

### 11.2 详细解释版

前缀索引的关键是选合适长度。太短区分度差，扫描多；太长节省空间不明显。  
可以通过不同前缀长度的 distinct 比例来判断：

```sql
select
  count(distinct left(email, 6)) / count(*) as p6,
  count(distinct left(email, 10)) / count(*) as p10,
  count(distinct email) / count(*) as p_full
from user_tab;
```

### 11.3 例子

```sql
create index idx_email_prefix
on user_tab(email(10));
```

适合：

```sql
select id
from user_tab
where email = 'foo@example.com';
```

不适合指望它完全覆盖：

```sql
select email
from user_tab
where email = 'foo@example.com';
```

因为索引里只存前 `10` 个字符，完整 `email` 仍可能需要回表确认。

## 12. 唯一索引和普通索引有什么差异？（中等难度）

### 12.1 最简练版

**唯一索引保证列值唯一，普通索引只负责加速查询。**  
查询上，唯一索引等值命中后最多一条记录，可以更快结束；普通索引可能继续向后找重复值。  
写入上，唯一索引需要检查冲突，普通索引不需要做唯一性校验。  
选择时先看业务约束，如果字段天然唯一，就应该用唯一索引表达约束。

### 12.2 详细解释版

唯一索引的价值不只是性能，更重要的是数据正确性。  
例如手机号、订单号、业务唯一单号，如果业务要求不能重复，就应该依赖数据库唯一索引兜底。  
普通索引用于非唯一字段，例如状态、类型、创建时间、租户 ID 等。

| 维度 | 唯一索引 | 普通索引 |
|---|---|---|
| 约束 | 保证唯一 | 不保证唯一 |
| 查询 | 命中后可停止 | 可能继续扫描重复值 |
| 写入 | 要检查冲突 | 维护索引即可 |
| 适用 | 订单号、手机号 | 状态、时间、分类 |

### 12.3 例子

```sql
create unique index uk_order_no
on order_tab(order_no);
```

这不仅能加速：

```sql
select *
from order_tab
where order_no = ?;
```

还能防止重复订单号写入。这个约束不能只放在应用层判断，因为并发下应用层检查可能有竞态。

## 13. 索引越多越好吗？（基本难度）

### 13.1 最简练版

**索引不是越多越好，索引会加速读，也会拖慢写。**  
每次插入、更新、删除都要维护相关索引，还会占用磁盘和 Buffer Pool（缓冲池）。  
过多索引还可能让优化器选择空间变复杂，甚至误选低效计划。  
索引应该围绕高频查询和核心链路建设，并定期清理重复和低价值索引。

### 13.2 详细解释版

索引成本：

| 成本 | 说明 |
|---|---|
| 写入成本 | 插入和更新要维护索引树 |
| 存储成本 | 每个索引都占磁盘 |
| 缓存成本 | 索引页占用 Buffer Pool |
| DDL 风险 | 大表加索引有锁、耗时、复制延迟风险 |
| 优化器成本 | 索引过多可能误选 |

治理思路：

- 保留服务核心查询的索引。
- 合并能被联合索引覆盖的单列索引。
- 删除长期不用且写入成本高的索引。
- 新增索引前先评估读收益和写代价。

### 13.3 例子

如果已有：

```sql
key idx_tenant_status_time (tenant_id, status, created_at),
key idx_tenant (tenant_id),
key idx_status (status)
```

`idx_tenant` 很可能能被联合索引前缀覆盖，但 `idx_status` 是否保留要看有没有大量只按 `status` 查询的高频 SQL。  
不能机械删除，也不能无脑保留，要用慢日志和索引使用情况判断。

## 14. order by 怎么利用索引？（中等难度）

### 14.1 最简练版

**`order by` 能否利用索引，关键看排序字段是否接在连续等值过滤条件后，并且排序方向是否一致。**  
联合索引既能过滤又能排序时，可以避免 `Using filesort`。  
如果中间断了最左前缀、排序字段顺序不一致或混合方向不支持，就可能额外排序。  
优化列表页时通常按 `where + order by + limit` 一起设计索引。

### 14.2 详细解释版

索引 `(tenant_id, status, created_at, id)` 对下面查询很友好：

```sql
where tenant_id = ?
  and status = ?
order by created_at desc, id desc
limit 20
```

因为 `tenant_id` 和 `status` 是等值条件，后面的 `created_at, id` 可以形成有序扫描。  
如果只按 `created_at` 排，但没有固定前面的 `tenant_id`、`status`，就很难利用这个索引做全局排序。

### 14.3 例子

```sql
create index idx_tenant_status_time_id
on order_tab(tenant_id, status, created_at, id);
```

| SQL | 是否友好 |
|---|---|
| `where tenant_id=? and status=? order by created_at,id` | 友好 |
| `where tenant_id=? order by status,created_at` | 可能友好 |
| `where status=? order by created_at` | 不匹配最左前缀 |
| `where tenant_id=? and status>? order by created_at` | 范围后排序能力下降 |

## 15. 深分页和索引有什么关系？（高难度）

### 15.1 最简练版

**索引能减少深分页的排序和回表成本，但不能消除大 offset 本身。**  
`limit 100000, 20` 仍然要先扫描并丢弃前 10 万条，再返回 20 条。  
优化深分页优先用游标翻页，把 offset 改成基于上一页最后一条继续查。  
如果必须保留页码，可以用覆盖索引加延迟关联，减少无效回表。

### 15.2 详细解释版

深分页慢的本质是扫描量随 offset 线性增长。  
联合索引可以让这次扫描按正确顺序走索引，覆盖索引可以让被丢弃的记录少回表，但数据库仍要走过 offset。  
因此真正的优化方向是改变访问方式，而不是只加索引。

| 方案 | 作用 | 代价 |
|---|---|---|
| 联合索引 | 减少排序 | offset 仍存在 |
| 覆盖索引 | 减少回表 | 索引变宽 |
| 游标翻页 | 扫描稳定 | 不支持任意跳页 |
| 延迟关联 | 兼容页码 | 仍要扫 offset |

### 15.3 例子

原 SQL：

```sql
select id, title, created_at
from article
where status = 1
order by created_at desc, id desc
limit 100000, 20;
```

游标翻页：

```sql
select id, title, created_at
from article
where status = 1
  and (
    created_at < ?
    or (created_at = ? and id < ?)
  )
order by created_at desc, id desc
limit 20;
```

配套索引：

```sql
create index idx_status_time_id
on article(status, created_at, id);
```

口述重点：**深分页不是返回多，而是为了返回后面 20 条，前面很多条也要先走一遍。**
