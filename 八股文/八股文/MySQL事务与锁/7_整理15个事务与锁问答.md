## 1. 事务的 ACID 是什么？

### 1.1 最简练版

**ACID 是事务正确性的四个维度：原子性、一致性、隔离性、持久性。**  
原子性保证一组 SQL 要么全成功，要么全失败，主要靠 undo log 支持回滚。  
一致性保证事务前后满足数据库约束和业务约束。  
隔离性保证并发事务之间影响可控，主要靠锁和 MVCC。  
持久性保证提交后崩溃可恢复，主要靠 redo log。

### 1.2 一句话背诵

**undo 保证能撤销，redo 保证能恢复，锁和 MVCC 保证并发下隔离，最终目标是业务一致。**

## 2. MySQL 有哪些隔离级别？

### 2.1 最简练版

MySQL 常见四种隔离级别是：

| 隔离级别 | 特点 |
|---|---|
| READ UNCOMMITTED | 可能读到未提交数据 |
| READ COMMITTED | 只能读已提交数据 |
| REPEATABLE READ | 同一事务内普通快照读稳定 |
| SERIALIZABLE | 最严格，并发最低 |

InnoDB 默认通常是 `REPEATABLE READ`。

### 2.2 一句话背诵

**隔离级别越高，并发异常越少，但锁和等待成本通常越高。**

## 3. RC 和 RR 的核心区别是什么？

### 3.1 最简练版

**RC 和 RR 的核心区别是 ReadView 创建时机。**  
RC 下每条普通 select 都创建新的 ReadView，所以能看到其他事务刚提交的数据。  
RR 下同一事务内普通快照读通常复用第一次 ReadView，所以多次读取结果稳定。  
这就是 RC 可能不可重复读，而 RR 能实现普通读可重复的关键。

### 3.2 例子

```text
A 第一次读余额 = 100
B 更新余额 = 80 并提交
A 第二次读

RC：读到 80
RR：普通快照读仍读到 100
```

## 4. 脏读、不可重复读、幻读怎么区分？

### 4.1 最简练版

| 问题 | 含义 |
|---|---|
| 脏读 | 读到其他事务未提交的数据 |
| 不可重复读 | 同一事务两次读同一行，行内容变了 |
| 幻读 | 同一事务两次按条件查，结果集合变了 |

一句话：**脏读看提交没提交，不可重复读看同一行，幻读看结果集合。**

---

## 5. MVCC 到底解决了什么问题？

### 5.1 最简练版

**MVCC 解决的是普通读和写之间互相阻塞的问题。**  
它让写操作生成新版本，同时保留旧版本。  
普通读根据 ReadView 读取自己可见的版本，而不是一定读最新版本。  
这样读不用等写，写也不用等普通读。

### 5.2 追问补充

MVCC 不能完全解决写写冲突。  
两个事务同时更新同一行，仍然要靠行锁控制。

---

## 6. ReadView 是怎么工作的？

### 6.1 最简练版

**ReadView 是一致性读的可见性快照。**  
它记录创建快照时哪些事务还活跃，以及事务 ID 的上下界。  
判断某个版本是否可见时，用版本上的 `DB_TRX_ID` 和 ReadView 比较。  
如果版本来自快照前已提交事务，就可见；如果来自快照后事务或当时未提交事务，就不可见。

### 6.2 可见性规则

| 条件 | 结果 |
|---|---|
| 是当前事务自己的修改 | 可见 |
| 小于最小活跃事务 ID | 可见 |
| 大于等于下一个待分配事务 ID | 不可见 |
| 在活跃事务列表中 | 不可见 |
| 不在活跃事务列表中 | 可见 |

## 7. undo log 在事务里有什么作用？

### 7.1 最简练版

**undo log 有两个核心作用：回滚和 MVCC 历史版本。**  
事务修改数据前，InnoDB 会把旧值写入 undo log。  
如果事务失败，可以用 undo log 回滚。  
如果其他事务需要读旧快照，也可以沿着 undo 版本链找到可见版本。

### 7.2 追问补充

长事务会拖住 undo 清理，因为老 ReadView 可能还需要历史版本。  
所以长事务会导致 undo 膨胀和版本链变长。

---

## 8. 快照读和当前读有什么区别？

### 8.1 最简练版

**快照读读的是 ReadView 可见版本，当前读读的是最新已提交版本并加锁。**  
普通 select 通常是快照读。  
`select ... for update`、`select ... for share`、`update`、`delete` 是当前读。  
RR 下普通 select 可重复，不代表当前读也读旧快照。

### 8.2 一句话背诵

**普通查询看快照，修改和加锁查询看当前。**

---

## 9. InnoDB RR 为什么大部分场景能避免幻读？

### 9.1 最简练版

**RR 下要分快照读和当前读。**  
普通快照读复用 ReadView，所以不会看到其他事务后来插入的新记录。  
范围当前读会使用 next-key lock，锁住记录和间隙，阻止其他事务插入满足条件的新记录。  
所以 InnoDB RR 在大部分常见场景能避免幻读。

### 9.2 追问补充

不要简单说“RR 绝对解决所有幻读”。  
更好的说法是：**InnoDB 的 RR 通过 MVCC 和 next-key lock 分别处理快照读和当前读下的幻读风险。**

## 10. 行锁、间隙锁、next-key lock 是什么？

### 10.1 最简练版

| 锁 | 含义 |
|---|---|
| record lock | 锁住已有索引记录 |
| gap lock | 锁住索引记录之间的间隙 |
| next-key lock | record lock + gap lock |

record lock 防止已有记录被改。  
gap lock 防止间隙里插入新记录。  
next-key lock 同时锁记录和间隙，常用于 RR 下范围当前读防幻读。

---

## 11. select for update 会加什么锁？

### 11.1 最简练版

**`select ... for update` 是当前读，会读取最新数据并加排他锁。**  
如果主键或唯一索引等值命中，通常锁具体记录。  
如果是范围查询或普通索引查询，在 RR 下可能加 next-key lock。  
如果没有合适索引，可能扫描大量记录并扩大锁范围。

### 11.2 追问补充

具体加锁范围取决于：

1. 隔离级别。
2. 索引类型。
3. 查询条件。
4. 记录是否存在。
5. 实际执行计划。

---

## 12. 为什么没有索引会导致锁范围变大？

### 12.1 最简练版

**InnoDB 的行锁基于索引实现，没有合适索引时就要扫描更多索引记录。**  
扫描越多，需要判断和加锁的记录越多。  
在更新或 `for update` 场景下，这会导致大量行被锁住，线上看起来像锁表。  
所以锁优化经常先看执行计划和索引。

### 12.2 一句话背诵

**锁不是只看 where 条件写得窄不窄，而是看执行计划实际扫得宽不宽。**

## 13. 什么情况下容易发生死锁？

### 13.1 最简练版

**死锁通常来自多个事务以不同顺序持有并等待对方的锁。**  
常见原因包括加锁顺序不一致、范围更新锁太多、事务太长、热点行竞争、索引不合理。  
InnoDB 能检测死锁，并回滚其中一个事务。  
业务侧要能重试，同时从根因上减少锁范围和持锁时间。

### 13.2 规避方式

1. 统一访问表和行的顺序。
2. 使用精准索引缩小锁范围。
3. 小事务，避免事务内 RPC。
4. 批量更新拆小。
5. 热点数据拆分或异步化。
6. 捕获死锁错误并做幂等重试。

## 14. 读写分离下如何保证一致性？

### 14.1 最简练版

**读写分离的一致性问题本质是主从延迟。**  
强一致读直接读主库。  
写后立刻读可以做会话级短时间读主，或等待 GTID/binlog 位点追上。  
普通列表、统计可以读从库，接受最终一致。  
同时要监控从库延迟，延迟过大时摘从或关键读切主。

### 14.2 追问补充

半同步复制不等于读从强一致，因为从库收到日志不代表已经回放完成。

## 15. 线上锁等待或主从延迟怎么排查？

### 15.1 最简练版

**锁等待先看事务、锁、执行计划；主从延迟先看复制状态、位点差、心跳和大事务。**

锁等待常用：

```sql
show engine innodb status\G
select * from information_schema.innodb_trx;
show processlist;
explain ...
```

主从延迟常用：

```sql
show replica status\G
show processlist;
```

同时结合慢 SQL、任务发布、流量峰值和机器资源判断根因。

### 15.2 回答要点

排查不是只看数据库命令，还要关联业务时间线：

1. 是否有批量任务。
2. 是否刚发布。
3. 是否流量突增。
4. 是否 DDL。
5. 是否某个 SQL 扫描行数异常。
