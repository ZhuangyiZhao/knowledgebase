## 1. Redis 有哪些持久化方式？（基本难度）

### 1.1 最简练版

**Redis 主要有 RDB（Redis Database，快照持久化）和 AOF（Append Only File，追加日志持久化）。**  
RDB 保存某个时间点的数据快照，恢复快但可能丢一段数据。  
AOF 记录写命令，数据丢失更少，但文件更大、恢复可能更慢。生产常用 AOF 或混合持久化。

### 1.2 详细解释版

RDB 适合备份、全量复制和快速恢复；AOF 更适合对数据丢失敏感的场景。开启持久化不等于绝对不丢数据，还要看 fsync（File Synchronization，文件同步）策略、磁盘可靠性、主从复制和备份策略。  
如果 Redis 只做可重建缓存，可以弱化持久化；如果承担库存、任务、计数等存储职责，就必须设计恢复链路。

### 1.3 例子

```text
RDB：某个时间点的内存照片
AOF：每次写操作的追加记录
```

## 2. RDB 的优缺点是什么？（中等难度）

### 2.1 最简练版

**RDB 优点是文件紧凑、恢复快、适合备份；缺点是快照间隔内的数据可能丢失。**  
生成 RDB 通常通过子进程完成，主进程继续处理请求。  
但 fork 和写时复制可能带来内存和延迟压力。

### 2.2 详细解释版

RDB 保存的是二进制快照，适合冷备和灾难恢复。它不会像 AOF 那样不断追加命令，所以文件通常更小。  
风险是如果两次快照之间 Redis 崩溃，这段时间写入会丢失。对于高写入实例，`bgsave` 期间父进程修改页面会触发 COW（Copy On Write，写时复制），导致额外内存占用。

### 2.3 例子

| 维度 | RDB |
|---|---|
| 恢复速度 | 快 |
| 数据完整性 | 可能丢快照后的写入 |
| 适用场景 | 备份、全量复制、缓存恢复 |

## 3. AOF 的优缺点是什么？（中等难度）

### 3.1 最简练版

**AOF 通过追加写命令提高数据安全性，通常比 RDB 丢数据更少。**  
缺点是文件更大，需要重写压缩，恢复速度可能慢于 RDB。  
它的可靠性和性能主要取决于 `appendfsync` 策略。

### 3.2 详细解释版

AOF 记录的是协议格式写命令，Redis 重启时重放命令恢复数据。`appendfsync always` 最安全但慢，`everysec` 是常见折中，`no` 性能好但风险高。  
AOF 长期追加会膨胀，需要 `BGREWRITEAOF` 生成更小的新文件。重写期间同样要关注 fork、COW 和磁盘 I/O。

### 3.3 例子

```text
set a 1
incr counter
hset user:1 name zhangsan
```

这些写命令会追加到 AOF 文件，重启时再重放。

## 4. RDB 和 AOF 怎么选择？（中等难度）

### 4.1 最简练版

**只做缓存且可重建，可以用 RDB 或不开强持久化；有数据安全要求，优先 AOF everysec 或混合持久化。**  
RDB 重恢复速度和备份，AOF 重数据完整性。  
关键业务还要配合主从、备份、监控和恢复演练。

### 4.2 详细解释版

选择不是只看性能，还要看 RPO（Recovery Point Objective，恢复点目标）和 RTO（Recovery Time Objective，恢复时间目标）。RPO 越小，越倾向 AOF；RTO 越短，越关注 RDB 或混合持久化。  
如果 Redis 只是数据库缓存，丢失后可以预热；如果 Redis 保存限流计数、延迟任务、会话状态，就要明确最大可丢失窗口。

### 4.3 例子

| 场景 | 推荐 |
|---|---|
| 商品详情缓存 | RDB 或弱持久化 |
| 分布式会话 | AOF everysec |
| 轻量消息流 | AOF + 备份 |

## 5. AOF 重写解决什么问题？（中等难度）

### 5.1 最简练版

**AOF 重写用于压缩历史命令，减少文件体积和恢复时间。**  
它不是读取旧 AOF 逐行改写，而是根据当前内存数据生成等价写命令。  
重写期间新写入会追加到增量缓冲，最后合并。

### 5.2 详细解释版

例如一个 key 被 `incr` 一万次，旧 AOF 可能有一万条命令，重写后只需要一条 `set` 保存最终值。  
重写由子进程完成，避免主进程长时间阻塞。但 fork、COW、磁盘写入和重写缓冲合并仍可能造成抖动，应该监控 `aof_rewrite_in_progress`、磁盘 I/O 和延迟。

### 5.3 图示

```text
旧 AOF 大量历史命令 -> 当前内存状态 -> 新 AOF 等价命令
```

## 6. 混合持久化是什么？（中等难度）

### 6.1 最简练版

**混合持久化是在 AOF 文件中同时包含 RDB 快照和后续增量 AOF。**  
前半段用 RDB 加快恢复，后半段用 AOF 降低数据丢失。  
它兼顾恢复速度和数据完整性，是常见生产选择。

### 6.2 详细解释版

开启混合持久化后，AOF 重写生成的新文件前部是 RDB 格式快照，之后追加重写期间的增量写命令。重启时先加载快照，再重放增量。  
这减少了纯 AOF 重放大量命令的成本，但仍要关注 AOF 损坏检测、备份可用性和版本兼容。

### 6.3 例子

```text
AOF 文件：
[RDB 快照数据][增量写命令][增量写命令]
```

## 7. bgsave 为什么可能造成延迟抖动？（高难度）

### 7.1 最简练版

**`bgsave` 会 fork 子进程，fork 本身和写时复制都可能造成延迟抖动。**  
内存越大、页表越多，fork 耗时越明显。  
如果写入很频繁，COW 会复制更多内存页，增加内存压力和系统抖动。

### 7.2 详细解释版

`bgsave` 子进程负责把快照写盘，父进程继续处理请求。fork 时需要复制页表；之后父进程修改某个内存页，操作系统会复制旧页给子进程保存快照视图。  
如果实例很大、写入很密集、机器内存余量不足或磁盘慢，就容易出现延迟尖刺甚至 OOM（Out Of Memory，内存不足）。

### 7.3 例子

```bash
info stats | grep latest_fork_usec
info persistence
```

## 8. 写时复制是什么？（高难度）

### 8.1 最简练版

**写时复制是 fork 后父子进程共享内存页，只有某页被修改时才复制一份。**  
它避免 fork 时直接复制全部内存。  
但如果持久化期间写入很多，会产生大量复制页，导致额外内存占用。

### 8.2 详细解释版

Redis 做 RDB 或 AOF 重写时，子进程需要看到 fork 那一刻的数据视图。操作系统用 COW 保证子进程视图稳定：未修改页面父子共享，修改页面复制旧版本给子进程。  
因此 COW 成本与写入量、对象分布、内存页修改情况有关。高峰期做快照更容易放大延迟和内存压力。

### 8.3 图示

```text
fork 后：父子共享 page A
父进程修改 page A -> 复制旧 page A 给子进程
```

## 9. appendfsync 三种策略有什么区别？（中等难度）

### 9.1 最简练版

**`always` 每次写都刷盘，最安全但最慢；`everysec` 每秒刷盘，常用折中；`no` 交给操作系统，性能好但丢数据风险最大。**  
生产里最常见是 `everysec`。  
选择时要结合可接受的数据丢失窗口和磁盘性能。

### 9.2 详细解释版

AOF 写入先进入内核页缓存，fsync 决定何时强制落盘。`always` 理论上最多丢当前命令，但延迟受磁盘影响很大；`everysec` 通常最多丢 1 秒左右；`no` 可能在系统崩溃时丢更多。  
对于大多数业务，`everysec` 是可靠性和性能的平衡点。

### 9.3 表格

| 策略 | 性能 | 数据风险 |
|---|---|---|
| always | 低 | 最小 |
| everysec | 中 | 较小 |
| no | 高 | 最大 |

## 10. Redis 重启恢复数据的流程是什么？（中等难度）

### 10.1 最简练版

**Redis 重启时会按配置加载持久化文件，通常 AOF 开启时优先加载 AOF。**  
纯 RDB 直接加载快照；纯 AOF 重放命令；混合持久化先加载 RDB 部分再重放增量。  
恢复速度取决于文件大小、磁盘性能和数据结构复杂度。

### 10.2 详细解释版

恢复前要检查持久化文件是否完整，AOF 异常时可用 `redis-check-aof` 修复，RDB 可用 `redis-check-rdb` 检查。  
大实例恢复不能只看 Redis 进程启动，还要看主从同步、客户端重连、缓存预热和业务流量是否恢复。

### 10.3 例子

```bash
redis-check-aof --fix appendonly.aof
redis-check-rdb dump.rdb
```

## 11. maxmemory 有什么作用？（基本难度）

### 11.1 最简练版

**`maxmemory` 用来限制 Redis 最大可用内存，达到上限后按淘汰策略处理写入。**  
没有合理限制时，Redis 可能把机器内存打满。  
生产实例应预留系统、复制、持久化和 COW 所需内存。

### 11.2 详细解释版

Redis 内存不只包括业务数据，还包括对象元数据、复制缓冲区、客户端输出缓冲区、AOF 缓冲、碎片和 fork 期间 COW。  
如果 `maxmemory` 设置过高，持久化或重写时可能 OOM；设置过低，又可能过早淘汰关键缓存。

### 11.3 例子

```bash
config get maxmemory
info memory
```

## 12. Redis 有哪些内存淘汰策略？（中等难度）

### 12.1 最简练版

**Redis 淘汰策略主要分为只淘汰设置过期时间的 key、从所有 key 中淘汰，以及不淘汰直接报错。**  
算法上有 LRU（Least Recently Used，最近最少使用）、LFU（Least Frequently Used，最不经常使用）、random 和 ttl。  
缓存场景常用 `allkeys-lru` 或 `allkeys-lfu`。

### 12.2 详细解释版

`volatile-*` 只在设置 TTL 的 key 中选择，适合只希望缓存类数据被淘汰；`allkeys-*` 在所有 key 中选择，适合整个实例都是缓存；`noeviction` 会在内存满时让写命令报错。  
策略选择要匹配数据分层，不能把不可丢的状态和可淘汰缓存混放在同一实例。

### 12.3 表格

| 策略 | 范围 | 场景 |
|---|---|---|
| allkeys-lru | 所有 key | 通用缓存 |
| volatile-ttl | 有 TTL 的 key | 只淘汰临期数据 |
| noeviction | 不淘汰 | 关键状态存储 |

## 13. LRU 和 LFU 有什么区别？（中等难度）

### 13.1 最简练版

**LRU 看最近有没有访问，LFU 看一段时间内访问频率。**  
LRU 对短期热点敏感，LFU 更能识别长期高频 key。  
如果业务热点短促变化，LRU 简单有效；如果热点稳定，LFU 通常更稳。

### 13.2 详细解释版

Redis 使用近似 LRU/LFU，不是维护全量精确链表。LRU 可能被一次性扫描污染，比如冷数据被批量访问后挤掉真正热点。LFU 增加访问计数和衰减机制，可以降低偶发访问影响。  
选型时要结合命中率、热点周期和内存成本观察。

### 13.3 例子

```text
热点新闻：LRU 更敏感
长期高频配置：LFU 更稳定
```

## 14. noeviction 策略有什么风险？（基本难度）

### 14.1 最简练版

**`noeviction` 在内存满时不会淘汰 key，而是让写命令失败。**  
它适合不允许静默丢数据的状态存储。  
风险是业务没有处理写失败时，会直接放大成可用性问题。

### 14.2 详细解释版

如果 Redis 做缓存，`noeviction` 可能让新缓存无法写入，造成数据库压力上升；如果 Redis 做状态存储，它能避免关键 key 被淘汰，但必须有容量告警和扩容机制。  
使用该策略时，要监控 `used_memory`、`mem_fragmentation_ratio`、写错误和客户端异常。

### 14.3 例子

```bash
config set maxmemory-policy noeviction
```

## 15. 什么是大 key？（中等难度）

### 15.1 最简练版

**大 key 是 value 很大或集合成员很多的 key。**  
例如几 MB 的 String、几十万成员的 Set、超大 Hash 或 ZSet。  
大 key 没有绝对标准，要结合网络、延迟、内存和业务访问频率判断。

### 15.2 详细解释版

大 key 的本质问题是单次命令处理的数据量太大。它会增加网络传输、阻塞事件循环、拉长持久化和主从同步时间，还会让迁移、删除和故障恢复变慢。  
常见治理是拆分、分页、压缩、只存索引不存大对象、异步删除。

### 15.3 例子

```text
user:followers:1001 包含 200 万成员
sku:detail:123 是 8MB JSON
```

## 16. 大 key 会造成哪些问题？（中等难度）

### 16.1 最简练版

**大 key 会造成命令阻塞、网络拥塞、内存倾斜、主从同步变慢和删除抖动。**  
在集群里，大 key 还可能让某个 slot 或节点负载特别高。  
它是 Redis 延迟尖刺和容量不均的重要来源。

### 16.2 详细解释版

读取大 key 会占用 Redis 主线程和网络输出缓冲；删除大 key 会释放大量对象，可能阻塞；迁移大 key 会拖慢 slot 迁移；持久化时大对象也会增加 I/O 压力。  
因此大 key 不只是内存问题，也是延迟、稳定性和运维问题。

### 16.3 图示

```text
一个大 key -> 单命令耗时长 -> 后续命令排队 -> 延迟升高
```

## 17. 如何发现和治理大 key？（高难度）

### 17.1 最简练版

**发现大 key 可以用 `redis-cli --bigkeys`、`--memkeys`、采样 scan、监控慢日志和内存分析。**  
治理上优先拆分结构、分页访问、限制返回量、用 `unlink` 异步删除。  
关键是建立准入规范，避免新大 key 继续产生。

### 17.2 详细解释版

排查时先确认类型、大小、访问频率和业务归属；再判断是否能拆成多 key，比如按用户、日期、分片号拆分。对于历史大 key，删除前要评估是否用 `unlink`、是否分批清理、是否避开高峰。  
对于集合类 key，禁止 `hgetall`、`smembers` 直接全量拉取，改为 `hscan`、`sscan`、`zscan` 或按 score/id 分页。

### 17.3 例子

```bash
redis-cli --bigkeys
redis-cli --memkeys
unlink big:key
```

## 18. 什么是热 key？（中等难度）

### 18.1 最简练版

**热 key 是访问量远高于其他 key 的 key。**  
它会让单个 Redis 节点、单个分片甚至单个 CPU 核成为瓶颈。  
热 key 常出现在爆款商品、热点新闻、活动配置和全局计数上。

### 18.2 详细解释版

热 key 的危险在于请求集中，即使 value 不大，也会造成 Redis QPS（Queries Per Second，每秒查询数）集中、网络输出集中和主线程繁忙。  
集群模式下热 key 不会因为分片自动分散，因为同一个 key 只落到一个 slot。需要业务侧做本地缓存、复制 key、读写分离或限流。

### 18.3 例子

```text
activity:config:618 在活动开始瞬间被所有服务读取
```

## 19. 如何治理热 key？（高难度）

### 19.1 最简练版

**热 key 治理思路是发现、分散、缓存、限流和降级。**  
读热点可以用本地缓存、多副本 key、读从库；写热点可以拆分计数、批量合并。  
热点不可避免时，要有降级值和熔断保护。

### 19.2 详细解释版

发现方式包括客户端埋点、代理层统计、Redis `hotkeys` 采样、慢日志和实例 QPS 异常。治理读热点时，可以把 `hot:key` 扩展成 `hot:key:0..N` 随机读，或者在服务内用短 TTL 本地缓存。治理写热点时，可以按分片计数，再异步汇总。  
不要盲目把所有 key 都复制多份，否则会增加一致性和内存成本。

### 19.3 图示

```text
请求 -> 本地缓存命中 -> 返回
    -> 未命中 -> Redis 多副本 key 随机读
```

## 20. Redis 内存碎片率高怎么办？（高难度）

### 20.1 最简练版

**内存碎片率高说明 Redis 申请的内存和实际使用内存差距较大。**  
原因可能是频繁增删改、大对象变化、分配器碎片或系统页回收不及时。  
处理方式包括观察指标、开启主动碎片整理、重启迁移和优化数据模型。

### 20.2 详细解释版

重点看 `used_memory`、`used_memory_rss`、`mem_fragmentation_ratio`。碎片率长期过高会浪费内存，甚至触发机器内存压力。  
可以开启 `activedefrag`，但它会消耗 CPU，需要灰度观察。极端情况下，通过主从切换、迁移重建、重启实例释放碎片。根治仍要减少大 key、频繁扩缩容 value 和不合理的数据结构。

### 20.3 例子

```bash
info memory
config set activedefrag yes
```
