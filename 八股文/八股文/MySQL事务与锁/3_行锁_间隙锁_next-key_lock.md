## 1. InnoDB 行锁是什么？

### 1.1 最简练版

**InnoDB 的行锁不是直接锁“数据行”这个抽象概念，而是锁索引记录。**  
如果 SQL 能走到合适索引，锁范围通常比较小。  
如果没有合适索引，InnoDB 可能扫描大量记录并加锁，看起来就像锁了整张表。  
所以锁问题往往和索引设计、执行计划强相关。

### 1.2 详细解释版

很多人会说“InnoDB 支持行锁”，这句话没错，但不够精确。更准确的说法是：

```text
InnoDB 的行级锁是基于索引实现的。
```

例如：

```sql
update user_tab
set name = 'new'
where id = 1001;
```

如果 `id` 是主键，InnoDB 可以直接定位到主键索引上的一条记录，然后对这条索引记录加锁。

但如果条件没有索引：

```sql
update user_tab
set name = 'new'
where phone = '13800000000';
```

而 `phone` 没有索引，数据库可能需要扫描很多记录。扫描过程中命中的记录会被加锁，在一些场景下锁范围会非常大，线上表现就是大量事务等待。

### 1.3 为什么要关注执行计划？

因为锁加在哪里，取决于 SQL 实际走哪个索引，而不是你以为它会走哪个索引。

排查锁问题时，要看：

```sql
explain update user_tab set name = 'new' where phone = '13800000000';
```

重点关注：

- 是否走了预期索引。
- 扫描行数是否异常。
- 是否出现全表扫描。
- where 条件是否导致索引失效。

## 2. 常见锁类型

### 2.1 按粒度看

| 锁类型 | 作用 |
|---|---|
| 表锁 | 锁整张表，粒度大 |
| 行锁 | 锁索引记录或索引范围，粒度小 |
| 意向锁 | 表级锁，用来表示事务准备在表中加行锁 |

InnoDB 的意向锁常见有：

| 意向锁 | 含义 |
|---|---|
| IS | 意向共享锁 |
| IX | 意向排他锁 |

意向锁不是面试最核心，但可以用来解释为什么表锁和行锁能快速判断冲突。

### 2.2 按读写模式看

| 锁类型 | 含义 | 兼容性 |
|---|---|---|
| 共享锁 S Lock | 允许读，不允许别人改 | S 和 S 兼容 |
| 排他锁 X Lock | 当前事务要改 | X 和其他锁通常不兼容 |

例子：

```sql
select * from order_tab where id = 1 lock in share mode;
select * from order_tab where id = 1 for update;
```

`lock in share mode` 或 MySQL 8.0 的 `for share` 加共享锁。  
`for update` 加排他锁。

## 3. record lock、gap lock、next-key lock

### 3.1 record lock

**record lock 是锁住索引上的某条记录。**

例如主键等值命中：

```sql
select *
from user_tab
where id = 10
for update;
```

如果 `id = 10` 存在，通常锁住主键索引中 `id = 10` 这条记录。

### 3.2 gap lock

**gap lock 是锁住两个索引记录之间的间隙，不锁具体记录本身。**  
它主要用来阻止其他事务往这个间隙插入新记录，从而减少幻读。

假设索引里已有：

```text
10, 20, 30
```

那么间隙包括：

```text
(-∞, 10)
(10, 20)
(20, 30)
(30, +∞)
```

如果事务锁住 `(10, 20)` 这个间隙，其他事务就不能插入 `id = 15`。

### 3.3 next-key lock

**next-key lock = record lock + gap lock。**  
它锁的是一个左开右闭区间，既锁记录，也锁记录前面的间隙。

假设索引值是：

```text
10, 20, 30
```

常见 next-key 区间可以理解为：

```text
(-∞, 10]
(10, 20]
(20, 30]
(30, +∞)
```

它的作用是：既防止已有记录被改，也防止范围内插入新记录。

### 3.4 三者对比

| 类型 | 锁什么 | 主要目的 |
|---|---|---|
| record lock | 已存在的索引记录 | 防止记录被并发修改 |
| gap lock | 索引记录之间的间隙 | 防止插入新记录 |
| next-key lock | 记录 + 前面间隙 | 同时防止修改和插入 |

## 4. 什么 SQL 会加什么锁？

### 4.1 普通 select

普通 `select` 是快照读，通常不加行锁。

```sql
select *
from order_tab
where id = 1;
```

它通过 MVCC 读取可见版本，不阻塞写，也不被普通写直接阻塞。

### 4.2 主键等值 + for update

```sql
select *
from order_tab
where id = 1
for update;
```

如果 `id` 是主键且记录存在，通常加 record lock。  
如果记录不存在，在 RR 下可能锁住对应插入间隙，防止其他事务插入这个主键值。

### 4.3 唯一索引等值查询

```sql
select *
from user_tab
where email = 'a@test.com'
for update;
```

如果 `email` 是唯一索引且等值命中，通常可以退化为 record lock，因为唯一索引能精确定位一条记录。  
如果没有命中，则可能锁住对应 gap，防止其他事务插入这个唯一值。

### 4.4 普通索引等值查询

```sql
select *
from order_tab
where user_id = 1001
for update;
```

如果 `user_id` 是普通索引，可能有多条记录。  
在 RR 下，为了防止其他事务继续插入 `user_id = 1001` 的新记录，通常会对匹配记录及相关间隙加 next-key lock。

### 4.5 范围查询

```sql
select *
from order_tab
where amount between 100 and 200
for update;
```

范围当前读在 RR 下常见会加 next-key lock，锁住扫描到的记录和范围间隙，防止其他事务插入满足条件的新记录。

### 4.6 无索引更新

```sql
update order_tab
set status = 'closed'
where buyer_phone = '13800000000';
```

如果 `buyer_phone` 没有索引，可能扫描聚簇索引大量记录，并对扫描命中的记录加锁。  
线上效果可能接近“锁表”，这是很多锁事故的来源。

## 5. RR 和 RC 下锁行为有什么差异？

### 5.1 最简练版

**RR 下为了防止幻读，范围当前读更容易使用 gap lock 和 next-key lock。**  
RC 下普通查询每条语句读最新提交数据，搜索和索引扫描中的 gap lock 通常更少。  
但 RC 下也不是完全没有 gap lock，外键检查、唯一性检查等场景仍可能使用。  
所以不能简单说 RC 没有间隙锁，只能说锁范围通常更小。

### 5.2 对比表

| 隔离级别 | 普通 select | 当前读 | 间隙锁使用 |
|---|---|---|---|
| RC | 快照读，每条语句新快照 | 读最新并加锁 | 通常较少 |
| RR | 快照读，事务内快照稳定 | 读最新并加锁 | 范围当前读常见 |

### 5.3 工程选择

如果业务大量范围更新，且锁冲突严重，可以评估：

1. 是否可以缩小事务范围。
2. 是否可以补充更精准的联合索引。
3. 是否可以把范围操作拆小批次。
4. 是否可以把部分业务切到 RC。
5. 是否能用状态机、乐观锁、唯一约束替代大范围悲观锁。

## 6. 死锁和锁等待怎么分析？

### 6.1 常用命令

查看最近一次死锁信息：

```sql
show engine innodb status\G
```

查看当前事务和锁等待：

```sql
select * from information_schema.innodb_trx;
```

MySQL 8.0 也常看 `performance_schema` 里的锁等待表。

### 6.2 分析思路

遇到锁等待或死锁，不要只看 SQL 文本，要按下面顺序分析：

1. 哪些事务在等待，哪些事务持锁。
2. SQL 的执行计划是否符合预期。
3. 是否走了正确索引。
4. 锁的是记录、间隙，还是范围。
5. 事务里是否有外部 RPC、慢查询、批量操作导致持锁时间过长。
6. 多个事务访问表和行的顺序是否一致。

### 6.3 常见规避方式

| 问题 | 规避方式 |
|---|---|
| 锁范围太大 | 加精准索引，避免全表扫描 |
| 持锁太久 | 缩短事务，把外部调用移出事务 |
| 访问顺序不一致 | 统一加锁顺序 |
| 批量更新太大 | 分批提交，控制每批大小 |
| 热点行竞争 | 拆分热点、异步化、乐观锁 |

## 7. 面试口述版

### 7.1 30 秒回答

InnoDB 的行锁是基于索引实现的，不是抽象地锁某一行。常见锁有 record lock、gap lock 和 next-key lock。record lock 锁已有索引记录，gap lock 锁记录之间的间隙，防止插入，next-key lock 是记录锁加间隙锁，通常用来解决范围当前读下的幻读问题。SQL 到底加什么锁和隔离级别、索引类型、查询条件、执行计划都有关。没有合适索引时，锁范围可能非常大。

### 7.2 90 秒回答

InnoDB 支持行锁，但它的行锁本质是加在索引记录上的。如果主键等值查询并且命中，`for update` 通常只锁一条记录；如果是普通索引等值或范围查询，在 RR 下为了防止其他事务插入新的匹配记录，可能会加 next-key lock，也就是锁住记录和记录前面的间隙。gap lock 本身不锁具体记录，主要阻止插入。next-key lock 则同时防止已有记录被改和范围内出现新记录。

所以分析锁问题时一定要看执行计划。如果 SQL 没走到合适索引，InnoDB 可能扫描很多索引记录并加锁，线上看起来就像锁表。优化方向通常是补充合适索引、缩短事务、统一加锁顺序、避免事务内做外部调用，以及把大批量范围更新拆小。

## 8. 高频追问

### 8.1 为什么说行锁基于索引？

因为 InnoDB 的记录是组织在索引里的，锁也是锁索引项。  
通过二级索引定位时，可能既锁二级索引记录，也涉及聚簇索引记录。  
如果没有合适索引，就只能扫描更多记录，锁范围也会变大。

### 8.2 gap lock 会阻塞查询吗？

gap lock 主要阻塞往间隙里插入记录，不阻塞普通快照读。  
但当前读、插入、更新之间是否冲突，要看具体锁模式和范围。

### 8.3 next-key lock 为什么能防幻读？

幻读的关键是“范围内多了新记录”。  
next-key lock 锁住记录和间隙后，其他事务不能在这个范围插入新记录，所以当前事务再次做范围当前读时，不会看到新插入的幻影记录。

### 8.4 唯一索引等值查询一定不会有间隙锁吗？

不能绝对说。  
如果唯一索引等值查询命中已有记录，通常可以只锁记录。  
如果查询的唯一值不存在，在 RR 下可能锁住对应间隙，防止其他事务插入这个值。

## 9. 容易踩坑

1. 不要说 InnoDB 行锁和索引无关。
2. 不要认为 `where id = ?` 一定只锁一行，前提是 `id` 真的是有效索引且执行计划命中。
3. 不要说 gap lock 锁的是某条记录，它锁的是间隙。
4. 不要说 RC 完全没有间隙锁，外键和唯一性检查等场景仍可能用到。
5. 不要在事务里做慢 RPC、sleep 或大批量循环更新，这会把锁持有时间拉长。
