# 项目复盘：一次数据库主从延迟或锁冲突问题

**适用场景：** 项目深挖、线上故障复盘、主从延迟排障、锁等待/死锁治理、面试行为题。

**回答模板：** 用 STAR 结构讲：背景、故障现象、影响范围、排查过程、根因、止血措施、长期治理、复盘收益。回答时要给出指标、命令、取舍和后续机制，不要只说“加索引解决了”。

## 1. 复盘回答框架

### 1.1 STAR 结构

| 部分 | 要回答的问题 |
|---|---|
| Situation 背景 | 这是哪个业务、什么链路、数据量和 QPS 大概多少 |
| Task 目标 | 当时要解决什么问题，SLA 或业务目标是什么 |
| Action 行动 | 如何定位、如何止血、如何修复、如何验证 |
| Result 结果 | 延迟、错误率、锁等待、投诉量等指标怎么变化 |

### 1.2 面试官真正想听什么

面试官不是只想听你遇到过故障，而是想确认：

1. 你是否能快速判断影响范围。
2. 你是否知道数据库复制、锁、事务的底层原因。
3. 你是否有止血意识，而不是一上来就重构。
4. 你是否能沉淀长期机制，避免重复发生。
5. 你是否能把技术问题翻译成业务影响。

### 1.3 推荐回答顺序

```text
先讲业务影响
再讲关键指标
再讲排查路径
再讲根因
再讲短期止血
再讲长期治理
最后讲复盘收益
```

## 2. 案例一：主从延迟导致写后读旧

### 2.1 背景

业务是订单创建后，用户会立刻进入订单详情页和订单列表页。  
系统做了读写分离：创建订单写主库，详情和列表默认读从库。  
正常情况下主从延迟在几十毫秒以内，业务没有明显感知。

### 2.2 故障现象

某次活动期间，用户反馈下单成功后订单列表里看不到订单，刷新几次后才出现。  
监控上看到：

| 指标 | 异常表现 |
|---|---|
| 从库延迟 | 从几十毫秒上涨到数秒 |
| 订单详情查询 | 部分请求返回空 |
| 用户投诉 | 集中在下单后立即查看 |
| 主库写入 | 成功率正常 |

这说明问题不是写失败，而是写成功后读路径读到了落后的从库。

### 2.3 排查过程

第一步，确认写链路是否成功：

```sql
select *
from order_tab
where order_id = ?;
```

直接查主库能查到订单，说明订单创建成功。

第二步，查从库是否落后：

```sql
show replica status\G
```

或旧版本：

```sql
show slave status\G
```

关注：

```text
Replica_IO_Running
Replica_SQL_Running
Seconds_Behind_Source
Relay_Log_Pos
Exec_Source_Log_Pos
```

第三步，看是否有大事务或慢回放：

```sql
show processlist;
```

发现从库 SQL 线程在回放一批大事务，导致后续订单变更日志排队。

第四步，结合业务发布和活动流量，确认活动批量任务和在线订单写入共用同一主从链路，大事务放大了复制延迟。

### 2.4 根因

根因可以概括为三层：

1. 架构上默认订单详情读从库，没有区分写后读场景。
2. 活动期间批量任务产生大事务，从库回放慢，主从延迟从毫秒级变成秒级。
3. 路由层只按读写类型分流，没有根据从库延迟动态切换。

### 2.5 短期止血

短期措施：

1. 下单成功后订单详情强制读主库。
2. 对刚下单用户打 3 到 5 秒读主标记，订单列表短窗口读主。
3. 暂停或限速批量任务。
4. 延迟严重的从库从读池摘除。
5. 对非核心列表提示稍后刷新，避免反复打查询。

止血目标是先恢复用户体验，不让用户看到“订单消失”。

### 2.6 长期治理

长期治理：

1. 读请求按一致性分级：强一致读主、读己之写短窗口读主、普通展示读从。
2. 引入心跳表监控真实复制延迟，超过阈值自动摘从。
3. 对订单状态增加 `version` 或 `updated_at` 校验，发现从库版本低于期望时切主。
4. 批量任务拆小事务，控制单事务行数和执行时间。
5. 活动前压测复制延迟，把主从延迟纳入容量评估。
6. 对大促期间的核心链路配置独立读策略。

### 2.7 结果

优化后：

| 指标 | 优化前 | 优化后 |
|---|---|---|
| 写后读空 | 活动期间偶发集中出现 | 基本消除 |
| 从库延迟告警 | 人工发现 | 自动告警和摘从 |
| 批量任务事务大小 | 单批过大 | 固定小批量提交 |
| 核心读路由 | 简单读写分离 | 按一致性分级 |

### 2.8 可直接口述版本

我遇到过一次订单写后读旧的问题。下单写主库成功，但详情页和列表页默认读从库。活动期间有批量任务产生大事务，从库 SQL 线程回放变慢，主从延迟从毫秒级上升到秒级，导致用户刚下单后读从库查不到订单。

排查时我先确认主库有数据，排除写失败；再看复制状态和心跳延迟，发现从库明显落后；之后通过 processlist 和任务时间线定位到批量任务的大事务回放。短期止血是订单详情读主、刚写完的用户短时间读主、摘掉延迟从库，并暂停批量任务。长期治理是读请求按一致性分级，增加从库延迟自动摘除，批量任务拆小事务，并用版本号判断从库是否追上。这个问题之后，核心链路不再无脑读从，而是按一致性要求路由。

## 3. 案例二：范围更新导致锁冲突

### 3.1 背景

业务有一个订单超时关闭任务，定时把超过支付时间仍未支付的订单改为关闭状态：

```sql
update order_tab
set status = 'closed'
where status = 'created'
  and expire_time < now();
```

订单表数据量较大，同时在线用户仍然在创建订单、支付订单。

### 3.2 故障现象

某次任务执行时，支付回调和订单创建出现 RT 抖动，部分请求超时。  
数据库监控显示：

| 指标 | 表现 |
|---|---|
| 行锁等待 | 明显升高 |
| 死锁 | 偶发出现 |
| 慢 SQL | 超时关闭 update 扫描行数大 |
| CPU/IO | 从正常变为较高 |

### 3.3 排查过程

第一步，看当前锁等待和事务：

```sql
select * from information_schema.innodb_trx;
```

第二步，看最近死锁：

```sql
show engine innodb status\G
```

第三步，看执行计划：

```sql
explain
update order_tab
set status = 'closed'
where status = 'created'
  and expire_time < now();
```

发现索引不够精准，扫描了大量记录。  
在 RR 隔离级别下，范围更新会持有较多锁，影响在线支付更新同一批订单。

### 3.4 根因

根因：

1. 超时关闭任务是大范围 update，单事务影响行数过多。
2. 缺少适合 `status + expire_time` 的联合索引，扫描范围大。
3. 任务一次性提交，持锁时间长。
4. 在线支付和关闭任务访问同一批订单，且状态流转存在竞争。

### 3.5 短期止血

1. 暂停大范围更新任务。
2. 把任务改成小批量查询 ID，再按主键批量更新。
3. 降低每批数量，比如每批 500 或 1000。
4. 避开业务高峰执行。
5. 对支付回调优先保证，关闭任务允许延迟。

示例：

```sql
select id
from order_tab
where status = 'created'
  and expire_time < now()
order by expire_time
limit 500;
```

再按主键更新：

```sql
update order_tab
set status = 'closed'
where id in (...)
  and status = 'created';
```

### 3.6 长期治理

1. 增加联合索引：

```sql
create index idx_status_expire_id
on order_tab(status, expire_time, id);
```

2. 关闭任务固定小批量提交，避免长事务。
3. 状态更新加条件，避免覆盖支付成功：

```sql
update order_tab
set status = 'closed'
where id = ?
  and status = 'created';
```

4. 对订单状态机做幂等和合法流转校验。
5. 增加锁等待、死锁、扫描行数监控。
6. 任务执行前做 explain 检查和灰度。

### 3.7 可直接口述版本

我也可以讲一次锁冲突问题。我们有个超时关单任务，会把过期未支付订单批量改成关闭。某次任务执行时，支付回调 RT 明显升高，并出现锁等待。排查时先看 `information_schema.innodb_trx` 和 `show engine innodb status`，确认是关单任务和支付更新同一批订单互相等待。再看 explain，发现关单 SQL 缺少合适的 `status + expire_time` 联合索引，范围扫描较大，而且单事务更新行数太多。

短期先暂停任务，把它改成按索引查小批量 ID，再按主键分批更新。长期增加 `status, expire_time, id` 联合索引，控制每批数量，状态更新加 `where status = 'created'` 防止覆盖支付成功，并把锁等待和扫描行数接入监控。优化后，关单任务可以慢一点，但不能影响支付主链路。

## 4. 排查命令和指标

### 4.1 主从延迟

```sql
show replica status\G
```

关注：

```text
Replica_IO_Running
Replica_SQL_Running
Seconds_Behind_Source
Read_Source_Log_Pos
Exec_Source_Log_Pos
Relay_Log_Space
```

旧版本 MySQL：

```sql
show slave status\G
```

### 4.2 当前连接和慢操作

```sql
show processlist;
```

关注：

- 长时间执行的查询。
- Waiting for lock。
- 复制线程状态。
- 大事务或 DDL。

### 4.3 事务和锁等待

```sql
select * from information_schema.innodb_trx;
```

```sql
show engine innodb status\G
```

MySQL 8.0 可结合 `performance_schema` 的 data locks 和 lock waits 表。

### 4.4 执行计划

```sql
explain
select id
from order_tab
where status = 'created'
  and expire_time < now()
order by expire_time
limit 500;
```

重点看：

- 是否走目标索引。
- `type` 是否合理。
- `rows` 是否过大。
- 是否 Using filesort。
- 是否需要回表。

## 5. 面试口述版

### 5.1 60 秒版本

我遇到过一次订单链路的主从延迟问题。用户下单写主库成功，但订单详情和列表默认读从库。活动期间批量任务产生大事务，从库回放变慢，导致用户刚下单后读从库看不到订单。我先查主库确认数据存在，排除写失败；再看复制状态和心跳延迟，发现从库落后；最后通过 processlist 和任务时间线定位到大事务回放。

短期止血是核心详情读主、刚写完的用户短时间读主、摘掉延迟从库，并暂停批量任务。长期治理是按一致性分级路由读请求，批量任务拆小事务，增加从库延迟自动摘除和版本校验。这个复盘的结论是，读写分离不能只按 SQL 类型路由，核心链路要按一致性要求路由。

### 5.2 追问扩展

如果面试官问“你怎么证明不是缓存问题”，可以说：

```text
我会同时查主库、从库和缓存里的数据版本。
如果主库有、从库没有、缓存也没有，就说明主要矛盾是复制延迟。
如果主库有、从库也有、缓存没有或旧，就转向缓存失效链路。
```

如果问“为什么不全部读主库”，可以说：

```text
全部读主可以止血，但长期会把主库变成瓶颈。
所以我只把强一致和写后短窗口读切主，普通展示读仍然走从库。
```

## 6. 高频追问

### 6.1 主从延迟最常见原因有哪些？

1. 大事务回放慢。
2. 从库机器资源不足。
3. 从库有慢查询抢资源。
4. DDL 或锁等待阻塞回放。
5. 主库写入峰值超过从库回放能力。

### 6.2 锁冲突最常见原因有哪些？

1. SQL 没有走合适索引。
2. 范围更新锁太多记录和间隙。
3. 事务太长。
4. 多个事务加锁顺序不一致。
5. 热点行被高频更新。

### 6.3 为什么大事务会加重主从延迟？

从库回放一个大事务需要较长时间，后续事务只能排队等待。  
大事务还可能造成锁持有时间长、回滚成本高、binlog 传输和回放压力大。

### 6.4 如何避免状态被并发覆盖？

状态更新要带当前状态条件：

```sql
update order_tab
set status = 'paid'
where order_id = ?
  and status = 'created';
```

然后根据 affected rows 判断是否更新成功。  
这比先查再无条件更新更安全。

## 7. 容易踩坑

1. 不要只说“加索引解决了”，要讲影响、排查、止血和长期治理。
2. 不要把主从延迟说成写失败。
3. 不要只依赖 `Seconds_Behind_Master`，要结合心跳和位点。
4. 不要在事务里做外部 RPC 或长时间循环处理。
5. 不要让低优先级批处理影响支付、下单等核心链路。
